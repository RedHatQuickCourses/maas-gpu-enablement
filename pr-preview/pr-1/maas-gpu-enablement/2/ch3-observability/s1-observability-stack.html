The MaaS Observability Stack

Estimated reading time: 22 minutes.

Objective

Understand how hardware telemetry flows from GPU accelerators to the platform observability plane. Learn the integration between OpenShift’s built-in monitoring (Prometheus/Thanos) and standalone visualization using the Grafana Operator for correlating platform, node, and accelerator metrics.

Why GPU Observability Matters for MaaS

At 3:14 AM on node-gpu-7, DCGM detected GPU memory at 38GB/40GB (95% utilization). Alert fired at 3:18 AM. The platform team drained the node before 4:00 AM production peak serving 200 requests/second inference traffic.

Prevented: $50,000 revenue loss from cascading service failures.

Without monitoring: 4:00 AM crash during peak demand, 45-minute recovery window, customer SLA breach, manual root cause investigation without historical data.

This incident demonstrates why comprehensive GPU observability is non-negotiable for production Models-as-a-Service platforms. Operating MaaS at scale requires comprehensive visibility into the entire infrastructure stack. For GPU-accelerated deployments, this means monitoring:

  • Infrastructure health: CPU, memory, network, storage

  • GPU hardware health: Temperature 78°C, power consumption 280W/300W, ECC error rates

  • GPU utilization: Compute usage 85%, memory usage 32GB/40GB, MIG instance allocation

  • Workload performance: Inference throughput, latency P99 45ms, batch sizes

  • Cost allocation: Per-model GPU usage for chargeback and optimization

Without proper observability, organizations face:

  • Unexpected hardware failures without early warning (average cost: $50K-$200K per incident)

  • Suboptimal resource utilization and wasted capacity (typical: 40% idle GPUs undetected)

  • Inability to correlate application performance with infrastructure metrics

  • Lack of data for capacity planning and cost optimization

  • Difficulty troubleshooting inference performance issues (45 minutes vs. 4 minutes with monitoring)

Production GPU alert thresholds from NVIDIA deployments:

  • Temperature: 80°C warning, 85°C critical (GPU throttling begins at 87-90°C)

  • Memory: 85% warning, 92% critical (OOM kills occur at 95%+)

  • Power: 90% sustained indicates inadequate cooling or PSU capacity

  • ECC errors: Any correctable errors warrant investigation; uncorrectable errors require immediate node drain

These thresholds provide 15-30 minute early warning before hardware failure or performance degradation.

The Telemetry Flow

GPU metrics flow through a multi-tier observability stack:

Layer 1: GPU Hardware Telemetry

NVIDIA DCGM (Data Center GPU Manager) continuously collects telemetry directly from GPU hardware:

  • GPU utilization percentage

  • Memory utilization and allocation

  • GPU temperature and power draw

  • PCIe throughput

  • ECC (error-correcting code) errors

  • NVLink bandwidth (for multi-GPU systems)

  • Per-process GPU usage

  • MIG instance metrics

DCGM runs as a lightweight daemon on each GPU node with minimal performance overhead (<1% GPU utilization).

Integration with Chapter 1: Your ClusterPolicy from Chapter 1 already enabled DCGM:

spec:
  dcgm:
    enabled: true  (1)
1 Deployed nvidia-dcgm DaemonSet automatically across all GPU nodes

MIG-aware metrics: DCGM tracks MIG instances from Chapter 2 with GPU_I_ID labels, enabling per-instance monitoring:

  • GPU=0, GPU_I_ID=0 → First MIG instance (e.g., 1g.5gb profile)

  • GPU=0, GPU_I_ID=1 → Second MIG instance (e.g., 3g.20gb profile)

  • GPU=1, GPU_I_ID=0 → First instance on second physical GPU

This labeling enables per-MIG-instance billing, utilization tracking, and multi-tenant cost allocation for MaaS platforms.

Verification: Confirm DCGM is running on your cluster:

$ oc get pods -n nvidia-gpu-operator -l app=nvidia-dcgm
NAME                  READY   STATUS    RESTARTS   AGE
nvidia-dcgm-xxxxx     1/1     Running   0          5d
nvidia-dcgm-yyyyy     1/1     Running   0          5d

You should see one DCGM pod per GPU node.

Layer 2: Metric Export

DCGM Exporter translates DCGM telemetry into Prometheus-compatible metrics format:

  • Scrapes DCGM on a configurable interval (default: 30 seconds)

  • Exposes metrics via HTTP endpoint on port 9400

  • Adds Kubernetes labels (pod, namespace, node) for correlation

  • Supports custom metric collection profiles

ClusterPolicy configuration: Your GPU Operator ClusterPolicy also enabled the DCGM Exporter:

spec:
  dcgmExporter:
    enabled: true           (1)
    serviceMonitor:
      enabled: true         (2)
      interval: 30s         (3)
    config:
      name: ""              (4)
1 Deploys nvidia-dcgm-exporter DaemonSet on GPU nodes
2 Auto-creates ServiceMonitor for Prometheus scraping (discovered when namespace is labeled)
3 Scrape interval balances metric freshness vs overhead (default: 30 seconds)
4 Empty string uses default metrics profile; custom ConfigMap enables selective metric collection

Example metrics exported:

DCGM_FI_DEV_GPU_UTIL{gpu="0", UUID="GPU-xxxxx", device="nvidia0"} 85
DCGM_FI_DEV_FB_USED{gpu="0", UUID="GPU-xxxxx", device="nvidia0"} 34359738368
DCGM_FI_DEV_POWER_USAGE{gpu="0", UUID="GPU-xxxxx", device="nvidia0"} 280
DCGM_FI_DEV_GPU_TEMP{gpu="0", UUID="GPU-xxxxx", device="nvidia0"} 78
DCGM_FI_DEV_SM_CLOCK{gpu="0", UUID="GPU-xxxxx", device="nvidia0"} 1410

Hands-on verification: Check that the DCGM Exporter is exposing metrics:

$ EXPORTER_POD=$(oc get pods -n nvidia-gpu-operator -l app=nvidia-dcgm-exporter -o name | head -1)
$ oc exec -n nvidia-gpu-operator $EXPORTER_POD -- curl -s localhost:9400/metrics | grep DCGM_FI_DEV_GPU_UTIL
DCGM_FI_DEV_GPU_UTIL{gpu="0",UUID="GPU-abc123...",device="nvidia0",modelName="NVIDIA A100-PCIE-40GB"} 42

If you see metric output, the exporter is functioning correctly and ready for Prometheus collection.

Layer 3: Metric Collection and Storage

OpenShift Monitoring Stack provides the collection and storage tier:

Prometheus: * Scrapes DCGM Exporter endpoints automatically * Stores metrics with configurable retention (default: 15 days) * Evaluates alerting rules for GPU health * Provides PromQL query interface

Thanos Querier: * Federates metrics from multiple Prometheus instances * Provides long-term storage integration * Enables cross-cluster metric queries * Deduplicates metrics from HA Prometheus deployments

User Workload Monitoring: * Separate Prometheus instance for application-level metrics * Enables namespace-scoped metric collection * Isolates tenant metrics from platform metrics * Configurable resource limits and retention policies

Metric collection overhead considerations:

  • GPU impact: 30-second scrape intervals consume ~0.5% GPU utilization + 2-4MB memory per GPU

  • Network impact: Each GPU generates ~15KB/scrape (450KB/minute, ~650MB/day per GPU)

  • Storage impact: 100 GPUs with 15-day retention requires ~15GB Prometheus storage

For clusters with >500 GPUs:

  • Increase scrape interval to 60 seconds (reduces overhead by 50%)

  • Use custom DCGM metric profiles to collect only critical metrics

  • Consider Prometheus remote write for long-term storage offloading

  • Implement metric aggregation for fleet-wide dashboards

The overhead is negligible for typical deployments (<200 GPUs) but warrants optimization at scale.

Layer 4: Visualization and Analysis

Grafana Operator provides the visualization and dashboard layer through a standalone Grafana instance (deployed separately from OpenShift’s built-in console Observe dashboards):

  • Connects to Thanos Querier as primary data source

  • Supports complex PromQL queries for metric correlation

  • Provides pre-built NVIDIA DCGM dashboards via GrafanaDashboard CRs

  • Enables custom dashboard creation for specific use cases

  • Supports alerting, annotations, and templating

  • Implements Dashboard-as-Code via GrafanaDashboard Custom Resources

Why deploy Grafana Operator instead of using OpenShift console Observe dashboards?

While the OpenShift web console provides basic metric visualization, the Grafana Operator enables production-grade GPU monitoring capabilities:

  • Custom dashboard development: Build dashboards with PromQL variables, advanced panels (heatmaps, annotations), and MIG-aware filtering

  • Metric correlation: Combine GPU metrics (DCGM), node metrics (CPU, memory), and inference workload metrics (vLLM latency, throughput) in unified dashboards

  • MIG-aware visualizations: Filter and aggregate metrics by GPU_I_ID labels for per-instance tracking across multi-tenant deployments

  • Dashboard-as-Code: Store dashboards as GrafanaDashboard CRs in Git for version control and GitOps workflows

  • Richer visualization: Access to 200+ panel types, alert overlays, time-series annotations, and comparison views

The OpenShift console Observe section is excellent for quick metric inspection and troubleshooting, but Grafana provides the depth required for production GPU platform operations.

Understanding the GPU Metrics Landscape

GPU observability involves three distinct metric categories, each revealing different aspects of platform health and performance. Understanding this taxonomy helps platform engineers ask the right questions and build effective dashboards.

Metric Categories

Category What It Reveals Example Metrics

Infrastructure Metrics

Kubernetes resource allocation, scheduling decisions, pod lifecycle

kube_pod_container_resource_requests{resource="nvidia.com/gpu"}
kube_node_status_allocatable{resource="nvidia.com/gpu"}
kube_pod_status_phase{pod=~".-gpu-."}

Hardware Metrics

GPU device health, thermal performance, power consumption, error rates

DCGM_FI_DEV_GPU_UTIL (utilization %)
DCGM_FI_DEV_FB_USED (memory bytes)
DCGM_FI_DEV_GPU_TEMP (temperature °C)
DCGM_FI_DEV_POWER_USAGE (power watts)
DCGM_FI_DEV_XID_ERRORS (critical errors)

Workload Metrics

Application performance, inference throughput, model-specific behavior

vllm:request_success_total (inference requests)
vllm:time_to_first_token_seconds (latency)
vllm:request_params_n (batch size)
ovms_requests_success (OpenVINO requests)

Correlation Patterns for Troubleshooting

Combining metrics across categories reveals root causes that single-category analysis misses:

Pattern 1: Memory Bottleneck

  • Symptoms: High DCGM_FI_DEV_GPU_UTIL (85%+) but low vllm:request_success_total (throughput drops 60%)

  • Root cause: DCGM_FI_DEV_FB_USED at 38GB/40GB (95% memory) causing thrashing

  • Resolution: Reduce batch size or migrate to MIG instance with more memory

Pattern 2: OOM Kill Loop

  • Symptoms: Normal DCGM_FI_DEV_GPU_UTIL (45%) but kube_pod_container_status_restarts_total incrementing every 10 minutes

  • Root cause: Pod memory limit too low for GPU memory allocation (host RAM constraint, not GPU VRAM)

  • Resolution: Increase pod resources.limits.memory to accommodate GPU memory management overhead

Pattern 3: Thermal Throttling

  • Symptoms: DCGM_FI_DEV_GPU_TEMP spikes to 87°C, followed by DCGM_FI_DEV_SM_CLOCK drop from 1410MHz to 1200MHz, causing vllm:time_to_first_token_seconds increase from 45ms to 120ms

  • Root cause: Inadequate data center cooling or blocked airflow

  • Resolution: Verify HVAC settings, check for physical obstructions, reduce GPU power limit temporarily

Pattern 4: MIG Instance Imbalance

  • Symptoms: DCGM_FI_DEV_GPU_UTIL{GPU_I_ID="0"} at 95%, DCGM_FI_DEV_GPU_UTIL{GPU_I_ID="1"} at 12% on same physical GPU

  • Root cause: Workload scheduling imbalance across MIG instances

  • Resolution: Adjust pod resource requests or implement topology-aware scheduling

These patterns demonstrate why comprehensive observability across all three metric categories is essential for production GPU operations.

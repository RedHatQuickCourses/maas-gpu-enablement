How MIG Partitioning Works in Kubernetes

Estimated reading time: 20 minutes.

MIG partitioning in OpenShift involves coordination between the GPU Operator components you deployed in Chapter 1. Understanding this workflow helps you troubleshoot configuration issues and plan maintenance windows effectively.

Component Roles in MIG Lifecycle

From Chapter 1, recall these GPU Operator components and their specific roles in MIG enablement:

  • MIG Manager: Watches node labels, validates MIG profiles, applies MIG configuration via nvidia-smi mig commands

  • GPU Feature Discovery (GFD): Scans GPUs using NVML (NVIDIA Management Library), detects MIG instances, applies detailed node labels

  • Device Plugin: Reads GFD-generated labels, advertises MIG instances as allocatable Kubernetes resources

  • NVIDIA Driver: Provides kernel-level MIG mode enablement, instance creation, and hardware partitioning

These components coordinate through the operator reconciliation pattern you learned in Chapter 1, Section 2.

MIG Enablement Workflow

When you label a node with nvidia.com/mig.config=all-2g.10gb, this workflow executes:

1. MIG Manager DaemonSet (controller) detects label change
         ↓
2. MIG Manager validates profile against GPU capability
   - Checks GPU model supports requested profile
   - Verifies profile name exists in ConfigMap
         ↓
3. MIG Manager enables MIG mode on GPU (if not already enabled)
   - Executes: nvidia-smi -i 0 -mig 1
   - Requires GPU reset (brief workload interruption)
   - Takes ~10 seconds per GPU
         ↓
4. MIG Manager creates MIG instances per profile
   - Partitions SMs and memory per profile spec
   - Creates device handles for each instance
   - Takes ~30 seconds for profile application
         ↓
5. GPU Feature Discovery rescans GPU capabilities
   - Queries MIG instances via nvidia-smi
   - Detects MIG geometry and resource allocations
   - Applies labels: nvidia.com/mig-2g.10gb.count=3
         ↓
6. Device Plugin rediscovers allocatable resources
   - Reads GFD-generated MIG labels
   - Advertises: nvidia.com/mig-2g.10gb: 3
   - Updates Node.status.allocatable
         ↓
7. Kubernetes Scheduler sees new resources available
   - Pods requesting nvidia.com/mig-2g.10gb can schedule
   - Workloads begin utilizing MIG instances

Total Time: 10-20 minutes from label application to resource availability (OpenShift documentation guidance)

MIG Reconfiguration Requires Workload Drain

Changing MIG profiles on a node with running GPU workloads requires these steps:

  1. Cordon node: oc adm cordon worker-gpu-0 (prevent new pod scheduling)

  2. Drain node: oc adm drain worker-gpu-0 --ignore-daemonsets --delete-emptydir-data (evict running pods, 5-10 min for workload migration)

  3. Apply new label: oc label node worker-gpu-0 nvidia.com/mig.config=all-3g.20gb --overwrite

  4. Wait for reconfiguration: Monitor MIG Manager logs (10-20 min)

  5. Uncordon node: oc adm uncordon worker-gpu-0 (allow new workloads)

Plan MIG configuration changes during maintenance windows. Unlike time-slicing (which only restarts device plugin pods), MIG mode changes reset the GPU hardware, terminating all running workloads on that GPU.

Integration with ClusterPolicy (Chapter 1)

The ClusterPolicy Custom Resource you configured in Chapter 1 enables MIG Manager capability:

apiVersion: nvidia.com/v1
kind: ClusterPolicy
metadata:
  name: gpu-cluster-policy
spec:
  migManager:
    enabled: true  (1)
  mig:
    strategy: mixed  (2)
1 Deploys MIG Manager DaemonSet—actual profile configuration happens via node labels
2 Sets advertisement strategy (single or mixed) for cluster-wide MIG resource naming

This decouples MIG capability from MIG configuration:

  • Capability (ClusterPolicy): Enables MIG Manager cluster-wide, sets strategy

  • Configuration (node labels): Applies specific profiles per node

This design allows you to:

  • Enable MIG capability cluster-wide once via ClusterPolicy

  • Apply MIG profiles per-node dynamically via labels

  • Change profiles without modifying ClusterPolicy or restarting operators

Enable MIG Manager Even If Not Immediately Using MIG

Set migManager.enabled: true in ClusterPolicy during initial GPU Operator deployment, even if you don’t plan to use MIG immediately. This deploys the MIG Manager DaemonSet in standby mode, allowing you to enable MIG on individual nodes later via labels without cluster-wide operator reconfiguration.

This follows the production best practice from Chapter 1, Section 3: prepare infrastructure capabilities early, activate them as needed.

MIG Configuration Auto-Generation

Starting with GPU Operator v26.3.0, MIG Manager automatically generates MIG configurations for your hardware:

  • Queries GPU capabilities via NVML (NVIDIA Management Library)

  • Discovers supported MIG profiles for each GPU model

  • Creates a ConfigMap named <node-name>-mig-config for each MIG-capable node

  • Includes built-in profiles: all-disabled, all-balanced, and per-profile configs (e.g., all-1g.10gb, all-2g.10gb)

Legacy behavior (pre-v26.3.0): Used static ConfigMap default-mig-parted-config with predefined profiles.

This auto-generation means you typically don’t need custom ConfigMaps unless you require specialized heterogeneous combinations not covered by all-balanced.

Enabling MIG via Node Labels

MIG configuration uses a label-driven workflow. The MIG Manager watches for specific labels on GPU nodes and applies the corresponding profile from its configuration.

Label-Driven Configuration Pattern

The GPU Operator uses this label schema:

nvidia.com/mig.config=<profile-name>

Built-in Profile Names (no additional ConfigMap configuration needed):

  • all-disabled - MIG mode off, GPU operates as single full device (default)

  • all-1g.5gb - Partition into 7x 1g.5gb instances (A100-40GB)

  • all-1g.10gb - Partition into 7x 1g.10gb instances (A100-80GB)

  • all-2g.10gb - Partition into 3x 2g.10gb instances (A100-40GB)

  • all-3g.20gb - Partition into 2x 3g.20gb instances (A100-40GB)

  • all-balanced - Balanced heterogeneous mix (2x 1g.5gb + 1x 2g.10gb + 1x 3g.20gb on A100-40GB)

Example: Applying MIG Profile to Nodes

# Set MIG strategy in ClusterPolicy (one-time cluster-wide configuration)
$ oc patch clusterpolicy/gpu-cluster-policy --type='json' \
  -p='[{"op": "replace", "path": "/spec/mig/strategy", "value": "mixed"}]'

# Label GPU nodes for MIG configuration
$ oc label node worker-gpu-0.example.com nvidia.com/mig.config=all-2g.10gb --overwrite
node/worker-gpu-0.example.com labeled

$ oc label node worker-gpu-1.example.com nvidia.com/mig.config=all-2g.10gb --overwrite
node/worker-gpu-1.example.com labeled

# Monitor MIG Manager applying configuration
$ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager -f
time="2024-04-10T14:23:15Z" level=info msg="Applying MIG config" node=worker-gpu-0 profile=all-2g.10gb
time="2024-04-10T14:23:45Z" level=info msg="MIG configuration complete" instances=3

# Verify MIG instances are advertised (wait 10-20 minutes)
$ oc describe node worker-gpu-0.example.com | grep -A 10 "Allocatable:"
Allocatable:
  cpu:                          31500m
  memory:                       252455456Ki
  nvidia.com/mig-2g.10gb:       3  (1)
  pods:                         250
1 Three MIG instances now available for workload scheduling

Custom MIG Profiles via ConfigMap

For mixed profiles or custom combinations beyond all-balanced, create a mig-parted configuration:

apiVersion: v1
kind: ConfigMap
metadata:
  name: custom-mig-parted-config
  namespace: nvidia-gpu-operator
data:
  config.yaml: |
    version: v1
    mig-configs:
      mixed-inference:  (1)
        - devices: all
          mig-enabled: true
          mig-devices:
            "3g.20gb": 1  (2)
            "2g.10gb": 2  (3)
1 Custom profile name for node labeling
2 One large instance for LLaMA-7B inference
3 Two medium instances for BERT-class models

Apply Custom Profile to ClusterPolicy:

$ oc patch clusterpolicy gpu-cluster-policy --type=merge -p '
  {
    "spec": {
      "migManager": {
        "config": {
          "name": "custom-mig-parted-config"  (1)
        }
      }
    }
  }'
1 Reference your custom ConfigMap

Use Custom Profile on Nodes:

$ oc label node worker-gpu-2.example.com \
  nvidia.com/mig.config=mixed-inference  (1)
1 Must match profile name from ConfigMap mig-configs section

You’ll apply these configurations hands-on in Section 2’s lab. This overview prepares you for the practical workflow and troubleshooting during the lab.

For exhaustive lists of supported MIG profile combinations and placement constraints, consult the [NVIDIA MIG User Guide](https://docs.nvidia.com/datacenter/tesla/mig-user-guide/).

Multi-Strategy Deployments in Large Platforms

Production platforms often use multiple strategies simultaneously based on workload characteristics:

  • Training node pool: Full GPU allocation

    • Workloads need maximum GPU memory and compute

    • Checkpointing requires exclusive access

    • Label: node-role.kubernetes.io/gpu-training=true

  • Inference node pool (A100-equipped): MIG partitioning

    • Multi-tenant model serving with SLA requirements

    • Hardware isolation for predictable latency

    • Label: node-role.kubernetes.io/gpu-inference-mig=true

  • Inference node pool (T4/V100-equipped): Time-slicing

    • No MIG support on these older architectures

    • Acceptable for non-SLA workloads

    • Label: node-role.kubernetes.io/gpu-inference-timeslice=true

Use node selectors and taints/tolerations to route workloads to appropriate node pools:

nodeSelector:
  node-role.kubernetes.io/gpu-inference-mig: "true"

This architecture provides workload placement flexibility while optimizing each pool for its use case.

MIG Benefits for MaaS

For Models-as-a-Service architectures, MIG provides measurable advantages that directly impact platform economics and operational reliability.

Cost Efficiency and ROI

  • Deploy 7 concurrent small models on single A100-40GB (vs. 1 with full GPU allocation)

  • Reduce per-model GPU cost from $15,000 to $2,143 (7x cost reduction for 1g.5gb profiles)

  • Increase cluster-wide GPU utilization from 33% to 78% (2.4x improvement over exclusive allocation)

  • Achieve ROI break-even 2.4x faster than full GPU deployments (10 months vs. 24 months baseline)

  • Right-size GPU resources to actual model requirements, eliminating overprovisioning waste

Concrete example: Platform serving 21 small models previously required 21x A100 GPUs ($315,000 capital). With MIG 1g.5gb profiles, same workload runs on 3x A100 GPUs ($45,000 capital), saving $270,000.

Performance Isolation and Predictability

  • Guarantee P99 latency variance <1% (vs. 15-40% with time-slicing)

  • Hardware-enforced memory isolation prevents out-of-memory (OOM) crosstalk between workloads

  • Dedicated streaming multiprocessors eliminate compute contention and throttling

  • Enable SLA-backed inference with measurable, enforceable performance guarantees

  • Prevent noisy neighbor problems through physical resource partitioning

Time-slicing workloads experience latency spikes when concurrent requests arrive. MIG instances maintain consistent latency regardless of neighboring workload activity.

Operational Flexibility and Multi-Tenancy

  • Scale model instances independently without affecting neighboring services (e.g., scale LLaMA-7B from 1 to 3 replicas without reprovisioning hardware)

  • Mix small and large models on same physical GPU (e.g., 1g.5gb microservices alongside 3g.20gb large language models)

  • Support true multi-tenant deployments with hardware isolation between teams

  • Assign dedicated MIG instances to specific tenants or namespaces for guaranteed capacity

  • Enable chargeback and cost allocation with accurate per-MIG-instance metrics from DCGM

Resource Predictability and Capacity Planning

  • Guaranteed GPU resources per deployment: Each InferenceService gets dedicated SMs and VRAM

  • Reduce capacity planning variance from ±40% to ±5%: MIG instances have predictable, fixed resource allocations

  • Simplify quota management: Assign 2x 2g.10gb instances per team, enforceable via Kueue (Chapter 4)

  • Enable accurate per-model billing: DCGM provides per-MIG-instance utilization metrics for cost tracking

  • Predictable failure domains: OOM or crash in one MIG instance doesn’t affect others on same GPU

Production Considerations and Gotchas

While MIG provides significant benefits, production deployments require careful planning around reconfiguration downtime, workload placement, and ongoing monitoring.

Reconfiguration Downtime Planning

Unlike time-slicing (which only requires device plugin pod restart), MIG reconfiguration requires GPU hardware reset and workload migration.

Downtime Components for Profile Changes:

  1. Node cordon: Immediate (prevents new scheduling)

  2. Node drain: 5-10 minutes (waiting for workload migration to other nodes)

  3. MIG mode enablement: 10-15 seconds (GPU hardware reset)

  4. Profile application: 30-60 seconds (instance creation)

  5. GFD label update: 30-60 seconds (capability rescan)

  6. Device plugin restart: 30-60 seconds (resource rediscovery)

  7. Node uncordon + scheduling: 30-60 seconds

Total: 10-20 minutes per node for profile changes (per OpenShift documentation)

Plan MIG Profile Changes During Maintenance Windows

Changing MIG profiles on production nodes serving live traffic causes:

  • Immediate termination of all GPU workloads on that node

  • Pod eviction and rescheduling to other nodes

  • Temporary capacity reduction during reconfiguration

  • Potential cascading failures if spare capacity insufficient

For rolling MIG updates across N nodes:

  • Plan for N × 20 minutes total reconfiguration time

  • Ensure spare GPU capacity exists for pod rescheduling (recommend N+2 redundancy)

  • Use node cordoning to control blast radius: oc adm cordon worker-gpu-0

  • Test profile changes in dev environment first

  • Schedule during low-traffic windows (e.g., weekend maintenance)

  • Document rollback procedure (revert to all-disabled, then previous profile)

Workload Placement Strategies

MIG instances require explicit resource requests. Workloads must request specific MIG profiles when using mixed advertisement strategy.

Correct Resource Request (Mixed Strategy):

resources:
  limits:
    nvidia.com/mig-2g.10gb: 1  (1)
1 Request specific MIG profile, matches advertised resource name

Common Mistake:

resources:
  limits:
    nvidia.com/gpu: 1  (1)
1 ❌ Will NOT schedule on MIG-partitioned nodes using mixed strategy

Consequences of incorrect requests:

  • Pod stuck in Pending state with FailedScheduling event

  • Error: "Insufficient nvidia.com/gpu" (even though MIG instances available)

  • Requires pod specification update and redeployment

Use Admission Controllers for Default MIG Profiles

For platforms deploying many inference services, create a MutatingWebhookConfiguration that automatically injects appropriate MIG resource requests based on pod annotations or namespace labels. This prevents scheduling failures from incorrect resource requests.

Example implementation:

  • Pods in namespace small-models automatically get nvidia.com/mig-1g.5gb: 1 injected

  • Pods with annotation mig-profile: medium get nvidia.com/mig-2g.10gb: 1

  • Pods without annotations remain unchanged (for flexibility)

This pattern reduces operational toil and prevents common scheduling errors, especially useful for teams with 50+ inference services.

Monitoring MIG Utilization

DCGM (Data Center GPU Manager, deployed in Chapter 1) provides per-MIG-instance metrics. Monitor these to validate your profile choices and identify optimization opportunities.

Key Metrics to Track:

  • DCGM_FI_PROF_GR_ENGINE_ACTIVE: Compute utilization percentage per MIG instance

  • DCGM_FI_DEV_FB_USED: Memory usage in bytes per MIG instance

  • DCGM_FI_DEV_GPU_TEMP: Temperature per MIG instance (thermal throttling indicator)

  • DCGM_FI_DEV_POWER_USAGE: Power consumption per instance

Red Flags Indicating Misconfigurations:

  • MIG instance averaging >85% memory usage: Profile too small, workload may OOM, resize to larger profile

  • MIG instance averaging <20% compute utilization: Profile too large, wasted resources, resize to smaller profile

  • Frequent pod evictions (OOMKilled): Memory oversubscription, increase profile memory allocation

  • High scheduling failure rate: Insufficient MIG capacity for demand, add nodes or adjust profiles

Example Prometheus query for MIG memory usage:

DCGM_FI_DEV_FB_USED{GPU_I_ID=~".*"} /
  DCGM_FI_DEV_FB_TOTAL{GPU_I_ID=~".*"} * 100

You’ll build comprehensive MIG monitoring dashboards in Chapter 3.

MIG Instance Naming in Metrics

DCGM metrics use GPU_I_ID label to distinguish MIG instances:

  • GPU=0, GPU_I_ID=0: First MIG instance on GPU 0

  • GPU=0, GPU_I_ID=1: Second MIG instance on GPU 0

This differs from Kubernetes resource names (nvidia.com/mig-2g.10gb). Correlation requires mapping GFD labels (showing profile types) to DCGM metrics (showing instance IDs).

Dashboard design tip: Group metrics by profile type using GFD label joins to show "all 2g.10gb instances" rather than raw instance IDs.

Hardware Requirements and GPU Model Support

MIG is available exclusively on NVIDIA Ampere (A-series) and Hopper (H-series) architecture GPUs:

Supported GPU Models:

  • NVIDIA A30 (24GB): Ampere architecture, up to 4 MIG instances (profiles: 1g.6gb, 2g.12gb, 4g.24gb)

  • NVIDIA A100-40GB: Ampere architecture, up to 7 MIG instances (profiles: 1g.5gb through 7g.40gb)

  • NVIDIA A100-80GB: Ampere architecture, up to 7 MIG instances (profiles: 1g.10gb through 7g.80gb)

  • NVIDIA H100 (80GB/94GB): Hopper architecture, enhanced MIG with up to 7 instances plus confidential computing per-instance support

Unsupported GPU Models (must use time-slicing instead):

  • NVIDIA V100 (Volta architecture)

  • NVIDIA T4 (Turing architecture)

  • NVIDIA A10, A16, A40 (Ampere, but consumer/workstation GPUs without MIG feature)

  • NVIDIA L4, L40 (Ada Lovelace architecture)

H100 Enhancements Over A100

H100 GPUs offer additional MIG capabilities beyond A100:

  • Confidential computing support per MIG instance: Each instance can run in secure encrypted mode

  • Improved memory bandwidth allocation: Better isolation between instances

  • Faster reconfiguration: H100 MIG mode changes complete ~15% faster than A100

For most Models-as-a-Service inference workloads, A100 provides optimal cost/performance balance. H100 advantages primarily benefit specialized secure computation or extremely memory-bandwidth-intensive workloads. Evaluate H100 premium pricing (~2.5x A100 cost) against actual workload requirements before procurement.

3.4: Expanding the catalog with selective tier access

ChocoTech’s Marketing team requests access to a smaller, faster model for sentiment analysis. To accommodate this, we will deploy a second model to demonstrate multi-model catalog management.

This exercise shows how easily you can expand the model catalog as new business requirements emerge. More importantly, it demonstrates how to apply strict, selective access controls so that expensive or restricted models do not appear for unauthorized users.

Exercise 1: The restricted deployment manifest

We will deploy the nvidia/nemotron-4-8b-instruct model. Unlike our general-purpose Llama-3 deployment, this model is restricted to bronze and silver tiers, demonstrating selective access control.

To maintain our secure GitOps and Vault strategy, we will assume the weights have been staged in our local MinIO vault (s3://models/nemotron). We will use the exact same llm-d routing structure as before, but we will alter the "Bouncer List" annotation.

Review and apply the nemotron-deployment.yaml manifest:

cat <<EOF | oc apply -f -
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: nemotron-8b
  namespace: chocotech-models
  annotations:
    # THE BOUNCER LIST: Explicitly omitting 'gold-tier'
    alpha.maas.opendatahub.io/tiers: '["silver-tier", "bronze-tier"]'
spec:
  publishToGenAIStudio: true
  model:
    uri: s3://models/nemotron
    name: nvidia/nemotron-4-8b-instruct
  servingRuntime: "vllm-cuda-runtime"
  router:
    scheduler:
      template:
        containers:
          - name: main
            args:
              - '--pool-group'
              - 'inference.networking.x-k8s.io'
              - '--pool-name'
              - '{{ .ObjectMeta.Name }}-inference-pool'
  template:
    spec:
      serviceAccountName: fast-track-sa
      containers:
        - name: main
          env:
            - name: VLLM_ADDITIONAL_ARGS
              value: "--max-model-len=8192 --gpu-memory-utilization=0.90"
  placement:
    accelerator:
      name: "nvidia.com/gpu"
      count: 1
EOF

Wait for the deployment to become ready:

oc wait --for=condition=ready llminferenceservice/nemotron-8b -n chocotech-models --timeout=600s

Expected output: llminferenceservice.serving.kserve.io/nemotron-8b condition met.

Exercise 2: Verify the multi-model catalog

With a single oc apply command, the MaaS operator automatically registers the new endpoint, attaches the Kuadrant rate limit policies for Silver and Bronze, and updates the Gen AI Studio UI.

Let’s verify this multi-model environment.

1. List all deployed models:

oc get llminferenceservice -n chocotech-models

Expected output:

NAME           READY   REASON   AGE
llama-3-8b     True    Ready    20m
nemotron-8b    True    Ready    5m

2. Check the model catalog via API: Because MaaS is OpenAI-compliant, applications can dynamically discover available models.

GATEWAY_HOST=$(oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.addresses[0].value}')
curl -s http://$GATEWAY_HOST/v1/models | jq

Expected output showing both models:

{
  "object": "list",
  "data": [
    {
      "id": "meta-llama/Meta-Llama-3-8B-Instruct",
      "object": "model",
      "owned_by": "chocotech-models"
    },
    {
      "id": "nvidia/nemotron-4-8b-instruct",
      "object": "model",
      "owned_by": "chocotech-models"
    }
  ]
}

3. The UI Payoff Refresh the Gen AI Studio Models tab in the dashboard. You should now see both models listed with their respective tier access controls.

Verify the restrictions: Models show different tier access (Llama-3: all tiers, Nemotron: bronze/silver only).

ChocoTech now operates a fully governed, multi-model AI factory. Quality Control relies on the universally available Llama-3, while Marketing and R&D can utilize the Nemotron model without risking exposure to the Gold tier production workloads.

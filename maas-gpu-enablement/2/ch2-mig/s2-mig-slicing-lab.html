Lab: MIG Slicing Configuration

Estimated reading time: 45 minutes.

Objective

Update node labels to apply MIG partitioning profiles, create custom mig-parted configuration resources for specific hardware deployments, and verify that sliced MIG instances are exposed as allocatable resources on GPU nodes.

Before you begin

This lab requires:

  • Completed installation of NVIDIA GPU Operator from Chapter 1

  • OpenShift cluster with NVIDIA A30, A100, or H100 GPU nodes

  • Cluster administrator privileges

  • MIG-capable GPU hardware (Ampere or Hopper architecture)

Applying MIG configuration requires GPU reset and will interrupt any running GPU workloads. Schedule this activity during a maintenance window.

Instructions

1. Verify MIG Capability

Before applying MIG configuration, verify your GPU hardware supports MIG. This prevents configuration errors and clarifies which profiles are available.

Why Verify Before Labeling?

The GPU Operator’s MIG Manager will fail silently if you label a non-MIG-capable GPU (e.g., V100, T4). Verifying capability first:

  • Confirms hardware supports MIG (A30/A100/H100 only)

  • Shows current MIG mode state (enabled/disabled)

  • Prevents wasted 10-20 minute reconfiguration attempts

Reference: MIG Hardware Requirements in s1

First, confirm your GPU hardware supports MIG.

  1. Check GPU model on a worker node

    $ oc debug node/worker-gpu-0.example.com
Starting pod/worker-gpu-0examplecom-debug ...
sh-4.4# chroot /host
sh-4.4# nvidia-smi -L
GPU 0: NVIDIA A100-PCIE-40GB (UUID: GPU-xxxxx)

  2. Verify MIG mode capability

    sh-4.4# nvidia-smi -i 0 --query-gpu=mig.mode.current,mig.mode.pending --format=csv
mig.mode.current, mig.mode.pending
Disabled, Disabled

    If MIG mode shows as available (even if disabled), your hardware supports MIG. Exit the debug pod.

2. Enable MIG Mode Using Node Labels

The GPU Operator can enable MIG mode by applying specific node labels.

  1. Label GPU nodes to enable MIG mode

    $ oc label node worker-gpu-0.example.com nvidia.com/mig.config=all-1g.10gb --overwrite
node/worker-gpu-0.example.com labeled

$ oc label node worker-gpu-1.example.com nvidia.com/mig.config=all-1g.10gb --overwrite
node/worker-gpu-1.example.com labeled

    The all-1g.10gb profile will partition each GPU into seven 1g.10gb MIG instances (for A100 80GB) or seven 1g.5gb instances (for A100 40GB).

  2. Monitor the MIG manager daemonset as it reconfigures GPUs

    $ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager -f
time="2024-04-10T14:23:15Z" level=info msg="Applying MIG configuration" node=worker-gpu-0.example.com profile=all-1g.10gb
time="2024-04-10T14:23:18Z" level=info msg="Enabling MIG mode on GPU 0"
time="2024-04-10T14:23:20Z" level=info msg="Creating MIG instances: 1g.10gb x7"
time="2024-04-10T14:23:25Z" level=info msg="MIG configuration complete"

  3. Wait for the mig-manager pod to complete the reconfiguration

    $ oc get pods -n nvidia-gpu-operator -l app=nvidia-mig-manager
NAME                          READY   STATUS      RESTARTS   AGE
nvidia-mig-manager-xxxxx      0/1     Completed   0          3m

3. Create Custom MIG Configuration

For more advanced MIG profiles or mixed configurations, create a custom ConfigMap for mig-parted.

  1. Create a custom MIG configuration file

    $ cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
*  name: custom-mig-parted-config*
*  namespace: nvidia-gpu-operator*
data:
*  config.yaml: |*
*    version: v1*
*    mig-configs:*
*      all-2g.20gb:*
*        - devices: all*
*          mig-enabled: true*
*          mig-devices:*
*            "2g.20gb": 3*
*      mixed-profile:*
*        - devices: all*
*          mig-enabled: true*
*          mig-devices:*
*            "1g.10gb": 2*
*            "3g.40gb": 1*
EOF
configmap/custom-mig-parted-config created

  2. Update the ClusterPolicy to use the custom config

    $ oc patch clusterpolicy gpu-cluster-policy --type=merge -p '
*  {*
*    "spec": {*
*      "migManager": {*
*        "config": {*
*          "name": "custom-mig-parted-config"*
*        }*
*      }*
*    }*
*  }'*
clusterpolicy.nvidia.com/gpu-cluster-policy patched

  3. Apply a custom profile to specific nodes

    $ oc label node worker-gpu-2.example.com nvidia.com/mig.config=all-2g.20gb --overwrite
node/worker-gpu-2.example.com labeled

4. Verify MIG Instance Allocation

  1. Check that MIG instances are now visible as allocatable resources

    $ oc describe node worker-gpu-0.example.com | grep -A 10 "Allocatable:"
Allocatable:
  cpu:                          31500m
  ephemeral-storage:            487267823448
  hugepages-1Gi:                0
  hugepages-2Mi:                0
  memory:                       252455456Ki
  nvidia.com/mig-1g.10gb:       7
  pods:                         250

    You should see nvidia.com/mig-1g.10gb: 7 indicating seven MIG instances are available.

  2. View detailed MIG device information

    $ oc debug node/worker-gpu-0.example.com
sh-4.4# chroot /host
sh-4.4# nvidia-smi -L
GPU 0: NVIDIA A100-PCIE-40GB (UUID: GPU-xxxxx)

  MIG 1g.10gb     Device  2: (UUID: MIG-xxxxx-2)
  MIG 1g.10gb     Device  3: (UUID: MIG-xxxxx-3)
  MIG 1g.10gb     Device  4: (UUID: MIG-xxxxx-4)
  MIG 1g.10gb     Device  5: (UUID: MIG-xxxxx-5)
  MIG 1g.10gb     Device  6: (UUID: MIG-xxxxx-6)

  3. Test MIG instance allocation with a sample workload

    $ cat <<EOF | oc apply -f -
apiVersion: v1
kind: Pod
metadata:
*  name: mig-test*
spec:
*  restartPolicy: OnFailure*
*  containers:*
*  - name: cuda-test*
*    image: nvidia/cuda:12.2.0-base-ubi8*
*    command: ["nvidia-smi"]*
*    resources:*
*      limits:*
*        nvidia.com/mig-1g.10gb: 1*
EOF
pod/mig-test created

You have successfully configured Multi-Instance GPU partitioning on your OpenShift cluster. Physical GPUs are now split into multiple isolated instances that can be independently allocated to model serving workloads, maximizing GPU utilization and return on investment.

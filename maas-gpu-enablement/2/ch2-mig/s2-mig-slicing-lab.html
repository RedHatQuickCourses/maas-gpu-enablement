Lab: MIG Slicing Configuration

Estimated reading time: 45 minutes.

Objective

Update node labels to apply MIG partitioning profiles, create custom mig-parted configuration resources for specific hardware deployments, and verify that sliced MIG instances are exposed as allocatable resources on GPU nodes.

Before you begin

This lab requires:

  • Completed installation of NVIDIA GPU Operator from Chapter 1

  • OpenShift cluster with NVIDIA A30, A100, or H100 GPU nodes

  • Cluster administrator privileges

  • MIG-capable GPU hardware (Ampere or Hopper architecture)

Applying MIG configuration requires GPU reset and will interrupt any running GPU workloads. Schedule this activity during a maintenance window.

Instructions

1. Verify MIG Capability

Before applying MIG configuration, verify your GPU hardware supports MIG. This prevents configuration errors and clarifies which profiles are available.

Why Verify Before Labeling?

The GPU Operator’s MIG Manager will fail silently if you label a non-MIG-capable GPU (e.g., V100, T4). Verifying capability first:

  • Confirms hardware supports MIG (A30/A100/H100 only)

  • Shows current MIG mode state (enabled/disabled)

  • Prevents wasted 10-20 minute reconfiguration attempts

Reference: MIG Hardware Requirements in s1

First, confirm your GPU hardware supports MIG.

  1. Check GPU model on a worker node

    $ oc debug node/worker-gpu-0.example.com
Starting pod/worker-gpu-0examplecom-debug ...
sh-4.4# chroot /host
sh-4.4# nvidia-smi -L
GPU 0: NVIDIA A100-PCIE-40GB (UUID: GPU-xxxxx)

  2. Verify MIG mode capability

    sh-4.4# nvidia-smi -i 0 --query-gpu=mig.mode.current,mig.mode.pending --format=csv
mig.mode.current, mig.mode.pending
Disabled, Disabled

    If MIG mode shows as available (even if disabled), your hardware supports MIG. Exit the debug pod.

2. Configure MIG Strategy and Enable MIG Mode

The GPU Operator supports two MIG advertisement strategies (see MIG Advertisement Strategies in s1):

  • Single: All instances advertised as nvidia.com/gpu (backward compatible)

  • Mixed: Instances advertised with profile-specific names like nvidia.com/mig-2g.10gb

For this lab, we’ll use mixed strategy to demonstrate heterogeneous profiles.

  1. Set MIG strategy to mixed in ClusterPolicy

    $ oc patch clusterpolicy/gpu-cluster-policy --type='json' \
  -p='[{"op": "replace", "path": "/spec/mig/strategy", "value": "mixed"}]'
clusterpolicy.nvidia.com/gpu-cluster-policy patched

    Why Set Strategy First?

    The MIG strategy determines how resources are advertised to Kubernetes:

    • Single: nvidia.com/gpu: 7 (all instances identical)

    • Mixed: nvidia.com/mig-1g.5gb: 7 (profile-specific)

    Setting strategy before labeling nodes ensures consistent resource naming. This is a cluster-wide setting in ClusterPolicy, not per-node.

    Reference: Strategy Selection Matrix in s1

  2. Label GPU nodes with MIG profile appropriate for your hardware

    For A100-40GB GPUs, use profiles with smaller memory allocations:

    $ oc label node worker-gpu-0.example.com nvidia.com/mig.config=all-1g.5gb --overwrite
node/worker-gpu-0.example.com labeled

    For A100-80GB GPUs, use profiles with larger memory allocations:

    $ oc label node worker-gpu-1.example.com nvidia.com/mig.config=all-1g.10gb --overwrite
node/worker-gpu-1.example.com labeled

    Profile Naming Convention: A100-40GB vs. A100-80GB

    MIG profile names encode memory allocation:

    • A100-40GB: 1g.5gb (5GB per instance), 2g.10gb (10GB), 3g.20gb (20GB)

    • A100-80GB: 1g.10gb (10GB per instance), 2g.20gb (20GB), 3g.40gb (40GB)

    The all-1g.5gb profile on A100-40GB creates 7 instances of 5GB each. The all-1g.10gb profile on A100-80GB creates 7 instances of 10GB each.

    Both maximize instance density (7x) for small model serving.

    Reference: MIG Profile Table in s1

  3. Monitor the MIG Manager DaemonSet as it reconfigures GPUs

    $ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager -f
time="2024-04-10T14:23:15Z" level=info msg="Applying MIG configuration" node=worker-gpu-0.example.com profile=all-1g.5gb
time="2024-04-10T14:23:18Z" level=info msg="Enabling MIG mode on GPU 0"
time="2024-04-10T14:23:20Z" level=info msg="Creating MIG instances: 1g.5gb x7"
time="2024-04-10T14:23:25Z" level=info msg="MIG configuration complete"

    Expected Reconfiguration Time: 10-20 Minutes

    The log message "MIG configuration complete" indicates MIG Manager finished its work, but allocatable resources may not appear immediately. The full workflow requires:

    1. MIG Manager enables MIG mode and creates instances (~60 seconds)

    2. GPU Feature Discovery rescans GPU capabilities (~60 seconds)

    3. Device Plugin rediscovers resources (~60 seconds)

    4. Kubernetes scheduler updates node status (~30 seconds)

    Wait 10-20 minutes before checking oc describe node for MIG resources. This timing is normal for GPU hardware reconfiguration.

    Reference: MIG Enablement Workflow Diagram in s1

  4. Verify MIG configuration state on node labels

    Wait 10-20 minutes, then check the node label showing configuration state:

    $ oc get node worker-gpu-0.example.com -o jsonpath='{.metadata.labels.nvidia\.com/mig\.config\.state}'
success

    The success state indicates MIG Manager completed reconfiguration without errors.

3. Create Custom MIG Configuration

The GPU Operator auto-generates MIG configurations for standard profiles (all-1g.5gb, all-2g.10gb, all-balanced). For advanced heterogeneous combinations not covered by all-balanced, create a custom mig-parted ConfigMap.

  1. Create a custom MIG configuration file for mixed A100-40GB profiles

    $ cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
*  name: custom-mig-parted-config*
*  namespace: nvidia-gpu-operator*
data:
*  config.yaml: |*
*    version: v1*
*    mig-configs:*
*      all-2g.10gb:*                      (1)
*        - devices: all*
*          mig-enabled: true*
*          mig-devices:*
*            "2g.10gb": 3*                (2)
*      mixed-inference:*                 (3)
*        - devices: all*
*          mig-enabled: true*
*          mig-devices:*
*            "1g.5gb": 2*                 (4)
*            "2g.10gb": 1*
*            "3g.20gb": 1*
EOF
configmap/custom-mig-parted-config created
    1 Homogeneous profile: 3 medium instances
    2 Each instance gets 2/7 compute + 10GB memory
    3 Heterogeneous profile matching s1 multi-tenant scenario
    4 Creates 4 total instances: 2 small + 1 medium + 1 large

    Custom Profile Naming and Validation

    Profile names in mig-configs section become node labels. The mixed-inference profile can be applied with:

    oc label node worker-gpu-2.example.com nvidia.com/mig.config=mixed-inference --overwrite

    Profile Validation Rules (A100-40GB):

    • Compute slices must sum to ≤7g (e.g., 1g + 1g + 2g + 3g = 7g ✅)

    • Memory must match compute (1g → 5GB, 2g → 10GB, 3g → 20GB)

    • Exceeding 7g total compute will fail MIG Manager validation ❌

    Reference: Profile Combinations Table in s1

  2. Update the ClusterPolicy to use the custom ConfigMap

    $ oc patch clusterpolicy gpu-cluster-policy --type=merge -p '
*  {*
*    "spec": {*
*      "migManager": {*
*        "config": {*
*          "name": "custom-mig-parted-config"*
*        }*
*      }*
*    }*
*  }'*
clusterpolicy.nvidia.com/gpu-cluster-policy patched

  3. Apply a custom profile to specific node

    $ oc label node worker-gpu-2.example.com nvidia.com/mig.config=mixed-inference --overwrite
node/worker-gpu-2.example.com labeled

    Wait 10-20 minutes for reconfiguration to complete.

5. Configure Heterogeneous MIG with All-Balanced Profile

The built-in all-balanced profile creates a heterogeneous mix of instance sizes on a single GPU, ideal for multi-tenant platforms with diverse workload tiers.

  1. Apply the all-balanced profile to a GPU node

    $ oc label node worker-gpu-1.example.com nvidia.com/mig.config=all-balanced --overwrite
node/worker-gpu-1.example.com labeled

    What Does All-Balanced Create?

    For A100-40GB, all-balanced creates:

    • 2x 1g.5gb instances (small models: 5GB each)

    • 1x 2g.10gb instance (medium models: 10GB)

    • 1x 3g.20gb instance (large models: 20GB)

    Total: 4 instances spanning three size tiers, matching the multi-tenant scenario from s1.

    Reference: Multi-Tenant Scenario in s1

  2. Wait 10-20 minutes, then verify heterogeneous instances are created

    $ oc describe node worker-gpu-1.example.com | grep nvidia.com/mig
  nvidia.com/mig-1g.5gb:       2
  nvidia.com/mig-2g.10gb:      1
  nvidia.com/mig-3g.20gb:      1
  nvidia.com/mig.config=all-balanced
  nvidia.com/mig.config.state=success
  nvidia.com/mig.strategy=mixed

    You should see three different resource types advertised on the same node.

Troubleshooting Common Issues

MIG Configuration Not Applied After 20 Minutes

Symptom: After labeling node, oc describe node still shows nvidia.com/gpu: X instead of nvidia.com/mig-* resources

Diagnosis Steps:

  1. Verify node label was applied correctly

    $ oc get node worker-gpu-0.example.com -o jsonpath='{.metadata.labels.nvidia\.com/mig\.config}'
all-1g.5gb

    If output is empty, the label wasn’t applied. Re-apply with --overwrite flag.

  2. Check MIG Manager logs for errors

    $ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager --tail=50

    Look for: * level=error messages indicating validation failures * MIG configuration complete success message * Applying MIG configuration followed by profile name

  3. Check MIG configuration state label

    $ oc get node worker-gpu-0.example.com -o jsonpath='{.metadata.labels.nvidia\.com/mig\.config\.state}'
success

    Possible states: * success: MIG Manager completed configuration ✅ * pending: Reconfiguration in progress (wait 10-20 min) * failed: Error occurred, check MIG Manager logs ❌

  4. Verify GPU supports MIG mode

    $ oc debug node/worker-gpu-0.example.com
sh-4.4# chroot /host
sh-4.4# nvidia-smi -i 0 --query-gpu=mig.mode.current --format=csv
mig.mode.current
Enabled

    If shows Disabled after 20 minutes, GPU may not support MIG or MIG Manager failed to enable it.

Custom ConfigMap Not Recognized

Symptom: After creating custom ConfigMap and patching ClusterPolicy, custom profiles not available

Solution:

  1. Verify ConfigMap exists in correct namespace

    $ oc get configmap -n nvidia-gpu-operator custom-mig-parted-config
NAME                        DATA   AGE
custom-mig-parted-config    1      5m

  2. Verify ConfigMap is referenced in ClusterPolicy

    $ oc get clusterpolicy gpu-cluster-policy -o jsonpath='{.spec.migManager.config.name}'
custom-mig-parted-config

    If output is empty, the patch didn’t apply. Re-run oc patch clusterpolicy command.

  3. Restart MIG Manager pods to pick up new ConfigMap

    $ oc delete pods -n nvidia-gpu-operator -l app=nvidia-mig-manager
pod "nvidia-mig-manager-xxxxx" deleted

    Pods will automatically recreate and load new ConfigMap.

Pod Scheduling Fails: "Insufficient nvidia.com/mig-X"

Symptom: Pod stuck in Pending state with event:

0/3 nodes are available: 3 Insufficient nvidia.com/mig-2g.10gb.

Root Causes and Solutions:

Cause 1: Nodes have different MIG profile

  1. Check what profiles are available on nodes

    $ oc get nodes -o custom-columns=NAME:.metadata.name,MIG-1G:.status.allocatable.nvidia\\.com/mig-1g\\.5gb,MIG-2G:.status.allocatable.nvidia\\.com/mig-2g\\.10gb
NAME                       MIG-1G   MIG-2G
worker-gpu-0.example.com   7        <none>
worker-gpu-1.example.com   <none>   3

    Solution: Either: * Update pod resource request to match available profile (nvidia.com/mig-1g.5gb: 1) * Relabel node to desired profile (triggers 10-20 min reconfiguration) * Add nodeSelector to target specific node with correct profile

Cause 2: All MIG instances already allocated

  1. Check if MIG instances are exhausted

    $ oc describe node worker-gpu-0.example.com | grep -A 2 "Allocated resources:"
Allocated resources:
  nvidia.com/mig-1g.5gb:  7  (100% of capacity)

    Solution: Either: * Wait for running pods to complete and release instances * Add more GPU nodes to cluster * Scale down lower-priority workloads

Cause 3: Using single strategy with profile-specific request

If ClusterPolicy has mig.strategy: single, pods must request nvidia.com/gpu: 1, NOT nvidia.com/mig-*.

  1. Check MIG strategy

    $ oc get clusterpolicy gpu-cluster-policy -o jsonpath='{.spec.mig.strategy}'
single

    Solution: Change pod resource request from nvidia.com/mig-2g.10gb: 1 to nvidia.com/gpu: 1

MIG Reconfiguration Takes Longer Than 20 Minutes

Symptom: Node labeled 30+ minutes ago but mig.config.state still shows pending

Diagnosis:

  1. Check if node has running GPU workloads

    $ oc get pods --all-namespaces -o wide | grep worker-gpu-0.example.com | grep -E "nvidia.com/(gpu|mig)"

    If GPU workloads are running, MIG Manager cannot reconfigure until they terminate.

  2. Drain node to evict GPU workloads

    $ oc adm drain worker-gpu-0.example.com --ignore-daemonsets --delete-emptydir-data

    Wait for pod eviction (5-10 minutes), then check mig.config.state again.

  3. Check MIG Manager pod status

    $ oc get pods -n nvidia-gpu-operator -l app=nvidia-mig-manager
NAME                         READY   STATUS    RESTARTS   AGE
nvidia-mig-manager-xxxxx     1/1     Running   3          45m

    If RESTARTS > 0 and increasing, check logs for crash loop:

    $ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager --previous

Verification: nvidia-smi Shows Different Instance Count Than Expected

Symptom: Applied all-1g.5gb expecting 7 instances but nvidia-smi -L shows 3 instances

Possible Causes:

  1. Node has A100-80GB instead of A100-40GB: Profile all-1g.5gb doesn’t exist for 80GB model

    • Solution: Use all-1g.10gb for A100-80GB GPUs

  2. Custom ConfigMap overriding built-in profile: Your ConfigMap may define all-1g.5gb differently

    • Solution: Check ConfigMap content, remove custom definition to use auto-generated profile

  3. MIG reconfiguration incomplete: Device Plugin hasn’t updated yet

    • Solution: Wait full 20 minutes, check nvidia.com/mig.config.state=success

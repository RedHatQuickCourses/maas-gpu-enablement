2.1: Understanding subscription-based governance

Before we configure OpenShift groups or write any YAML, we must define the core mechanism that makes OpenShift AI 3.4 governance possible: Subscriptions.

In a traditional, decentralized AI setup, any user with access to an API endpoint can theoretically send massive, computationally expensive prompts to an LLM. This leads to immediate resource exhaustion and makes cross-departmental chargebacks impossible.

Models as a Service solves this by placing a strict, subscription-based governance layer between the user and the underlying KServe infrastructure.

What is a Subscription?

A subscription is a logical boundary that controls which AI models a user can access, how much compute they can consume, and how their usage is tracked.

When multiple business units share expensive large language models, MaaS subscriptions serve four critical enterprise functions:

The Strategic Capability The Enterprise Value

Preventing Resource Exhaustion

Generative AI compute costs are directly correlated to sequence length. Subscriptions enforce token limits, mathematically preventing a single team’s runaway script from overwhelming the cluster.

Differential Service Levels

Subscriptions allow you to prioritize mission-critical workloads. The system evaluates a user’s subscription priority level when handling simultaneous requests.

Granular Cost Tracking

Finance teams can track exact consumption metrics (prompt and completion tokens) based on subscription assignments, enabling highly accurate departmental billing.

Native API Key Authentication

Subscriptions eliminate the need for complex, manual Kubernetes ServiceAccounts. Developers can generate secure, revocable API keys directly through the Gen AI Studio UI.

The Two-Part Access Rule

In OpenShift AI 3.4, granting a team access to the AI platform requires exactly two custom resources working together. Think of it like joining a gym:

  1. The MaaSSubscription (The Membership Contract): This defines the terms. It states that the "Marketing Group" belongs to the "Bronze Plan," which grants them access to the nemotron-8b model with a limit of 1,000 tokens per minute.

  2. The MaaSAuthPolicy (The Keycard Reader): This is the physical enforcement mechanism. It attaches to the OpenShift API Gateway and says, "If someone presents a valid API key belonging to the Marketing Group, let them through the door."

If you deploy a Subscription without an Auth Policy, the user’s API key will be rejected at the network edge with a 403 Forbidden error.

The ChocoTech Governance Strategy

To solve ChocoTech’s GPU Silo Crisis, the CIO has mandated a custom subscription mapping tailored to the company’s business units.

In the upcoming sections, you will build the following structure using MaaSSubscription resources:

  • Gold Plan: Mapped to the chocotech-quality-control group. This tier will have the highest priority and massive token limits to ensure the factory floor’s defect detection models never experience throttling.

  • Silver Plan: Mapped to the chocotech-rd group. This tier will have moderate token limits for recipe optimization testing.

  • Bronze Plan: Mapped to the chocotech-marketing group. This tier will have strict token limits to prevent their exploratory sentiment analysis from wasting expensive GPU cycles.

Now that you understand the strategy and enforcement mechanics, proceed to the next section to begin building these OpenShift groups in the cluster.

Hands-On Lab: Deploy and Configure Models-as-a-Service

Estimated completion time: 45 minutes.

Objective

In this hands-on lab, you will deploy and configure the complete Models-as-a-Service (MaaS) platform for ChocoTech Industries. You will enable the MaaS backend component, configure TLS for secure authentication, create organizational groups, publish models to the MaaS gateway, establish subscription-based governance with token limits, configure authorization policies, and generate API keys for secure model access. By the end of this lab, you will have a production-ready MaaS platform that enforces the Gold, Silver, and Bronze subscription plans required by ChocoTech’s CIO.

Lab Scenario

Recall ChocoTech’s business requirements from the CIO:

Business Unit Subscription Plan Infrastructure Requirement

Quality Control

Gold Plan

Production-grade access with guaranteed availability and highest token limits for real-time defect detection.

R&D

Silver Plan

Development access with moderate token limits for recipe optimization testing.

Marketing

Bronze Plan

Exploratory access with strict token limits for sentiment analysis to prevent budget overruns.

Your mission: Build the governance infrastructure to deliver these three distinct service levels using MaaS subscriptions, authorization policies, and API key authentication.

Step 5: Verify User Workload Monitoring

User Workload Monitoring is critical for MaaS. Without it, the Tenant resource will show Degraded status and MaaS metrics will not be collected.

Check if User Workload Monitoring is enabled:

oc get configmap cluster-monitoring-config -n openshift-monitoring -o jsonpath='{.data.config\.yaml}' | grep enableUserWorkload

Expected output:

enableUserWorkload: true

Verify User Workload Monitoring pods are running:

oc get pods -n openshift-user-workload-monitoring

Expected output (pods Running):

NAME                                   READY   STATUS    AGE
prometheus-operator-5f8d9c7b6d-q5k8m   2/2     Running   5d
prometheus-user-workload-0             6/6     Running   5d
prometheus-user-workload-1             6/6     Running   5d
thanos-ruler-user-workload-0           4/4     Running   5d
thanos-ruler-user-workload-1           4/4     Running   5d

If User Workload Monitoring is not enabled, the MaaS Tenant will show Degraded status with the error:

User Workload Monitoring is not enabled

Enable User Workload Monitoring:

cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF

Wait 2-3 minutes for the openshift-user-workload-monitoring namespace to be created and pods to start.

Step 6: Verify PostgreSQL Database Secret

MaaS requires a PostgreSQL database for API key storage and subscription metadata.

Check for database secret:

oc get secret maas-db-config -n redhat-ods-applications

Expected output:

NAME             TYPE     DATA   AGE
maas-db-config   Opaque   1      5d

Verify secret contains database connection string:

oc get secret maas-db-config -n redhat-ods-applications -o jsonpath='{.data.connection-string}' | base64 -d | cut -c1-20

Expected output (first 20 characters):

postgresql://maas_us

The database secret should contain a connection string in the format:

postgresql://<username>:<password>@<host>:<port>/<database>?sslmode=require

Your cluster administrator has provisioned this PostgreSQL database during the initial OpenShift AI setup. The database stores:

  • API keys and their metadata (owner, subscription, expiration)

  • Subscription configurations

  • Token consumption metrics (for observability)

Troubleshooting:

If the secret is missing, contact your cluster administrator. The database and secret are prerequisites that must be configured before enabling MaaS.

Step 7: Verify Gateway API Resources

MaaS uses OpenShift’s Gateway API for external model access routing.

Check GatewayClass:

oc get gatewayclass openshift-gateway-api

Expected output:

NAME                     CONTROLLER                        ACCEPTED   AGE
openshift-gateway-api    openshift.io/gateway-controller   True       5d

Check MaaS Gateway:

oc get gateway maas-default-gateway -n openshift-ingress

Expected output:

NAME                   CLASS                    ADDRESS   PROGRAMMED   AGE
maas-default-gateway   openshift-gateway-api              True         5d

Verify Gateway is programmed:

oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.conditions[?(@.type=="Programmed")].status}'

Expected output: True

If the Gateway does not exist, your cluster administrator must create it. The Gateway defines the external API endpoint for MaaS model access.

Example Gateway configuration:

apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: maas-default-gateway
  namespace: openshift-ingress
spec:
  gatewayClassName: openshift-gateway-api
  listeners:
  - name: https
    port: 443
    protocol: HTTPS
    hostname: "maas.apps.<cluster-domain>"
    tls:
      mode: Terminate
      certificateRefs:
      - kind: Secret
        name: maas-tls-cert

Consult the OpenShift AI documentation for complete Gateway setup instructions.

Part 2: Enable and Deploy MaaS

Now that prerequisites are verified, enable the MaaS backend component.

Step 1: Enable MaaS in DataScienceCluster

Enable the MaaS backend by patching the DataScienceCluster custom resource:

oc patch datasciencecluster default -n redhat-ods-applications \
  --type merge \
  -p '{"spec":{"components":{"kserve":{"modelsAsService":{"managementState":"Managed"}}}}}'

Expected output:

datasciencecluster.datasciencecluster.opendatahub.io/default patched

What happens:

  1. The OpenShift AI operator detects the configuration change (reconciliation occurs every 30 seconds)

  2. Controller creates the maas-api Deployment in the redhat-ods-applications namespace

  3. MaaS API service starts listening on port 8080 for subscription and API key management

  4. Tenant custom resource default-tenant is automatically created in models-as-a-service namespace

  5. ModelsAsService custom resource default-modelsasservice is automatically created cluster-wide

Automatic Resource Creation: The default-tenant and default-modelsasservice resources are automatically created by the MaaS operator when you enable MaaS. You do not need to create these manually.

  • default-tenant - Singleton tenant resource that configures API key expiration, telemetry, and external OIDC settings

  • default-modelsasservice - Cluster-wide resource that configures platform-wide MaaS settings

While it’s technically possible to create additional Tenant resources via YAML (since Tenant is a CRD), OpenShift AI 3.4 MaaS is designed for single-tenant operation. The documentation only references default-tenant.

MaaS Custom Resource Definitions Explained:

CRD Purpose

MaaSSubscription

Defines subscription-based access control with token limits and group assignments

MaaSAuthPolicy

Authorizes groups to access model endpoints through the API gateway

MaaSModelRef

References models deployed in OpenShift AI and published to MaaS

ExternalModel (Tech Preview)

Routes requests to external cloud providers (OpenAI, Anthropic, AWS Bedrock, Azure OpenAI, Google Vertex AI) through MaaS governance

Tenant

Configures tenant-specific settings including API key expiration, external OIDC, and telemetry

ExternalModel Support: The externalmodels.maas.opendatahub.io CRD enables routing to external LLM providers without deploying additional containers. When you create an ExternalModel resource, the existing maas-api pod handles request routing:

  1. User authenticates with MaaS API key

  2. MaaS validates subscription and token limits

  3. MaaS injects the provider API key from a Kubernetes secret

  4. Request is routed through Istio ServiceMesh to the external provider (auto-generated ServiceEntry/DestinationRule)

No additional pods are deployed for external models. This is a Technology Preview feature in OpenShift AI 3.4.

Check the Tenant resource status:

oc get tenant default-tenant -n models-as-a-service -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'

Expected output: True

If the status shows False, check the condition message:

oc get tenant default-tenant -n models-as-a-service -o jsonpath='{.status.conditions[?(@.type=="Ready")].message}'

If the Tenant shows Degraded status, the most common cause is User Workload Monitoring not being enabled. Verify Step 4 in Part 1 and wait 2-3 minutes for the monitoring stack to stabilize.

Step 3: Configure TLS for Authorino

MaaS uses Authorino (from Kuadrant) for authentication. Configure TLS for secure communication between Authorino and the MaaS API:

oc -n kuadrant-system set env deployment/authorino \
  SSL_CERT_FILE=/etc/ssl/certs/openshift-service-ca/service-ca-bundle.crt \
  REQUESTS_CA_BUNDLE=/etc/ssl/certs/openshift-service-ca/service-ca-bundle.crt

Expected output:

deployment.apps/authorino updated

Annotate the Gateway to enable automatic TLS configuration:

oc annotate gateway maas-default-gateway \
  -n openshift-ingress \
  security.opendatahub.io/authorino-tls-bootstrap="true" \
  --overwrite

Verify the TLS configuration:

oc get deployment/authorino -n kuadrant-system -o jsonpath='{.spec.template.spec.containers[0].env[?(@.name=="SSL_CERT_FILE")].value}'

Expected output:

/etc/ssl/certs/openshift-service-ca/service-ca-bundle.crt

Wait ~15 seconds for the Authorino pod to restart with the new environment variables.

  -p '{
    "spec": {

#!/bin/bash
for i in {1..25}; do

Expected behavior:

  • First 10-12 requests: HTTP 200 (success)

  • After ~200 tokens consumed: HTTP 429 (rate limited)

  • Response body on 429:

{
  "error": {
    "message": "Token limit exceeded. You have consumed the maximum number of tokens allowed for this model in your subscription.",
    "type": "token_limit_error",
    "code": 429
  }
}

This demonstrates the MaaS token governance in action. The Gold subscription allows 200 tokens/minute. Once you exceed this limit, MaaS returns HTTP 429 until the time window resets. In production applications, implement exponential backoff retry logic to handle these scenarios gracefully.

Common Issues and Solutions

Issue: 403 Forbidden when accessing model

Cause: Missing authorization policy or user not in authorized group.

Solution:

# Verify authorization policy exists
oc get maasauthpolicy -n models-as-a-service | grep gold

# Verify your group membership
oc get group quality-control -o jsonpath='{.users}' | grep $(oc whoami)

If the authorization policy is missing, create it from the Subscriptions page by editing the subscription and ensuring the matching authorization policy option is enabled.

Issue: 401 Unauthorized

Cause: Invalid or expired API key.

Solution:

  • Verify the API key is prefixed with sk-oai-

  • Check expiration date in Gen AI Studio → API Keys

  • Regenerate a new API key if expired

Issue: Model not appearing in subscription model list

Cause: Model was not published to MaaS.

Solution:

# Check if MaaSModelRef exists
oc get maasmodelref -n chocotech-models

# If missing, edit the LLMInferenceService and ensure publishAsMaaS: true
oc edit llminferenceservice chocotech-defect-detection -n chocotech-models

Issue: Subscription shows Failed phase

Cause: Invalid model reference or group name.

Solution:

# Check subscription status conditions
oc describe maassubscription gold-quality-control -n models-as-a-service

# Review the status.conditions for error messages

Common failures include referencing a model that doesn’t exist or a group that isn’t a valid OpenShift group.

1.3: Enabling MaaS on your OpenShift AI platform

You have explored the business problem and understand the three-layer architecture. Now it is time to activate MaaS on ChocoTech’s OpenShift cluster.

ChocoTech’s GPU infrastructure is already operational from prior AI/ML experiments. Rather than starting from scratch, you will transform this fragmented hardware into a centralized MaaS platform by enabling the components that provide governance, routing, and multi-tenant model serving.

This module guides you through the exact activation sequence: verifying required operators, creating the infrastructure foundation, deploying the required database, and patching the OpenShift AI platform to expose the MaaS features.

Step 1: Configuration check & operator verification

Before applying any configurations, ensure your cluster meets the baseline requirements. You must be running OpenShift version 4.20.0 (officially 4.19.9) or later. Furthermore, specific operators must be installed from the Ecosystem - Software Catalog (previously OperatorHub).

If any of these are missing, the deployment will silently fail—your commands will appear to succeed, but the expected pods and resources will never materialize.

Ensure the following operators are installed and healthy:

  • Red Hat OpenShift Service Mesh 3: Provides the underlying Istio networking that powers the Gateway API implementation.

  • Red Hat Connectivity Link (v1.2+): Delivers the Kuadrant governance framework.

  • Red Hat OpenShift AI (v3.4+): The core platform operator that manages the DataScienceCluster.

  • Leader Worker Set (LWS) Operator: Mandatory for llm-d orchestration to manage models that exceed single-node VRAM capacity.

Verify the installations:

oc get csv -n openshift-operators | grep -E "(servicemesh|rhcl|rhods|limitador|authorino)"

If any operators are missing, install them from the OpenShift console (Ecosystem → Software Catalog) and wait for their status to reach Succeeded before continuing.

Step 2: The infrastructure foundation

The OpenShift AI operator acts as a strict gatekeeper. It will not provision the MaaS API unless the foundational network routing, security engines, and databases are actively running on the cluster.

Initialize the Governance Engine (Kuadrant)

The Connectivity Link operator manages Kuadrant, but Kuadrant itself must be explicitly initialized. A Kuadrant custom resource (CR) must be created and ready in the kuadrant-system namespace.

Create the namespace and Kuadrant instance:

oc create namespace kuadrant-system

cat <<EOF | oc apply -f -
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata:
  name: kuadrant
  namespace: kuadrant-system
EOF

Create the MaaS Gateway

Next, we establish the Front Door. A Gateway named maas-default-gateway must exist in the openshift-ingress namespace. This becomes the single API endpoint for all of ChocoTech’s model inference traffic.

Create the Gateway resource:

cat <<EOF | oc apply -f -
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: maas-default-gateway
  namespace: openshift-ingress
spec:
  gatewayClassName: data-science-gateway-class
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    allowedRoutes:
      namespaces:
        from: All
EOF

Deploy the MaaS API Database

In OpenShift AI 3.4, the platform features native API key management. To store these keys securely, you must have a PostgreSQL database and create a Secret named maas-db-config in the redhat-ods-applications namespace.

Deploy a lightweight PostgreSQL instance and the required secret for the MaaS API:

cat <<EOF | oc apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: maas-postgres
  namespace: redhat-ods-applications
spec:
  replicas: 1
  selector:
    matchLabels:
      app: maas-postgres
  template:
    metadata:
      labels:
        app: maas-postgres
    spec:
      containers:
      - name: postgres
        image: registry.redhat.io/rhel9/postgresql-15:latest
        env:
        - name: POSTGRESQL_USER
          value: "maasuser"
        - name: POSTGRESQL_PASSWORD
          value: "maaspass"
        - name: POSTGRESQL_DATABASE
          value: "maasdb"
        ports:
        - containerPort: 5432
---
apiVersion: v1
kind: Service
metadata:
  name: maas-postgres
  namespace: redhat-ods-applications
spec:
  selector:
    app: maas-postgres
  ports:
  - port: 5432
    targetPort: 5432
---
apiVersion: v1
kind: Secret
metadata:
  name: maas-db-config
  namespace: redhat-ods-applications
type: Opaque
stringData:
  DB_CONNECTION_URL: 'postgresql://maasuser:maaspass@maas-postgres:5432/maasdb?sslmode=disable'
EOF

Step 3: Platform activation

With the operators verified, Kuadrant running, the Gateway established, and the database online, you can now instruct the OpenShift AI operator to build the MaaS API layer.

Enable the MaaS Backend and Llama Stack

You must edit the default-dsc CR to enable two critical components: 1. Models As a Service: Provisions the core governance and routing controllers. 2. Llama Stack Operator: Provides the required API layer for the Gen AI Studio UI to function.

Apply the patch:

oc patch datasciencecluster default-dsc --type=merge -p '{
  "spec": {
    "components": {
      "llamastackoperator": {
        "managementState": "Managed"
      },
      "kserve": {
        "managementState": "Managed",
        "modelsAsService": {
          "managementState": "Managed"
        }
      }
    }
  }
}'

Verify that both the maas-api and maas-controller-manager pods are running:

oc get pods -n redhat-ods-applications | grep maas

Enable the Dashboard (Gen AI Studio)

Even when deploying models via GitOps, administrators require the dashboard UI to manage user access tokens and monitor model usage. You have enabled Gen AI Studio: spec.dashboardConfig.genAiStudio: true in the OdhDashboardConfig custom resource.

Apply the dashboard configuration patch:

oc patch odhdashboardconfig odh-dashboard-config -n redhat-ods-applications --type=merge -p '{
  "spec": {
    "dashboardConfig": {
      "modelAsService": true,
      "genAiStudio": true,
      "maasAuthPolicies": true
    }
  }
}'

Refresh your OpenShift AI browser window. The Gen AI Studio menu option will now appear in your left-hand navigation pane.

ChocoTech’s underlying infrastructure is now primed for centralized governance. Proceed to the next module to configure the explicit policies and subscription definitions!

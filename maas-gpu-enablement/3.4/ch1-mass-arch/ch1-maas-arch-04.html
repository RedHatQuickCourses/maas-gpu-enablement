1.3: Enabling MaaS on your OpenShift AI platform

You have explored the business problem and understand the three-layer architecture. Now it is time to activate MaaS on ChocoTech’s OpenShift cluster. ChocoTech’s GPU infrastructure is already operational from prior AI/ML experiments. Rather than starting from scratch, you will transform this fragmented hardware into a centralized MaaS platform by enabling the components that provide governance, routing, and multi-tenant model serving.

This module guides you through the exact activation sequence: verifying required operators, creating the infrastructure foundation, deploying the required database, and patching the OpenShift AI platform to expose the MaaS features.

Step 1: Configuration check & operator verification

Before applying any configurations, ensure your cluster meets the baseline requirements. You must be running OpenShift version 4.20.0 (officially 4.19.9) or later. Furthermore, specific operators and monitoring stacks must be installed. If any of these are missing, the deployment will silently fail.

Ensure the following operators and settings are active:

  • User Workload Monitoring: This is absolutely critical for MaaS. Without it, the MaaS Tenant resource will show a Degraded status, and the platform will fail to collect token consumption metrics.

  • Red Hat OpenShift Service Mesh 3: Provides the underlying Istio networking that powers the Gateway API implementation.

  • Red Hat Connectivity Link (v1.2+): Delivers the Kuadrant governance framework.

  • Red Hat OpenShift AI (v3.4+): The core platform operator that manages the DataScienceCluster.

  • Leader Worker Set (LWS) Operator: Mandatory for llm-d orchestration to manage models that exceed single-node VRAM capacity.

  • Cluster Observability Operator (Optional but Recommended): While MaaS core routing works without it, ChocoTech’s Finance team needs this to view the MaaS observability dashboard (Tech Preview) for token consumption tracking and showback reporting.


Need a quick sanity check, click & run this: 
oc get csv -n openshift-lws-operator | grep -E "(servicemesh|rhcl|rhods|limitador|authorino|leader)"

If any operators are missing, install them from the OpenShift console (Ecosystem → Software Catalog) and wait for their status to reach Succeeded before continuing.

Step 2: The infrastructure foundation

The OpenShift AI operator acts as a strict gatekeeper. It will not provision the MaaS API unless the foundational network routing, security engines, and databases are actively running on the cluster.

Initialize the Governance Engine (Kuadrant)

The Connectivity Link operator manages Kuadrant (Authorino & Limitador), but Kuadrant itself must be explicitly initialized. A Kuadrant custom resource (CR) must be created and ready in the kuadrant-system namespace.

  • Authorino: Validates tokens and enforces zero-trust authentication.

  • Limitador: Tracks token consumption and enforces subscription-based token limits.

Create the namespace and Kuadrant instance:

oc create namespace kuadrant-system

cat <<EOF | oc apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: kuadrant-system
---
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata:
  name: kuadrant
  namespace: kuadrant-system
spec: {}
EOF

Step 3: Enable User Workload Monitoring:

User Workload Monitoring is an OpenShift cluster setting that enables the Models as a Service (MaaS) platform to collect and expose usage data

  • It is required for the following reasons:

    • Tracking Consumption: It is the underlying mechanism that tracks token consumption, request counts, and rate limiting across the platform.

    • Without it, the platform cannot collect the token metrics necessary for enforcing subscription quotas or providing Finance teams with accurate chargeback data.

    • Platform Health: If User Workload Monitoring is not enabled, the core MaaS installation and the centralized Tenant custom resource will fail to function properly and will display a Degraded status.

cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF

Verify User Workload Monitoring pods are running:

$ oc get pods -n openshift-user-workload-monitoring

Expected output (pods Running):

NAME                                   READY   STATUS    AGE
prometheus-operator-6bd7f9ddfc-vn49s   2/2     Running    0
prometheus-user-workload-0             6/6     Running    0
thanos-ruler-user-workload-0           4/4     Running    0

Step 4: Create the MaaS Gateway

Next, we establish the Front Door. A Gateway named maas-default-gateway must exist in the openshift-ingress namespace. This becomes the single API endpoint for all of ChocoTech’s model inference traffic.

Create the Gateway resource:

cat <<EOF | oc apply -f -
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: maas-default-gateway  (2)
  namespace: openshift-ingress
  annotations:
    # Mandatory for MaaS to bootstrap TLS and Authorino integration
    security.opendatahub.io/authorino-tls-bootstrap: "true"
    # Mandatory to allow maas-controller to manage policies and prevent overwriting
    opendatahub.io/managed: "false". (3)
spec:
  gatewayClassName: data-science-gateway-class  # Alternatively, use 'openshift-default' (1)
  listeners:
  - name: https
    port: 443
    protocol: HTTPS
    tls:
      mode: Terminate
      certificateRefs:
      - name: router-certs-default # This Secret MUST exist in the openshift-ingress namespace. Change to 'router-certs-default' to use the cluster default.
    allowedRoutes:
      namespaces:
        from: All
EOF
1 GatewayClass (The Blueprint): This tells the cluster who is responsible for building the load balancer and what kind of underlying technology to use. By specifying gatewayClassName: data-science-gateway-class, you instruct the native OpenShift Gateway Controller to deploy the physical proxy infrastructure. If this is misconfigured or missing, your cluster won’t know how to build the gateway, and it will sit in an unprogrammed state forever.
2 Gateway (The Entry Endpoint): The Gateway resource (which we named maas-default-gateway) is the instantiated infrastructure that listens for traffic. It defines the physical or virtual IP address, the open ports (like 80 or 443), and the TLS certificates. It essentially says, "I am built according to the data-science-gateway-class blueprint, and I am actively listening for connections."
3 This (opendatahub.io/managed: "false") annotation allows the maas-controller to manage the AuthPolicies and prevents the Open Data Hub (ODH) Model Controller from overwriting your gateway configurations.

Understanding the Gateway API architecture

  • HTTPRoute (The Router): This is where the actual routing logic lives. An HTTPRoute attaches to a Gateway and contains the specific URL rules (e.g., "If a request path is /llm/llama-3b/v1/chat/completions, forward it to the model pod in the chocotech-models namespace").

How it all ties together in MaaS

The benefit of this architecture is the separation of infrastructure from application logic. As a platform engineer, you only build the gateway.

ChocoTech deploys a model and checks the "Publish as MaaS" box, they do not need to write any networking rules. The MaaS controller automatically generates the HTTPRoute behind the scenes. That route dynamically attaches itself to the maas-default-gateway, securely linking the public endpoint through the Kuadrant governance layer directly to the internal KServe model pod.

Step 5: Configure Authorino Environment

The OpenShift service-ca-operator automatically handles the population of the cluster CA bundle. However, you are responsible for annotating the Gateway to trigger the automatic creation of the required EnvoyFilter for TLS. To get Models-as-a-Service (MaaS) running, you must complete two main configuration steps: setting the Authorino environment and annotating the Gateway.

First, you must configure the Authorino deployment with SSL environment variables so it can communicate securely with the MaaS API.

# Set SSL environment variables for outbound communication
oc -n kuadrant-system set env deployment/authorino \
SSL_CERT_FILE=/etc/ssl/certs/openshift-service-ca/service-ca-bundle.crt \
REQUESTS_CA_BUNDLE=/etc/ssl/certs/openshift-service-ca/service-ca-bundle.crt

  1. Annotate the Gateway Instead of manually generating a new certificate within the Gateway YAML, you apply an annotation that enables the platform to bootstrap the TLS connection automatically

# Annotate the existing gateway to enable TLS bootstrap
oc annotate gateway maas-default-gateway \
-n openshift-ingress \
security.opendatahub.io/authorino-tls-bootstrap="true" \
--overwrite

  1. Verification Commands After running the code above, verify that the configuration was applied correctly to ensure the MaaS deployment doesn’t show a "Degraded" status.

Verify Authorino SSL Variables:

oc get deployment/authorino -n kuadrant-system -o jsonpath='{.spec.template.spec.containers[0].env[?(@.name=="SSL_CERT_FILE")].value}'

Expected output:

/etc/ssl/certs/openshift-service-ca/service-ca-bundle.crt

Verify Gateway Annotation:oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.metadata.annotations.security\.opendatahub\.io/authorino-tls-bootstrap}'

Expected output: true

Check Tenant Readiness:

oc get tenant default-tenant -n models-as-a-service -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'

Expected output: True

Note on Certificates: You must ensure your cluster already has a functional ingress controller with valid TLS certificates for external access before starting this procedure. If you are missing the maas-db-config secret or User Workload Monitoring, the Tenant resource will likely remain in a False or Degraded state.

Step 6: Deploy the MaaS API Database

In OpenShift AI 3.4, the platform features native API key management. To store these keys securely, you must have a PostgreSQL database and create a Secret named maas-db-config in the redhat-ods-applications namespace.

Deploy a lightweight PostgreSQL instance and the required secret for the MaaS API:

cat <<EOF | oc apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: maas-postgres
  namespace: redhat-ods-applications
spec:
  replicas: 1
  selector:
    matchLabels:
      app: maas-postgres
  template:
    metadata:
      labels:
        app: maas-postgres
    spec:
      containers:
      - name: postgres
        image: registry.redhat.io/rhel9/postgresql-15:latest
        env:
        - name: POSTGRESQL_USER
          value: "maasuser"
        - name: POSTGRESQL_PASSWORD
          value: "maaspass"
        - name: POSTGRESQL_DATABASE
          value: "maasdb"
        ports:
        - containerPort: 5432
---
apiVersion: v1
kind: Service
metadata:
  name: maas-postgres
  namespace: redhat-ods-applications
spec:
  selector:
    app: maas-postgres
  ports:
  - port: 5432
    targetPort: 5432
---
apiVersion: v1
kind: Secret
metadata:
  name: maas-db-config
  namespace: redhat-ods-applications
type: Opaque
stringData:
  DB_CONNECTION_URL: 'postgresql://maasuser:maaspass@maas-postgres:5432/maasdb?sslmode=disable'
EOF

Step 7: Platform activation

With the operators verified, Kuadrant running, the Gateway established, and the database online, you can now instruct the OpenShift AI operator to build the MaaS API layer.

Enable the MaaS Backend and Llama Stack

You must edit the default-dsc CR to enable two critical components.

  1. Models As a Service: Provisions the core governance and routing controllers.

  2. Llama Stack Operator: Activating this operator is strictly required for the MaaS dashboard features to function. If you do not enable the Llama Stack Operator, developers will be entirely unable to access the Gen AI Studio menu, the API Keys management page, or the Playground for model testing.

Apply the patch:

oc patch datasciencecluster default-dsc --type=merge -p '{
  "spec": {
    "components": {
      "llamastackoperator": {
        "managementState": "Managed"
      },
      "kserve": {
        "modelsAsService": {
          "managementState": "Managed"
        }
      }
    }
  }
}'

Enable the Dashboard (Gen AI Studio)

Even when deploying models via GitOps, administrators require the dashboard UI to manage user access tokens and monitor model usage. You must enable Gen AI Studio by patching the OdhDashboardConfig custom resource.

Apply the dashboard configuration patch:

oc patch odhdashboardconfig odh-dashboard-config -n redhat-ods-applications --type=merge -p '{
  "spec": {
    "dashboardConfig": {
      "modelAsService": true,
      "genAiStudio": true,
      "maasAuthPolicies": true
    }
  }
}'

Refresh your OpenShift AI browser window. The Gen AI Studio menu option will now appear in your left-hand navigation pane.

Step 8: Verification Summary

Before moving on to building ChocoTech’s subscription plans in Module 2, run this comprehensive verification script to confirm all components deployed successfully.

cat > verify-maas-prereqs.sh <<'EOF'
#!/bin/bash

echo "=== MaaS Prerequisites Verification ==="
echo ""

# 1. OpenShift Version
echo "1. OpenShift Platform Version:"
OCP_VERSION=$(oc version -o json 2>/dev/null | jq -r '.openshiftVersion' 2>/dev/null || echo "ERROR")
echo "   Version: ${OCP_VERSION}"
[[ "${OCP_VERSION}" == "ERROR" ]] && echo "   ❌ FAILED" || echo "   ✅ PASSED"
echo ""

# 2. OpenShift AI Operator
echo "2. Red Hat OpenShift AI Operator:"
RHOAI_CSV=$(oc get csv -n redhat-ods-applications 2>/dev/null | grep rhods-operator | awk '{print $1}')
echo "   CSV: ${RHOAI_CSV:-NOT FOUND}"
[[ -z "${RHOAI_CSV}" ]] && echo "   ❌ FAILED" || echo "   ✅ PASSED"
echo ""

# 3. Red Hat Connectivity Link
echo "3. Red Hat Connectivity Link Operator:"
RCL_CSV=$(oc get csv -n openshift-operators 2>/dev/null | grep connectivity-link | awk '{print $1}')
echo "   CSV: ${RCL_CSV:-NOT FOUND}"
KUADRANT_STATUS=$(oc get kuadrant kuadrant -n kuadrant-system -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}' 2>/dev/null || echo "ERROR")
echo "   Kuadrant Ready: ${KUADRANT_STATUS}"
[[ "${KUADRANT_STATUS}" == "True" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 4. Llama Stack Operator
echo "4. Llama Stack Operator:"
LLAMA_STATE=$(oc get datasciencecluster default-dsc -o jsonpath='{.spec.components.llamastackoperator.managementState}' 2>/dev/null || echo "ERROR")
echo "   Management State: ${LLAMA_STATE}"
[[ "${LLAMA_STATE}" == "Managed" ]] && echo "   ✅ PASSED" || echo "   ⚠️  WARNING: Required for dashboard features"
echo ""

# 5. User Workload Monitoring
echo "5. User Workload Monitoring:"
UWM_ENABLED=$(oc get configmap cluster-monitoring-config -n openshift-monitoring -o jsonpath='{.data.config\.yaml}' 2>/dev/null | grep -o "enableUserWorkload: true" || echo "NOT ENABLED")
echo "   Status: ${UWM_ENABLED}"
[[ "${UWM_ENABLED}" == "enableUserWorkload: true" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 6. PostgreSQL Database Secret
echo "6. PostgreSQL Database Secret:"
DB_SECRET=$(oc get secret maas-db-config -n redhat-ods-applications -o jsonpath='{.metadata.name}' 2>/dev/null || echo "NOT FOUND")
echo "   Secret: ${DB_SECRET}"
[[ "${DB_SECRET}" == "maas-db-config" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 7. Gateway API
echo "7. Gateway API Resources:"
GW_STATUS=$(oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.conditions[?(@.type=="Programmed")].status}' 2>/dev/null || echo "NOT FOUND")
echo "   Gateway Programmed: ${GW_STATUS}"
[[ "${GW_STATUS}" == "True" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 8. KServe Component
echo "8. KServe Component (Required for MaaS):"
KSERVE_STATE=$(oc get datasciencecluster default-dsc -o jsonpath='{.spec.components.kserve.managementState}' 2>/dev/null || echo "ERROR")
echo "   Management State: ${KSERVE_STATE}"
[[ "${KSERVE_STATE}" == "Managed" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

echo "=== Summary ==="
echo "Review any ❌ FAILED items above before proceeding with MaaS deployment."
echo ""
EOF

chmod +x verify-maas-prereqs.sh
./verify-maas-prereqs.sh

If the output shows ✅ PASSED for the required components, ChocoTech’s underlying infrastructure is officially primed for centralized governance. Proceed to the Summary and troubleshooting page to close out this module.

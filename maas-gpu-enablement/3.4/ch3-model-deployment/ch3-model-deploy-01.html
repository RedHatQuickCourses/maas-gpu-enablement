Module 3: Deploying models to the MaaS catalog

This section is under development

With the governance fences securely built, it is time to bring the AI factory online.

ChocoTech’s R&D team has submitted an urgent request. "We need to deploy the Llama-3 8B model for recipe optimization," the lead data scientist explains. "But we don’t want to manage GPU drivers, Kubernetes manifests, or serving runtimes. Can this new MaaS platform handle the infrastructure for us?"

If ChocoTech used a cheap public API for this, the R&D team would be forced to send highly proprietary recipe formulations outside the corporate firewall. Furthermore, renting intelligence externally means their existing, expensive GPU hardware continues to sit idle.

By deploying models directly onto Red Hat OpenShift AI 3.3, ChocoTech keeps 100% of its intellectual property secure. Even better, OpenShift AI 3.x utilizes a disaggregated inference architecture powered by llm-d (Distributed Inference) and the vLLM engine. This maximizes throughput and minimizes latency by decoupling request handling from low-level tensor operations, ensuring ChocoTech squeezes every drop of ROI out of their hardware.

Your task is to deploy these generative AI models, configure distributed inference for multi-GPU scenarios, and publish them to the MaaS catalog where Kuadrant will automatically enforce the Gold/Silver/Bronze subscriptions you built in Module 2.

Learning objectives

By the end of this module, you will be able to:

  • Evaluate models based on architecture (dense vs. sparse) and match ChocoTech department requirements to the appropriate model types.

  • Explain the architectural difference between standalone KServe deployments and MaaS-integrated llm-d deployments.

  • Deploy a generative AI model using the LLMInferenceService custom resource.

  • Annotate models with specific tier access permissions (alpha.maas.opendatahub.io/subscriptions) to restrict access.

  • Monitor token metrics and verify that rate limit policies are actively throttling excessive requests.

The deployment strategy

Currently, only the Distributed inference with llm-d serving runtime supports MaaS integration. Deploying models using any other runtime bypasses the Gateway and Kuadrant, leaving the model completely ungoverned.

In this module, you will execute the following strategy to ensure complete governance:

Phase Action

1. Model Selection & Sizing

Analyze the hardware requirements for Llama-3 8B and Nemotron 8B, determining whether single-GPU deployment or multi-node tensor parallelism is required.

2. The llm-d Deployment

Provision the LLMInferenceService custom resources. You will configure the vLLM engine and explicitly define the publishToGenAIStudio flags to push the models to the catalog.

3. Tier Application

Apply the specific UI tier annotations directly to the model deployments. Llama-3 will be exposed to all subscriptions, while Nemotron will be restricted strictly to Gold and Silver users.

4. Day-2 Observability

Monitor the Limitador and vLLM logs to watch Kuadrant actively intercept and throttle requests as the models process real-time inference traffic.

By the end of this module, ChocoTech’s business units will finally have self-service, heavily governed access to state-of-the-art open-source models.

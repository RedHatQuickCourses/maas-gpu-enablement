3.3: Verifying the MaaS catalog and gateway publication

If ChocoTech had deployed this model using standard open-source tools, the Platform Engineering team’s job would not be finished. They would still need to manually write documentation, email the endpoint URL to the developers, and coordinate API keys.

With Models as a Service, the OpenShift AI operator handles all of this automatically. Because we set publishToGenAIStudio: true in our deployment manifest, the platform instantly registers the model, applies our governance tiers, and publishes it to a centralized self-service catalog.

Exercise 1: Verify the backend MaaS registration

When the KServe LLMInferenceService becomes ready, the MaaS operator detects the publish flag and automatically generates a companion MaaSModel custom resource. This resource acts as the bridge between the running container and the Gen AI Studio UI.

Verify the model was successfully registered with the MaaS API:

oc get maasmodel -n chocotech-models

Expected output:

NAME          MODEL                                  STATUS      AGE
llama-3-8b    meta-llama/Meta-Llama-3-8B-Instruct    Published   5m

Next, describe the resource to see how it mapped our governance rules:

oc describe maasmodel llama-3-8b -n chocotech-models

What to look for in the output: * Status: Must say Published. * Tier Access: You should see bronze-tier, silver-tier, and gold-tier explicitly listed. This confirms the "Bouncer List" we created in our YAML was successfully read. * Gateway Endpoint: The external URL that client applications will use to reach the model.

Exercise 2: The Developer Payoff (Gen AI Studio UI)

Let’s switch hats. You are now a developer on the Marketing team logging in to find a model for your sentiment analysis application. You do not need to know anything about vLLM or OpenShift routes; you simply use the self-service catalog.

  1. Open the OpenShift AI dashboard in your browser.

  2. Navigate to Gen AI Studio in the left-hand menu.

  3. Click on the Models tab.

You should immediately see the meta-llama/Meta-Llama-3-8B-Instruct model populated in the catalog grid.

Clicking on the model card will reveal: * Status: Available * Tier Access: Gold, Silver, Bronze * Endpoint: https://maas-default-gateway…​/v1/chat/completions

This completely eliminates developer friction. They can discover models, view their tier eligibility, and copy the exact endpoint URL without ever submitting an IT ticket.

Exercise 3: Test the Gateway API discovery

Finally, let’s verify that the Kuadrant Gateway is correctly broadcasting the model’s availability over the network. The MaaS Gateway is fully compliant with the OpenAI API specification, meaning client libraries can dynamically query the /v1/models endpoint to see what is available.

(Note: The /v1/models endpoint is used for discovery and does not require a Bearer token).

Retrieve your Gateway hostname and query the models list:

GATEWAY_HOST=$(oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.addresses[0].value}')

curl -s http://$GATEWAY_HOST/v1/models | jq

Decoding the Results

If MaaS integration is successful, the Gateway will return a JSON array containing your model:

{
  "object": "list",
  "data": [
    {
      "id": "meta-llama/Meta-Llama-3-8B-Instruct",
      "object": "model",
      "created": 1714345678,
      "owned_by": "chocotech-models"
    }
  ]
}

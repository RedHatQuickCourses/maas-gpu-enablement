3.4: Expanding the catalog with selective access

This section is under development

ChocoTech’s Marketing team requests access to a smaller, faster model for sentiment analysis. To accommodate this, we will deploy a second model to demonstrate multi-model catalog management.

This exercise shows how easily you can expand the model catalog as new business requirements emerge. More importantly, it demonstrates how to apply strict, selective access controls so that expensive or restricted models do not appear for unauthorized users.

Exercise 1: Stage the second model vault

Just like we did for Llama-3, we need to stage the weights for the nvidia/nemotron-4-8b-instruct model in our local MinIO vault (s3://models/nemotron).

Run the provided setup script to execute the Kubernetes Job that pulls the model from Hugging Face into your cluster storage:

./deploy/setup-vault-nemotron.sh

Expected Output: The script will output its progress. Wait for the final ✅ SUCCESS: Model is now available in MinIO at s3://models/nemotron message.

Exercise 2: The deployment manifest

We will use the exact same LLMInferenceService structure as before to deploy the model and automatically generate its MaaSModelRef.

Review and apply the nemotron-deployment.yaml manifest:

cat <<EOF | oc apply -f -
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: nemotron-8b
  namespace: chocotech-models
spec:
  publishToGenAIStudio: true
  model:
    uri: s3://models/nemotron
    name: nvidia/nemotron-4-8b-instruct
  servingRuntime: "vllm-cuda-runtime"
  placement:
    accelerator:
      name: "nvidia.com/gpu"
      count: 1
---
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSModelRef
metadata:
  name: nemotron-8b-ref
  namespace: chocotech-models
spec:
  modelRef:
    kind: LLMInferenceService
    name: nemotron-8b
EOF

Wait for the deployment to become ready:

oc wait --for=condition=ready llminferenceservice/nemotron-8b -n chocotech-models --timeout=600s

Expected output: llminferenceservice.serving.kserve.io/nemotron-8b condition met.

Exercise 3: Post-Deployment Governance (Selective Access)

Currently, the Nemotron model is deployed and registered as a MaaSModelRef, but no one has access to it. We want to restrict this model strictly to the Silver (R&D) and Bronze (Marketing) subscriptions. The Gold (Quality Control) team should not have access to it, as it is not validated for factory floor workloads.

In OpenShift AI 3.4, access is enforced entirely by MaaSSubscription and MaaSAuthPolicy resources. To grant access, we must append the new model reference to our existing governance manifests.

  1. First, append the model to the Silver and Bronze MaaSSubscription resources to grant a token quota:

    cat <<EOF | oc apply -f -
---
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSSubscription
metadata:
  name: silver-plan
  namespace: models-as-a-service
spec:
  priority: 50
  owner:
    groups:
      - name: chocotech-rd
    users: []
  modelRefs:
    - name: llama-3-8b-ref
      namespace: chocotech-models
      tokenRateLimits:
        - limit: 10000
          window: 1m
    # NEWLY ADDED MODEL QUOTA
    - name: nemotron-8b-ref
      namespace: chocotech-models
      tokenRateLimits:
        - limit: 20000
          window: 1m
---
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSSubscription
metadata:
  name: bronze-plan
  namespace: models-as-a-service
spec:
  priority: 10
  owner:
    groups:
      - name: chocotech-marketing
    users: []
  modelRefs:
    - name: llama-3-8b-ref
      namespace: chocotech-models
      tokenRateLimits:
        - limit: 2000
          window: 1m
    # NEWLY ADDED MODEL QUOTA
    - name: nemotron-8b-ref
      namespace: chocotech-models
      tokenRateLimits:
        - limit: 8000
          window: 1m
EOF

  2. Next, append the model to the Silver and Bronze MaaSAuthPolicy resources to explicitly open the API Gateway:

    cat <<EOF | oc apply -f -
---
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSAuthPolicy
metadata:
  name: silver-access
  namespace: models-as-a-service
spec:
  modelRefs:
    - name: llama-3-8b-ref
      namespace: chocotech-models
    # NEWLY ADDED MODEL ACCESS
    - name: nemotron-8b-ref
      namespace: chocotech-models
  subjects:
    groups:
      - name: chocotech-rd
    users: []
---
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSAuthPolicy
metadata:
  name: bronze-access
  namespace: models-as-a-service
spec:
  modelRefs:
    - name: llama-3-8b-ref
      namespace: chocotech-models
    # NEWLY ADDED MODEL ACCESS
    - name: nemotron-8b-ref
      namespace: chocotech-models
  subjects:
    groups:
      - name: chocotech-marketing
    users: []
EOF

The Gold Plan (Quality Control) remains completely untouched, meaning they are mathematically prohibited from routing traffic to Nemotron.

Exercise 4: Verify the multi-model catalog

1. The UI Payoff Refresh the Gen AI Studio Models tab in the dashboard.

  • If you are logged in as a member of the Marketing or R&D OpenShift group, you should now see both models listed with their respective subscription access controls.

  • If you switch to a user in the Quality Control group, the Nemotron model will vanish from the catalog entirely.

2. API Discovery (The Developer View) Retrieve your Gateway hostname and query the /v1/models endpoint using your Bronze Plan API Key:

curl -s -k https://$GATEWAY_HOST/v1/models \
  -H "Authorization: Bearer $MAAS_API_KEY" | jq

Expected output showing both models:

{
  "object": "list",
  "data": [
    {
      "id": "meta-llama/Meta-Llama-3-8B-Instruct",
      "object": "model",
      "owned_by": "chocotech-models"
    },
    {
      "id": "nvidia/nemotron-4-8b-instruct",
      "object": "model",
      "owned_by": "chocotech-models"
    }
  ]
}

(Note: If you run this exact same API call using a Gold Plan API Key, the JSON array will only return Llama-3).

ChocoTech now operates a fully governed, multi-model AI factory. Quality Control relies on the universally available Llama-3, while Marketing and R&D can utilize the Nemotron model without risking exposure to the Gold tier production workloads.

3.2: Deploying the governed model via GitOps

With the hardware math verified, it is time to deliver the AI capabilities to ChocoTech’s R&D team.

In a production environment, pulling massive 16GB model weights directly from the public internet every time a pod scales up is highly inefficient and a security risk. Instead, ChocoTech uses a "Vault" strategy: we download the model once to a local, high-speed S3-compatible storage array (MinIO) inside the cluster, and all inference pods pull their weights from there.

Exercise 1: Stage the local vault

We have packaged the infrastructure setup into an automation script. This script will provision the MinIO vault, create the necessary Data Connection secrets, and use a Kubernetes Job to download the Llama-3-8B model weights from Hugging Face into your local storage.

  1. Open your OpenShift Terminal.

  2. Clone the deployment repository:

    git clone https://github.com/RedHatQuickCourses/rhoai3-maas-lab.git
cd rhoai3-maas-lab

  3. Execute the setup script:

    chmod u+x deploy/setup-vault.sh
./deploy/setup-vault.sh

Expected Output: The script will output its progress. Wait for the final ✅ SUCCESS: Model is now available in MinIO at s3://models/llama3 message. This may take 5-15 minutes depending on the cluster’s network speed.

Exercise 2: Extract the Serving Runtime (The Engine)

OpenShift AI includes certified runtime templates. Instead of writing a complex vLLM container definition from scratch, we will extract the official template from the central RHOAI namespace and install it into our chocotech-models project.

Execute the following command to process and apply the template:

oc process vllm-cuda-runtime-template -n redhat-ods-applications | \
oc apply -f - -n chocotech-models

Expected Output: servingruntime.serving.kserve.io/vllm-cuda-runtime created.

Exercise 3: Wire the Data Connection

The setup script in Exercise 1 created a ServiceAccount (fast-track-sa) and a Secret containing the MinIO S3 credentials (storage-config). However, for the KServe storage puller to actually use those credentials to mount the model weights into the pod, we must explicitly link them.

Link the data connection secret to the service account:

oc secret link fast-track-sa storage-config -n chocotech-models

Exercise 4: The Infrastructure-as-Code (The Workload)

To deploy distributed inference and integrate with MaaS, we must use the LLMInferenceService custom resource. This manifest binds our extracted Runtime (the engine) to our MinIO Storage (the fuel), and attaches the Kuadrant MaaS policies (the governance).

Review and apply the llama3-deployment.yaml file:

cat <<EOF | oc apply -f -
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama-3-8b
  namespace: chocotech-models
  annotations:
    # THE BOUNCER LIST: Defines exactly which UI subscriptions are allowed to access this model
    alpha.maas.opendatahub.io/subscriptions: '["gold-tier", "silver-tier", "bronze-tier"]'
spec:
  # THE CATALOG PUBLISHER: Pushes this model to the Gen AI Studio UI
  publishToGenAIStudio: true

  # THE VAULT: Pointing to our secure local S3 storage connection
  model:
    uri: s3://models/llama3
    name: meta-llama/Meta-Llama-3-8B-Instruct

  # THE ENGINE: Utilizing the certified runtime we extracted in Exercise 2
  servingRuntime: "vllm-cuda-runtime"

  # THE SCHEDULER: Bypassing standard round-robin for intelligent llm-d dispatching
  router:
    scheduler:
      template:
        containers:
          - name: main
            args:
              - '--pool-group'
              - 'inference.networking.x-k8s.io'
              - '--pool-name'
              - '{{ .ObjectMeta.Name }}-inference-pool'

  # ENGINE TUNING: Injecting native vLLM parameters for maximum throughput
  template:
    spec:
      serviceAccountName: fast-track-sa
      containers:
        - name: main
          env:
            - name: VLLM_ADDITIONAL_ARGS
              value: "--max-model-len=8192 --gpu-memory-utilization=0.90"

  # THE HARDWARE: Requesting exact VRAM and GPU requirements
  placement:
    accelerator:
      name: "nvidia.com/gpu"
      count: 1
EOF

Breaking down the vLLM Tuning Variables

  • --max-model-len=8192: Prevents Out-Of-Memory (OOM) errors by capping the maximum context window. Limiting this ensures predictable memory allocation on ChocoTech’s 24GB GPUs.

  • --gpu-memory-utilization=0.90: Reserves 90% of the VRAM strictly for the KV Cache and model weights, leaving 10% for PyTorch and system overhead.

Exercise 5: Monitor the deployment

Because the weights are stored locally in your MinIO vault, the pod startup will be significantly faster than pulling from the public internet.

Check the KServe status condition to verify the model is healthy and ready to serve traffic:

oc wait --for=condition=Ready llminferenceservice/llama-3-8b -n chocotech-models --timeout=300s

Expected output: llminferenceservice.serving.kserve.io/llama-3-8b condition met.

Alternative: The UI Deployment Walkthrough

While GitOps (Infrastructure-as-Code) is the standard for Platform Engineers, OpenShift AI administrators can also deploy and publish governed models directly through the dashboard UI.

In the OpenShift AI dashboard, administrators simply navigate to the deployment wizard, select their data connection, and check the "Publish as MaaS endpoint" box. They can then use checkboxes to restrict the model to specific subscriptions.

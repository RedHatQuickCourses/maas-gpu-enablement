3.1: Model selection, hardware sizing, and serving strategy

This section is under development

Before writing a single line of YAML to deploy a model, a Platform Engineer must answer three critical questions: What model architecture does the business actually need? Will this model physically fit on our GPUs? and How do we ensure the deployment inherits our governance rules?

ChocoTech’s Platform Engineering team has received competing requests from four departments. To prevent wasting expensive GPU resources, we must evaluate models based on their architecture and capabilities before deploying them.

1. Evaluating model architecture: Dense vs. Sparse

Model architecture fundamentally impacts GPU memory consumption, inference speed, and deployment complexity.

Architecture Enterprise Characteristics

Dense Models

  • Every parameter is active for every token.

  • Provides a predictable memory footprint and consistent inference latency.

  • Offers a simpler deployment architecture.

  • Examples: Llama 2, Llama 3, Mistral 7B, Granite 3.0.

Sparse Models (Mixture of Experts)

  • Activates only a subset of parameters per token using a routing mechanism.

  • Features more total parameters with less compute required per token.

  • Results in variable latency depending on routing decisions.

  • Requires complex deployment with expert-parallelism.

  • Examples: Mixtral 8x7B, DBRX, Grok-1.

The ChocoTech Decision: For the initial MaaS deployment, ChocoTech is standardizing on dense models (like Llama-3 8B) to minimize deployment complexity and ensure predictable performance across all four departments.

2. Evaluating capabilities: Reasoning vs. Non-Reasoning

Beyond architecture, capabilities determine suitability for specific workloads and impact computational costs.

Capability Enterprise Characteristics

Reasoning Models

  • Generates intermediate reasoning steps before providing a final answer.

  • Results in higher token output per query (reasoning tokens + answer).

  • Exhibits a slower time-to-first-token (TTFT).

  • Ideal for complex problem-solving, scientific analysis, and logic puzzles.

  • ChocoTech Use Case: Quality Control needs this for defect root cause analysis, where the model must examine visual patterns and cross-reference production parameters.

Non-Reasoning Models

  • Generates direct answers without explicit intermediate steps.

  • Results in lower token output per query and faster time-to-first-token.

  • More cost-effective at scale for straightforward tasks.

  • ChocoTech Use Case: Marketing needs this for fast social media content generation, where speed and cost-effectiveness matter more than deep reasoning.

3. The Hardware Math: Sizing the Model

Once a model is selected, you must ensure it fits your infrastructure. A common mistake is assuming model size correlates to disk storage rather than Video RAM (VRAM).

As a rule of thumb, when running a dense Large Language Model at standard 16-bit precision (FP16 or BF16), each parameter requires approximately 2 bytes of VRAM.

  • Model Weights (Llama-3 8B): 8 Billion Parameters × 2 bytes = ~16 GB VRAM.

  • KV Cache: Generative AI requires memory to store conversational context. Allocate an additional 2-4 GB VRAM.

  • Total Required: ~20-24 GB VRAM.

This model fits comfortably on a single NVIDIA A10 (24GB) or A100 GPU.

If ChocoTech wanted to deploy a massive 70B parameter model, it would require ~140GB of VRAM. To mitigate this, engineers use Quantization:

  • INT8 quantization: Reduces memory by approximately 50% with minimal quality degradation.

  • INT4 quantization: Reduces memory by approximately 75%, but with noticeable quality degradation for reasoning tasks.

4. The Serving Strategy: Why llm-d?

Once we know the model fits, we must choose the serving engine. Only models deployed using the "Distributed Inference with llm-d" runtime can integrate with MaaS governance and appear in the Gen AI Studio catalog.

If you deploy a model using a standard standalone KServe InferenceService, it bypasses the MaaS Gateway completely. It will have no tier-based authentication, no rate limits, and no token consumption tracking for chargebacks.

Standalone KServe Deployment MaaS-Published Model (llm-d)

Bypasses the MaaS Gateway.

Routes directly through Gateway API and Kuadrant policy layer.

No token tracking or chargebacks.

Enforces token quotas and exact chargeback telemetry.

Direct access (no authentication required).

Requires Tier-mapped Bearer token (JWT) authentication.

What is llm-d?

llm-d (Large Language Model Distributed) is a specialized Kubernetes operator designed to orchestrate the vLLM inference engine across multiple accelerators. Even if you are only deploying Llama-3 8B onto a single GPU today, using llm-d future-proofs your architecture. When ChocoTech eventually deploys a 70B parameter model, llm-d seamlessly handles the tensor parallelism required to split that model across multiple GPUs using the LeaderWorkerSet architecture.

Now that we have verified the business requirements, calculated the hardware limits, and selected the llm-d runtime to guarantee governance, proceed to the next section to write the deployment manifest.

vLLM runtime with Models-as-a-Servce is Out of Scope for this course

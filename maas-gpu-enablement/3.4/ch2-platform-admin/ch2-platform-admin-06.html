2.3: Native API Key Management and programmatic access

With the organizational subscriptions defined, we must enable programmatic access for ChocoTech’s applications. The Marketing team’s sentiment analysis application runs in an automated container; it cannot perform an interactive human login.

OpenShift AI 3.4 introduces Native API Key Management, which completely replaces the legacy requirement of manually constructing complex Kubernetes ServiceAccounts, RoleBindings, and raw JWTs. This new capability allows you to transition from short-lived Kubernetes tokens to secure, long-lived credentials.

How API Keys Work in OpenShift AI 3.4

Before generating keys, it is crucial to understand how the MaaS controller handles identity and security at the network edge:

  • Show-Once Security: API keys are cryptographically secure strings starting with sk-oai-. The plaintext key is only displayed once during creation. The platform only stores a SHA-256 hash of the key in the PostgreSQL database.

  • Immutable Subscription Binding: Every API key is strictly bound to a single subscription at the time it is minted. If a user or script creates a key without explicitly requesting a specific subscription, the MaaS API automatically selects the highest-priority accessible subscription the user qualifies for.

  • Group Snapshotting: The user’s group membership is captured at API key creation time. API keys retain their group associations for performance reasons, meaning group information is not re-validated on every inference request.

Self-Service vs. Programmatic Access

The platform supports two distinct workflows for generating keys:

  1. Creating API Keys by Users (Self-Service): The platform empowers developers to manage their own access. For permanent keys, users navigate to Gen AI studio → API keys.

  2. Creating API Keys as Service Accounts (Programmatic Access): For automated applications, platform administrators centrally provision API keys on behalf of users or applications.

Exercise 1: Enforce Global Security Guardrails

By default, users can create "Permanent" keys. To enforce security policies, you can set a maximum expiration period for API keys across the entire platform.

Execute the following command to patch the centralized Tenant custom resource and enforce a strict 90-day maximum lifetime (maxExpirationDays) for all new API keys:

oc patch tenant default-tenant -n models-as-a-service \
  --type merge \
  -p '{"spec":{"apiKeys":{"maxExpirationDays":90}}}'

Expected behavior: Anyone attempting to create an API key with an expiration longer than 90 days will now receive an error.

Exercise 2: Provisioning the Application Key (Service Account Pattern)

We need to generate a key for the Marketing team’s automated sentiment analysis container. As a Platform Administrator, you will provision this key on their behalf.

  1. In the OpenShift AI dashboard, navigate to Gen AI studio → API keys.

  2. Click Create API key.

  3. Fill out the creation form:

    • Name: marketing-sentiment-app (Using a descriptive name is a best practice for tracking and simplifies revocation.

    • Subscription: Select the Bronze Plan (This scopes the key’s access to the models and token limits defined for Marketing).

    • Expiration: Select 90 days.

  4. Click Create.

Secure your key immediately! Save the API key in a secure location. Never commit API keys to version control. Store API keys in environment variables, Kubernetes secrets, or secret management systems such as HashiCorp Vault.

⚠️ The Group Membership "Gotcha"

Because API keys snapshot group membership, they do not automatically update if a user changes roles. If a user is removed from a group in the external OIDC provider, their existing API keys continue to work until they expire or are revoked by an administrator.

When removing a user from a group, you must manually revoke all associated API keys to immediately revoke access. Administrators must utilize the Revoke all keys for a single user administrative action in the dashboard to immediately cut off access when an employee leaves or changes departments.

Exercise 3: Test the Gateway

Now that the identity and permissions are established, let’s test the authorization chain against the MaaS Gateway using our new key.

# 1. Export your newly created API key
export MAAS_API_KEY="sk-oai-your-copied-key-here"

# 2. Retrieve the MaaS Gateway URL
export CLUSTER_DOMAIN=$(oc get ingresses.config.openshift.io cluster -o jsonpath='{.spec.domain}')
export MAAS_URL="https://maas.${CLUSTER_DOMAIN}"

# 3. Test authentication against the models endpoint
curl -s -w "\nHTTP Status: %{http_code}\n" \
  $MAAS_URL/v1/models \
  -H "Authorization: Bearer $MAAS_API_KEY" \
  -k

Decoding the Results

If your configuration is correct, you will receive the following output:

{
  "object": "list",
  "data": []
}
HTTP Status: 200

  • HTTP 200 OK: This is the critical success indicator. It proves that the Kuadrant gateway successfully validated your sk-oai- API key and mapped it to the correct Subscription and AuthPolicy.

  • Empty data array: This is completely expected. We have successfully authenticated, but we have not published any models to MaaS yet.

ChocoTech’s programmatic authentication is now secure and strictly governed. In Module 3, we will finally deploy the models and watch the MaaS catalog populate automatically!

Deconstructing the Hardware Stack

Estimated reading time: 15 minutes.

Objective

Understand the "GPU Silo" crisis and how the NVIDIA GPU Operator automates the management of NVIDIA software components. Learn the core components that power the GPU stack and their roles in enabling GPU workloads on OpenShift.

The GPU Silo Crisis

Organizations investing in GPU infrastructure often face the "GPU Silo" problem where expensive GPU hardware remains underutilized due to complex manual configuration, dependency management, and operational overhead. The NVIDIA GPU Operator addresses this challenge by automating the deployment and lifecycle management of all necessary GPU software components.

Core Components of the GPU Stack

The NVIDIA GPU Operator orchestrates several critical components that work together to enable GPU acceleration in containerized environments:

Node Feature Discovery (NFD)

Node Feature Discovery scans the node’s hardware via PCI IDs and applies Kubernetes labels to nodes where GPU hardware is detected. This enables the OpenShift scheduler to intelligently place GPU workloads on nodes with the appropriate hardware capabilities.

Key responsibilities: * Hardware detection via PCI bus scanning * Automatic node labeling based on discovered features * Integration with Kubernetes scheduler for GPU-aware placement

NVIDIA Driver Container

The NVIDIA Driver Container deploys necessary GPU drivers directly via containers, eliminating the need to manually install kernel modules on the host operating system. This containerized approach provides:

  • Simplified driver management and updates

  • Consistent driver versions across the cluster

  • Isolation from host OS dependencies

  • Automated driver installation during node provisioning

NVIDIA Container Toolkit

The NVIDIA Container Toolkit plugs into the OpenShift container runtime (CRI-O) to ensure pods can securely request and access underlying GPU hardware. It provides:

  • Runtime integration for GPU resource allocation

  • Secure GPU device passthrough to containers

  • Environment configuration for CUDA applications

  • Resource isolation between containerized workloads

GPU Feature Discovery (GFD)

GPU Feature Discovery gathers granular hardware capabilities including GPU model, VRAM size, and MIG profiles, then applies them as specific node labels. This detailed hardware inventory enables:

  • Fine-grained workload placement decisions

  • MIG-aware scheduling

  • Hardware capability matching for specific workloads

  • Resource inventory and capacity planning

Data Center GPU Manager (DCGM)

DCGM is a daemon that continuously monitors GPU health, diagnostics, and telemetry. It provides:

  • Real-time GPU health monitoring

  • Performance metrics collection

  • Diagnostic capabilities for troubleshooting

  • Integration with cluster monitoring systems

DCGM Exporter

The DCGM Exporter extracts metrics from DCGM and formats them for Prometheus scraping, enabling integration with OpenShift’s monitoring stack. This component bridges GPU telemetry with cloud-native observability platforms.

Architecture Overview

Together, these components form a complete GPU enablement stack:

  1. NFD detects GPU hardware presence

  2. Driver Container installs required drivers

  3. Container Toolkit enables runtime GPU access

  4. GFD advertises detailed GPU capabilities

  5. DCGM monitors GPU health and performance

  6. DCGM Exporter publishes metrics for observability

This automated approach transforms GPU infrastructure from manually-managed silos into a fully integrated, cloud-native compute resource suitable for Models-as-a-Service deployments at scale.

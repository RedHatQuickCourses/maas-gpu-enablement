= 3.2: Deploying the governed model via GitOps

With the hardware math verified and the `llm-d` runtime selected, it is time to deliver the AI capabilities to ChocoTech's R&D team. 

In a production environment, pulling massive 16GB model weights directly from the public internet every time a pod scales up is highly inefficient and a security risk. Instead, ChocoTech uses a "Vault" strategy: we download the model once to a local, high-speed S3-compatible storage array (MinIO) inside the cluster, and all inference pods pull their weights from there.

== Exercise 1: Stage the local vault

We have packaged the setup for the local S3 vault and the model download into an automation script. 

1. Open your OpenShift Terminal.
2. Clone the deployment repository:
+
[source,role="execute"]
----
git clone https://github.com/RedHatQuickCourses/rhoai3-maas-lab.git
cd rhoai3-maas-lab
----

3. Execute the setup script to provision the MinIO vault and download the `Llama-3-8B` model weights:
+
[source,role="execute"]
----
chmod u+x setup-vault.sh
./setup-vault.sh --model meta-llama/Meta-Llama-3-8B-Instruct
----

*Expected Output:* Wait for the script to output `✅ SUCCESS: MinIO Vault configured and Llama-3-8B staged in s3://models/llama3`.

This script automatically creates the OpenShift AI "Data Connection" (a Connection Secret) required for KServe to securely authenticate against your new local vault.

== Exercise 2: The Infrastructure-as-Code (The Workload)

To deploy distributed inference, we do not use the standard `InferenceService` CRD. Instead, we use the `LLMInferenceService` CRD. This single file tells the OpenShift AI operator to spin up the vLLM engine, pull the weights from our vault, and attach the Kuadrant MaaS policies.

Review the `llama3-deployment.yaml` file in your repository:

[source,yaml]
----
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama-3-8b
  namespace: chocotech-models
  annotations:
    # THE BOUNCER LIST: Defines exactly which UI subscriptions are allowed to access this model
    alpha.maas.opendatahub.io/subscriptions: '["gold-tier", "silver-tier", "bronze-tier"]'
spec:
  # THE CATALOG PUBLISHER: Pushes this model to the Gen AI Studio UI for developers
  publishToGenAIStudio: true
  
  # THE VAULT: Pointing to our secure local S3 storage connection
  model:
    uri: s3://models/llama3
    name: meta-llama/Meta-Llama-3-8B-Instruct
    
  # THE ENGINE: Utilizing vLLM to decouple request handling from tensor operations
  servingRuntime: "vllm-runtime"
  
  # ENGINE TUNING: Injecting native vLLM parameters for maximum throughput
  template:
    containers:
      - name: main
        env:
          - name: VLLM_ADDITIONAL_ARGS
            value: "--max-model-len=8192 --gpu-memory-utilization=0.90"
            
  # THE HARDWARE: Requesting exact VRAM and GPU requirements
  placement:
    accelerator:
      name: "nvidia.com/gpu"
      count: 1
----

=== Breaking down the vLLM Tuning Variables
* `--max-model-len=8192`: Prevents Out-Of-Memory (OOM) errors by capping the maximum context window. Generative AI requests involve varying prompt lengths; limiting this ensures predictable memory allocation.
* `--gpu-memory-utilization=0.90`: Reserves 90% of the VRAM for the KV Cache and model weights, leaving 10% for PyTorch overhead. 

Apply the deployment:

[source,role="execute"]
----
oc apply -f llama3-deployment.yaml
----

== Exercise 3: Monitor the deployment

Because the weights are stored locally in your MinIO vault, the deployment will be significantly faster than pulling from the public internet. 

Check the KServe status condition to verify the model is healthy:

[source,role="execute"]
----
oc get llminferenceservice llama-3-8b -n chocotech-models -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'
----

*Expected output:* `True`.

== Alternative: The UI Deployment Walkthrough

While GitOps (Infrastructure-as-Code) is the standard for Platform Engineers, OpenShift AI administrators can also deploy and publish governed models directly through the dashboard UI. 

In the OpenShift AI dashboard, the process has been updated to use a streamlined deployment wizard. When deploying, administrators simply navigate to the "Advanced settings" and check the **"Publish as MaaS endpoint"** box. They can then use multi-select checkboxes to restrict the model to "Specific subscriptions" (e.g., hiding a massive model from the Free tier).

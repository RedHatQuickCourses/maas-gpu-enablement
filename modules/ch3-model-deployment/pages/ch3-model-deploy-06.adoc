= 3.5: Day-2 operations and lifecycle management

ChocoTech's Quality Control team has been successfully using the `llama-3-8b` model for several weeks. Now, they want to upgrade to a new version that has been fine-tuned specifically on ChocoTech manufacturing defects.

"We cannot risk downtime," explains the QC Director. "Our production line depends on real-time defect detection. How do we update the model without causing an outage?"

This is a classic "Day 2" production challenge. Because we deployed the model using MaaS and GitOps, updating it is safe, auditable, and seamless.

== Version Management Strategy

Production AI systems require disciplined version management, just like application code. ChocoTech should use semantic versioning for its model weights stored in the MinIO vault:

* **v1.0.0:** Base Llama-3 8B Instruct (Currently deployed)
* **v1.1.0:** Fine-tuned on chocolate defect images (The requested upgrade)
* **v2.0.0:** Major architecture change (e.g., swapping to Llama-3 70B)

By keeping the model weights in distinct folders within the S3 vault (e.g., `s3://models/llama3/v1.1.0`), you ensure that the previous version is always preserved and ready for an instant rollback.

== The GitOps Update Workflow

To update the model, a Platform Engineer does *not* delete the running pod, nor do they click around the OpenShift UI. Instead, they update the declarative YAML file in their Git repository.

When updating the `LLMInferenceService`, the only line that needs to change is the `uri` pointing to the new vault location:

[source,yaml]
----
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama-3-8b
# ... [truncating for brevity] ...
spec:
  model:
    uri: s3://models/llama3/v1.1.0  # <--- Update the path to the new version
    name: meta-llama/Meta-Llama-3-8B-Instruct
----

=== Graceful Rollouts (Zero-Downtime)
When you commit this change and apply it to the cluster, OpenShift AI and KServe perform a **Rolling Update**:

1. The system spins up a completely new vLLM pod and begins downloading the `v1.1.0` weights from the MinIO vault.
2. During this time, the `v1.0.0` pod continues serving live traffic from the Gateway. No user requests are dropped.
3. Once the `v1.1.0` pod reports a `Ready` status (meaning the weights are fully loaded into VRAM), KServe automatically flips the network route.
4. All new traffic flows to `v1.1.0`. The old `v1.0.0` pod is then gracefully terminated.

=== The Rollback 
If the Quality Control team reports that the `v1.1.0` model is hallucinating or underperforming, the rollback process takes seconds. You simply revert the Git commit back to `uri: s3://models/llama3/v1.0.0` and KServe repeats the rolling update process to restore the known-good state.

= 2.3: Configuring programmatic access (ServiceAccounts & RBAC)

With the organizational structure in place, we must enable programmatic access for ChocoTech's applications. The Marketing team's sentiment analysis application runs in a container; it cannot perform an interactive human login. 

If ChocoTech were using a cheap public API, the developers would likely hardcode a static, permanent API key into their application. This is a massive enterprise security risk. If the key leaks, anyone can access the endpoint indefinitely. 

Instead, Models as a Service utilizes native Kubernetes **ServiceAccounts** combined with **JSON Web Tokens (JWTs)** and **Role-Based Access Control (RBAC)**. 

== The Enterprise Advantage: JWTs vs. Static Keys

[cols="1a,2a",options="header",frame="all",grid="rows"]
|===
| Feature | The MaaS Security Benefit

| **Time-Limited (Ephemeral)**
| JWTs automatically expire after a configured duration (e.g., 1 hour or 24 hours). This drastically reduces the blast radius of a compromised token. 

| **Instantly Revocable**
| If an application is compromised, you simply delete the ServiceAccount. All associated tokens become instantly invalid across the entire cluster.

| **Cryptographically Signed**
| Tokens are signed by the Kubernetes API server's private key. Kuadrant's Authorino engine validates this signature at the network edge, ensuring tampered tokens are dropped before they ever reach the GPU.

| **Strictly Auditable**
| Every API request includes the exact ServiceAccount identity, providing absolute transparency in OpenShift audit logs.
|===


== Exercise 1: Establish the Identity and RBAC Permissions

To follow the principle of least privilege, we must create an identity (the ServiceAccount), define what it is allowed to do (the Role), and link them together (the RoleBinding). 

The Marketing application only needs to discover available models and send inference requests. It does *not* need permission to deploy or delete models. We will grant it read-only access (`get` and `list` verbs) to the `llminferenceservices` resources.

Execute the following block to create all three components in the `chocotech-models` namespace:

[source,role="execute"]
----
# 1. Create the ServiceAccount
oc create serviceaccount marketing-sentiment-app -n chocotech-models

# 2. Create the strictly-scoped Role
cat <<EOF | oc apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: llm-inference-viewer
  namespace: chocotech-models
rules:
- apiGroups: ["serving.kserve.io"]
  resources: ["llminferenceservices", "llminferenceservices/status"]
  verbs: ["get", "list"]
EOF

# 3. Bind the ServiceAccount to the Role
oc create rolebinding marketing-sentiment-app-binding \
  --role=llm-inference-viewer \
  --serviceaccount=chocotech-models:marketing-sentiment-app \
  -n chocotech-models
----

*Expected output:* The terminal will confirm the creation of the `serviceaccount`, `role`, and `rolebinding`.

== Exercise 2: Generate and Test the JWT Token

Now that the identity and permissions are established, we can generate a temporary token and test the authorization chain against the MaaS Gateway. 

Generate a 1-hour token and simulate an API request from the Marketing application:

[source,role="execute"]
----
# Generate a 1-hour JWT token
TOKEN=$(oc create token marketing-sentiment-app -n chocotech-models --duration=1h)

# Retrieve the Gateway hostname
GATEWAY_HOST=$(oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.addresses[0].value}')

# Test authentication against the models endpoint
curl -s -w "\nHTTP Status: %{http_code}\n" \
  http://$GATEWAY_HOST/v1/models \
  -H "Authorization: Bearer $TOKEN" \
  -k
----

=== Decoding the Results

If your configuration is correct, you will receive the following output:

[source,json]
----
{
  "object": "list",
  "data": []
}
HTTP Status: 200
----

* **HTTP 200 OK:** This is the critical success indicator. It proves that Authorino successfully validated the token signature, and Kubernetes RBAC confirmed the ServiceAccount has the `llm-inference-viewer` role. 
* **Empty `data` array:** This is completely expected. We have successfully authenticated, but we have not deployed any models into the catalog yet (we will do this in Module 3). 

If you receive an `HTTP 401 Unauthorized` or `403 Forbidden`, Authorino has rejected the request. Ensure your token hasn't expired and your RoleBinding was created in the correct `chocotech-models` namespace.

ChocoTech's programmatic authentication is now secure. In the next section, we will map these authenticated identities to strict tier policies to govern exactly how much compute they can consume.

= 2.2: Establishing the organizational foundation

To enforce the CIO's subscription strategy, the MaaS platform needs a mathematical way to identify which request belongs to which team. We achieve this by utilizing native OpenShift Groups. 

By mapping OpenShift groups to our MaaS Subscriptions, we gain massive enterprise advantages over standard public APIs:

* **Scalable Onboarding:** You add a new data scientist to a group, and they automatically inherit the correct API token limits and model access. No manual policy tweaking required.
* **Auditable Transparency:** Every API request is logged against the business unit (e.g., `chocotech-quality-control`), providing Finance with the exact metrics needed for departmental chargebacks.

---

== Exercise 1: Create the business unit groups

Instead of creating individual users, let's create the three organizational groups that will soon be mapped to our Gold, Silver, and Bronze subscription plans. 

Execute the following commands to create the ChocoTech groups:

[source,role="execute"]
----
oc adm groups new chocotech-quality-control
oc adm groups new chocotech-rd
oc adm groups new chocotech-marketing
----

*Expected output:* The terminal will confirm creation with three lines reading `group.user.openshift.io/<group-name> created`.

Next, to successfully complete the upcoming labs, you need to simulate being a member of the Quality Control team (our Gold Plan). Add your current user account to the QC group:

[source,role="execute"]
----
oc adm groups add-users chocotech-quality-control $(oc whoami)
----

== Exercise 2: Prepare the governed namespace

Models must be deployed into a dedicated namespace (project). However, simply creating a namespace is not enough. You must explicitly label it so the MaaS operator knows to enforce governance rules on the traffic entering it.

Create the new project for ChocoTech's centralized model hub:

[source,role="execute"]
----
oc new-project chocotech-models
----

Now, apply the critical MaaS routing and governance labels:

[source,role="execute"]
----
oc label namespace chocotech-models maas.opendatahub.io/managed=true
oc label namespace chocotech-models modelmesh-enabled=false
----

=== Why are these labels required?
* `maas.opendatahub.io/managed=true`: This is the activation switch. It instructs the OpenShift AI operator to attach the API Gateway and governance policies to any model deployed within this namespace. 
* `modelmesh-enabled=false`: MaaS utilizes the KServe engine (specifically the `llm-d` distributed runtime) to serve massive generative AI models. This label explicitly prevents the older ModelMesh architecture from attempting to manage models in this namespace.

== Verify your foundation

Before moving on to the subscriptions, run this combined verification script to ensure your groups and labeled namespace are perfectly staged:

[source,role="execute"]
----
echo "=== Groups ===" && \
oc get groups | grep chocotech && \
echo "=== Gold Plan Membership ===" && \
oc get group chocotech-quality-control -o jsonpath='{.users[*]}' && echo && \
echo "=== Governed Namespace ===" && \
oc get namespace chocotech-models --show-labels | grep maas
----

If the output confirms your groups exist, your username is listed under the Quality Control group, and the `chocotech-models` namespace displays the managed label, your organizational foundation is secure. Proceed to define the subscription limits.

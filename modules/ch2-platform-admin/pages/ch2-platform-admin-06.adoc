= 2.3: Configuring programmatic access (The Service Account Pattern)

With the organizational subscriptions defined, we must enable programmatic access for ChocoTech's applications. The Marketing team's sentiment analysis application runs in an automated container; it cannot perform an interactive human login via OIDC. 

In legacy environments, you would have to manually construct complex Kubernetes ServiceAccounts, RoleBindings, and extract raw JWTs. In OpenShift AI 3.4, Models-as-a-Service (MaaS) introduces **Native API Key Management**. Administrators can centrally provision and manage secure API keys on behalf of users or automated applications (the "Service Account Pattern").


== The Enterprise Advantage: Governed API Keys

Unlike public APIs where developers use ungoverned static keys, MaaS API Keys provide enterprise-grade security:

* **Subscription-Bound:** Keys are strictly bound to a specific subscription (e.g., the Bronze Plan), guaranteeing the application cannot exceed its token quotas.
* **Instantly Revocable:** If a key is compromised, administrators can revoke it with a single click, immediately invalidating all requests using that key.
* **Centrally Auditable:** Use descriptive names for API keys (e.g., `production-chatbot` or `marketing-sentiment-app`) to track usage and simplify revocation.

== Exercise 1: Enforce Global Security Guardrails

While users and admins can select expiration dates when creating keys, we want to ensure no one accidentally creates a "Permanent" (`Never` expiring) key for a temporary script. 

You can set a maximum expiration period for API keys across the entire platform to enforce security policies.

Execute the following command to patch the centralized `Tenant` custom resource and enforce a strict 90-day maximum lifetime for all new API keys:

[source,role="execute"]
----
oc patch tenant default-tenant -n models-as-a-service \
  --type merge \
  -p '{"spec":{"apiKeys":{"maxExpirationDays":90}}}'
----
*Expected behavior:* Anyone attempting to create an API key with an expiration longer than 90 days will now receive an error.

== Exercise 2: Provisioning the Application Key (Service Account Pattern)

We need to generate a key for the Marketing team's sentiment analysis container. As a Platform Administrator, you will provision this key on their behalf.

1. In the OpenShift AI dashboard, navigate to **Gen AI studio** -> **API keys**.
2. Click **Create API key**.
3. Fill out the creation form:
* **Name:** `marketing-sentiment-app` (Using a descriptive name is a best practice for tracking).
* **Subscription:** Select the **Bronze Plan** (This scopes the key's access to the models and token limits defined for Marketing ).
* **Expiration:** Select `90 days`.
4. Click **Create**.

[IMPORTANT]
====
**Secure your key immediately!** The platform will display the API key string (which begins with `sk-oai-`) only once. Copy this key and save it to a secure location (like a password manager or your local terminal's environment variables). You will not be able to view it again. If you lose the key, you must revoke it and create a new one.
====

=== ⚠️ The Group Membership "Gotcha"

When an API key is created, the system permanently binds it to the user's current group memberships for performance reasons (group information is not re-validated on every inference request). 

**What this means for Administrators:** If a developer transfers from the Quality Control team to the Marketing team, their existing API keys will **continue to work with the original Quality Control (Gold Plan) permissions** until those keys expire or are revoked. When removing a user from an OpenShift group, you **must manually revoke all associated API keys** to immediately revoke access.

== Exercise 3: Test the Gateway

Now that the identity and permissions are established, let's test the authorization chain against the MaaS Gateway using our new key.

[source,role="execute"]
----
# 1. Export your newly created API key
export MAAS_API_KEY="sk-oai-your-copied-key-here"

# 2. Retrieve the MaaS Gateway URL
export CLUSTER_DOMAIN=$(oc get ingresses.config.openshift.io cluster -o jsonpath='{.spec.domain}')
export MAAS_URL="https://maas.${CLUSTER_DOMAIN}"

# 3. Test authentication against the models endpoint
curl -s -w "\nHTTP Status: %{http_code}\n" \
  $MAAS_URL/v1/models \
  -H "Authorization: Bearer $MAAS_API_KEY" \
  -k
----

=== Decoding the Results

If your configuration is correct, you will receive the following output:

[source,json]
----
{
  "object": "list",
  "data": []
}
HTTP Status: 200
----

* **HTTP 200 OK:** This is the critical success indicator. It proves that the Kuadrant gateway successfully validated your `sk-oai-` API key and mapped it to the correct Subscription and AuthPolicy.
* **Empty `data` array:** This is completely expected. We have successfully authenticated, but we have not published any models to MaaS yet.

If you receive an `HTTP 401 Unauthorized`, verify your API key is correct and not expired. If you receive an `HTTP 403 Forbidden`, ensure that your **MaaSAuthPolicy** was deployed correctly in the previous step and includes the appropriate group.

== Knowledge check

Test your understanding of the subscription governance architecture before moving on:

1. **Why are MaaS Subscription policies created before model deployment, rather than after?**
+
MaaS Subscription policies must exist before deployment so the MaaS operator and Kuadrant can enforce token limits and quotas from the very first inference request. Creating policies after deployment would leave a window of uncontrolled, ungoverned access.

2. **What happens if a Bronze subscription user (Marketing) tries to exceed 2,000 tokens per minute?**
+
Kuadrant's Limitador component, operating at the network edge, tracks the token consumption per subscription. It immediately rejects any requests exceeding the limit with an `HTTP 429 Too Many Requests` status code, preventing the request from ever reaching the KServe model pod. The user's application must wait until the next minute window resets.

3. **How does the system know which subscription limits to apply to a programmatic user's request?**
+
The API gateway intercepts the request and validates the provided API key (`sk-oai-...`). Because the key was permanently bound to the user's OpenShift group membership and highest priority subscription at the time it was created, the gateway immediately applies the correct token limits.

.🛠️ Troubleshooting: Governance & Authentication (Click to expand)
[%collapsible]
====
If you encountered any issues verifying your policies, check the following common resolutions:

**Issue: Token authentication returns `401 Unauthorized`**

* **Check API Key:** OpenShift tokens are not supported for MaaS inference. Ensure you are using a MaaS API key (starting with `sk-oai-`) generated from the Gen AI Studio dashboard.
* **Check Key Revocation:** Verify in the Gen AI Studio UI that your API Key is marked as "Active" and has not been revoked or expired.

**Issue: Request returns `403 Forbidden`**

* **Verify AuthPolicy:** A subscription grants limits, but an AuthPolicy grants access. Verify your `MaaSAuthPolicy` exists in the `models-as-a-service` namespace and explicitly lists both your model and your OpenShift group.
* **Check Group Membership:** Ensure your user account is actually a member of the required OpenShift group (e.g., `oc get group chocotech-quality-control -o yaml`).

**Issue: Authorino or Limitador pods are crashing / pending**

* **Check Pod Events:** Run `oc describe pod -l app=authorino -n kuadrant-system`.
* **Common Causes:** Insufficient CPU/Memory on the cluster nodes, or a network issue preventing the cluster from pulling the Kuadrant images from `quay.io`.

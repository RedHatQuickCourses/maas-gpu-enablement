== The Three-Layer MaaS Operator Stack

A production Models-as-a-Service platform requires orchestrating dozens of operators across three functional layers. Red Hat OpenShift AI provides this orchestration.

=== Layer 1: Core & Infrastructure Operators

**Purpose:** Enable hardware detection and GPU software stack automation

**Operators in this layer:**

* **Node Feature Discovery (NFD)**
  ** Scans node hardware via PCI bus
  ** Labels nodes with detected capabilities
  ** Namespace: `openshift-nfd`
  ** Catalog: `redhat-operators`

* **NVIDIA GPU Operator**
  ** Deploys GPU drivers as containers
  ** Installs NVIDIA Container Toolkit
  ** Manages device plugin for GPU resource allocation
  ** Deploys DCGM for monitoring
  ** Namespace: `nvidia-gpu-operator`
  ** Catalog: `certified-operators`

**Activation:**

[source,yaml]
----
# NFD activation
apiVersion: nfd.openshift.io/v1
kind: NodeFeatureDiscovery
metadata:
  name: nfd-instance
  namespace: openshift-nfd

# GPU Operator activation
apiVersion: nvidia.com/v1
kind: ClusterPolicy
metadata:
  name: gpu-cluster-policy
----

**You will deploy this layer in the upcoming lab.**

=== Layer 2: Operators for Inference at Scale

**Purpose:** Provide production-grade MaaS capabilities like certificate management, rate limiting, workload queueing, and distributed training.

**Operators in this layer:**

* **Cert-manager**
  ** Automates TLS certificate issuance and renewal
  ** Integrates with Let's Encrypt and internal CAs
  ** Namespace: `cert-manager`
  ** Required for secure inference endpoints

* **Connectivity Link (or Kuadrant)**
  ** Rate limiting for inference requests
  ** Authentication and authorization policies
  ** API gateway capabilities
  ** Namespace: `kuadrant-system`

* **LeaderWorkerSet**
  ** Orchestrates distributed training jobs
  ** Leader/worker pod coordination
  ** Namespace: Workload-specific

* **Kueue**
  ** Workload queueing and quota management
  ** Fair sharing of GPU resources across teams
  ** Namespace: `kueue-system`
  ** Critical for multi-tenant MaaS platforms

**Activation:** Each operator has its own Custom Resource for configuration.

---

=== Layer 3: Red Hat OpenShift AI Orchestration

**Purpose:** Coordinate the complete MaaS ecosystem with unified configuration and dependency management.

**Red Hat OpenShift AI Operator:**

* **Single control point** for MaaS platform lifecycle
* **Manages Layer 2 operator dependencies** automatically
* **Unified configuration** via DataScienceCluster Custom Resource
* **Coordinates upgrades** across the operator suite
* **Namespace:** `redhat-ods-operator`
* **Catalog:** `redhat-operators`

**DataScienceCluster CR Example:**

[source,yaml]
----
apiVersion: datasciencecluster.opendatahub.io/v1
kind: DataScienceCluster
metadata:
  name: default-dsc
spec:
  components:
    kueue:
      managementState: Unmanaged  # <1>
    kserve:
      managementState: Managed  # <2>
    dashboard:
      managementState: Managed
----
<1> OpenShift AI hands-off queueing to the Red Hat Kueue operator
<2> OpenShift AI automatically installs and manages KServe operator

**Key benefit:** Instead of manually installing 10+ operators, you install OpenShift AI once, and it orchestrates the dependencies.

=== How the Layers Work Together

**Example: Deploying an InferenceService**

[source,yaml]
----
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: llama-3-8b
spec:
  predictor:
    model:
      modelFormat: vllm
      runtime: vllm-runtime
    resources:
      limits:
        nvidia.com/gpu: 1  # <1>
----
<1> Request GPU resource

**Workflow across three layers:**

1. **Layer 3 (OpenShift AI):** Receives InferenceService creation request
2. **Layer 2 (Kueue):** Queues workload if GPUs unavailable, enforces quota
3. **Layer 2 (KServe):** Creates inference deployment pods
4. **Layer 1 (GPU Operator):** Kubernetes scheduler finds GPU node (via NFD labels), device plugin allocates GPU to pod, DCGM monitors GPU telemetry
5. **Layer 2 (Cert-manager):** Issues TLS certificate for inference endpoint
6. **Layer 2 (Kuadrant):** Applies rate limiting policies

All three layers coordinate to deliver a production-ready inference service with security, monitoring, and resource management.

**Deconstructing the Hardware Stack:**
Understand the "GPU Silo" crisis and how the NVIDIA GPU Operator automates the management of NVIDIA software components. Define the core components that power the GPU stack:
• **Node Feature Discovery (NFD):** Scans the node's hardware (via PCI IDs) and applies Kubernetes labels so the scheduler knows where GPUs physically exist.
• **NVIDIA Driver Container:** Deploys the necessary NVIDIA GPU drivers directly via containers, removing the need to manually install kernel modules on the host OS.
• **NVIDIA Container Toolkit:** Plugs into the OpenShift container runtime (CRI-O) to ensure pods can securely request and access the underlying GPU hardware.
• **GPU Feature Discovery (GFD):** Gathers granular hardware capabilities (GPU model, VRAM size, MIG profiles) and applies them as specific node labels.
• **DCGM (Data Center GPU Manager):** A daemon that continuously monitors GPU health, diagnostics, and telemetry.
• **DCGM Exporter:** Extracts metrics from DCGM and formats them for Prometheus scraping.

== The GPU Underutilization Problem

Enterprise AI platforms investing in GPU infrastructure often face a critical challenge: expensive GPU hardware remains dramatically underutilized despite $25,000-$50,000 per GPU capital investment. Understanding this problem and the solutions the GPU Operator enables is essential for maximizing return on GPU infrastructure investment.

=== The $50,000 Idle GPU Crisis

Organizations deploying AI platforms without GPU sharing capabilities typically see GPU utilization rates between 20-40% for inference workloads. This underutilization represents massive wasted capital and operational expense.

**Concrete Example:**

FinServe Corp purchases 10x NVIDIA A100-80GB GPUs at $25,000 per GPU ($250,000 total capital expenditure). They deploy 3 large language model inference services, each allocated a dedicated GPU. Monitoring shows:

* GPU 1 (Finance QA model): 30% average utilization
* GPU 2 (Risk Analysis model): 25% average utilization
* GPU 3 (Document Intelligence model): 20% average utilization
* GPUs 4-10: 0% utilization (reserved for "future workloads")

**Result:** 25% average utilization across the platform → 75% idle capacity → $175,000 in wasted GPU investment annually.

The GPU Operator addresses this crisis through three GPU sharing strategies, each with different isolation guarantees, reconfiguration requirements, and utilization outcomes:

[cols="2,2,1,3",options="header"]
|===
|Strategy |Isolation Level |Typical Utilization |Best Use Case

|**Full GPU Allocation**
|Complete hardware isolation
|20-40%
|Training workloads, <10 inference services, guaranteed performance requirements

|**Time-Slicing** (Software)
|Software-level temporal multiplexing
|60-80%
|Bursty inference workloads, dev/test environments, GPUs without MIG support (V100, T4, A10)

|**MIG** (Hardware Partitioning)
|Hardware-level spatial isolation
|85-95%
|Multi-tenant MaaS platforms, >10 concurrent workloads, SLA-backed services on Ampere/Hopper GPUs
|===

[TIP]
====
For MaaS platforms serving fewer than 10 models, time-slicing provides the fastest time-to-value with minimal configuration overhead. For platforms serving more than 10 models or requiring SLA-backed performance guarantees, invest in MIG-capable hardware (A100, A30, H100) for maximum utilization with guaranteed isolation.
====

=== Multi-Tenant Sharing Challenges

Traditional approaches to GPU sharing fail to meet production MaaS platform requirements:

**Approach 1: Full GPU Passthrough (No Sharing)**

* One workload per physical GPU
* Complete isolation but catastrophic underutilization
* Requires 1:1 GPU-to-workload ratio (10 models = 10 GPUs minimum)

**Approach 2: Manual Time-Sharing**

* Multiple pods scheduled to same GPU via Kubernetes resource oversubscription
* No performance isolation—runaway processes starve other workloads
* Unpredictable latency makes SLA compliance impossible

**Approach 3: MIG (Multi-Instance GPU)**

* Hardware-partitioned GPU instances with dedicated compute and memory
* Guaranteed performance isolation with high utilization
* Requires Ampere or Hopper architecture GPUs and configuration expertise

The GPU Operator automates MIG and time-slicing configuration, transforming GPU sharing from manual, error-prone processes into declarative, operator-managed capabilities.

**Transformation Example:**

----
Before GPU Operator + Sharing:
10 GPUs → 3 workloads → 25% utilization → $175K wasted annually

After GPU Operator + MIG:
10 A100 GPUs → 70 MIG instances (7 per GPU) → 24 active workloads
→ 85% utilization → $150K annual savings → 8x more concurrent workloads
----

[IMPORTANT]
====
GPU sharing strategy is not reversible without workload downtime. Choose based on 12-month workload projections, not current state. Migrating from time-slicing to MIG requires node drains, driver reconfiguration, and workload migration (10-15 minutes downtime per node).
====

== ClusterPolicy as the Control Plane

The xref:s2-operators-overview.adoc#_custom_resources_and_controllers[ClusterPolicy Custom Resource] (introduced in Section 2) serves as the single declarative control plane for the entire GPU software stack. Every component the GPU Operator manages—drivers, monitoring, device plugins, GPU sharing—is configured through ClusterPolicy specification fields.

=== Declarative GPU Stack Configuration

ClusterPolicy abstracts the complexity of managing eight interdependent components into a single YAML resource. The operator's controller continuously reconciles this desired state, deploying and maintaining DaemonSets for each enabled component.

**Component Hierarchy:**

----
ClusterPolicy Custom Resource
  ├── driver           # NVIDIA GPU drivers (containerized)
  ├── toolkit          # Container runtime integration
  ├── devicePlugin     # GPU resource advertisement to Kubernetes
  ├── gfd              # GPU Feature Discovery (fine-grained labels)
  ├── dcgm             # Data Center GPU Manager (telemetry)
  ├── dcgmExporter     # Prometheus metrics export
  ├── migManager       # MIG partitioning automation
  └── nodeStatusExporter  # Node-level GPU health events
----

When you create a ClusterPolicy CR, the GPU Operator controller:

1. Reads the desired state from `spec` fields
2. Creates DaemonSets for enabled components
3. Deploys pods to GPU nodes (identified by xref:s1-hardware-stack.adoc#_node_feature_discovery_nfd[NFD labels])
4. Monitors component health and reconciles drift every 5-10 seconds
5. Self-heals by recreating deleted or failed resources

This declarative approach replaces manual driver installation scripts, configuration management tools, and custom automation—reducing GPU platform deployment time from 150+ hours to under 2 hours.

=== Component Deep-Dive: Roles and Responsibilities

Understanding what each ClusterPolicy component does and why it exists is essential for production configuration decisions and troubleshooting.

==== Driver Container

The xref:s1-hardware-stack.adoc#_nvidia_driver_container[NVIDIA Driver Container] deploys GPU drivers as containerized workloads rather than host-installed kernel modules. This architectural shift provides several production advantages:

* **Version consistency:** All GPU nodes run identical driver versions specified in ClusterPolicy
* **Automated deployment:** New GPU nodes receive drivers automatically via DaemonSet scheduling
* **Self-healing:** Deleted or crashed driver pods are recreated by the operator within 30 seconds
* **Isolation from host OS:** Driver updates don't require host OS reboots or kernel version coordination

**ClusterPolicy Configuration:**

[source,yaml]
----
spec:
  driver:
    enabled: true
    version: "535.129.03"  # <1>
    repository: nvcr.io/nvidia
    image: driver
----
<1> Pin to specific driver version for production consistency—never use `latest` tag

**Self-Healing Example:**

An administrator accidentally deletes the NVIDIA driver DaemonSet during troubleshooting:

[source,bash]
----
$ oc delete daemonset nvidia-driver-daemonset -n nvidia-gpu-operator
daemonset.apps "nvidia-driver-daemonset" deleted

# GPU Operator controller detects the mismatch within 5 seconds

# 30 seconds later, the DaemonSet is automatically recreated:
$ oc get daemonset nvidia-driver-daemonset -n nvidia-gpu-operator
NAME                      DESIRED   CURRENT   READY   AGE
nvidia-driver-daemonset   2         2         2       18s
----

**What happened:** The controller detected that `driver.enabled: true` in ClusterPolicy but the DaemonSet was missing. It recreated the DaemonSet, pods started, and GPU access was restored in approximately 90 seconds total—no human intervention required.

[WARNING]
====
Never manually modify GPU driver DaemonSets or other operator-managed resources. Changes are overwritten by operator reconciliation within 30 seconds. All configuration must be declared in ClusterPolicy `spec` to persist.
====

==== Device Plugin

The NVIDIA Device Plugin is a Kubernetes device plugin that advertises GPU resources to the kubelet, enabling the scheduler to bind GPU-requesting pods to nodes with available GPUs.

**How It Works:**

1. Device plugin discovers GPUs on the node (via driver and NVIDIA libraries)
2. Registers GPU count with kubelet as `nvidia.com/gpu` extended resource
3. Kubelet advertises resource capacity to Kubernetes API
4. Scheduler binds pods requesting `nvidia.com/gpu` to nodes with availability
5. Device plugin allocates specific GPU device(s) to pod containers

**Resource Advertisement Example:**

A node with 2x NVIDIA A100-40GB GPUs advertises:

[source,bash]
----
$ oc describe node worker-gpu-0.example.com
...
Capacity:
  nvidia.com/gpu:  2
Allocatable:
  nvidia.com/gpu:  2
----

An inference pod requests GPU allocation:

[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: llama-inference
spec:
  containers:
  - name: vllm-server
    image: vllm/vllm-openai:latest
    resources:
      limits:
        nvidia.com/gpu: 1  # <1>
----
<1> Request 1 GPU—scheduler binds to node with `nvidia.com/gpu` availability

**ClusterPolicy Configuration:**

[source,yaml]
----
spec:
  devicePlugin:
    enabled: true
    config:
      name: ""  # <1>
----
<1> Leave empty for default single-GPU allocation; specify ConfigMap name for time-slicing configuration (covered in Section 3)

While xref:s1-hardware-stack.adoc#_node_feature_discovery_nfd[NFD] labels nodes with generic PCI vendor detection (`feature.node.kubernetes.io/pci-10de.present=true`), GFD provides GPU-specific details.

**GFD Label Examples:**

[source,bash]

  jq '.metadata.labels | with_entries(select(.key | contains("nvidia")))'
{
  "nvidia.com/cuda.driver.major": "535",
  "nvidia.com/cuda.driver.minor": "129",
  "nvidia.com/cuda.runtime.major": "12",
  "nvidia.com/cuda.runtime.minor": "2",
  "nvidia.com/gpu.count": "2",
  "nvidia.com/gpu.family": "ampere",
  "nvidia.com/gpu.memory": "40960",
  "nvidia.com/gpu.product": "A100-PCIE-40GB",

These labels enable intelligent workload placement:

**Example 1: Target specific GPU models**

[source,yaml]
----
# Large LLM inference requires A100 GPUs with 40GB+ VRAM
nodeSelector:
  nvidia.com/gpu.product: "A100-PCIE-40GB"
  nvidia.com/gpu.memory: "40960"
----

**Example 2: Ensure MIG capability for future flexibility**

[source,yaml]
----
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: nvidia.com/mig.capable
          operator: In
          values: ["true"]  # <1>
----
<1> Schedule only on MIG-capable GPUs (A30, A100, H100) even if MIG is not currently enabled

xref:s1-hardware-stack.adoc#_data_center_gpu_manager_dcgm[DCGM (Data Center GPU Manager)] continuously monitors GPU health, performance, and telemetry. ==== MIG Manager

The MIG Manager automates Multi-Instance GPU mode enablement and profile application based on node labels. While MIG configuration details are covered comprehensively in Chapter 2, the ClusterPolicy enables the manager as a prerequisite.

**How MIG Manager Works:**

1. Administrator applies MIG configuration label to GPU node
2. MIG Manager detects label via watch on Node resources
3. MIG Manager enables MIG mode on the GPU (if not already enabled)
4. MIG Manager applies specified profile (e.g., `all-1g.10gb`, `mixed`)
5. MIG Manager signals device plugin to rediscover GPU instances
6. Device plugin advertises MIG instances as allocatable resources

**Label-Driven Workflow Example:**

[source,bash]
----
# Administrator labels node with desired MIG configuration
$ oc label node worker-gpu-0.example.com \
  nvidia.com/mig.config=all-1g.10gb

# MIG Manager detects label, enables MIG mode, applies profile
# (Takes approximately 60 seconds, requires driver pod restart)

# Device plugin rediscovers and advertises MIG instances
$ oc describe node worker-gpu-0.example.com | grep mig
  nvidia.com/mig-1g.10gb:  7  # <1>
----
<1> Single A100-80GB partitioned into 7x `1g.10gb` instances, each allocatable to separate workloads

**ClusterPolicy Configuration:**

[source,yaml]
----
spec:
  migManager:
    enabled: true  # <1>
----
<1> Enable MIG Manager—actual MIG profile configuration happens via node labels (Chapter 2)

[TIP]
====
Enable `migManager: true` in ClusterPolicy even if you are not immediately using MIG. This prepares your platform for future MIG enablement without requiring ClusterPolicy reconfiguration and operator restarts.
====

== GPU Sharing Strategies

Choosing the right GPU sharing strategy is one of the most critical architectural decisions for MaaS platforms. The wrong choice leads to either wasted GPU capacity (underutilization) or unpredictable performance (resource contention). This section compares the three approaches and provides production decision frameworks.

=== Three GPU Sharing Modes

[cols="1,2,2,2,2,2",options="header"]
|===
|Strategy |Isolation Level |Reconfiguration Impact |Hardware Requirements |Best For |Avoid When

|**Full GPU**
|Complete hardware isolation, one workload per GPU
|None—workloads simply request different GPUs
|Any NVIDIA GPU
|Training workloads, <10 inference services, guaranteed performance
|Platform has >70% idle GPU capacity

|**Time-Slicing**
|Software-level temporal multiplexing, no isolation
|ConfigMap update + device plugin restart (2-3 min), no workload drain required
|Any NVIDIA GPU
|Bursty inference workloads, dev/test environments, non-MIG GPUs (V100/T4/A10)
|SLA-backed production services, predictable latency requirements

|**MIG**
|Hardware-level spatial partitioning, guaranteed isolation
|Node drain + driver restart + MIG reconfiguration (10-15 min downtime per node)
|Ampere or Hopper only (A30, A100, A100X, H100)
|Multi-tenant MaaS platforms, >10 concurrent workloads, SLA guarantees
|Mixed GPU generations (some MIG-capable, some not)
|===

=== Configuring Time-Slicing (Software Sharing)

Time-slicing allows multiple pods to share a single GPU by oversubscribing the `nvidia.com/gpu` resource. The device plugin advertises more GPU resources than physically exist, and the CUDA runtime time-multiplexes access.

**How Time-Slicing Works:**

Without time-slicing, a node with 1 GPU advertises `nvidia.com/gpu: 1`. Only one pod can allocate the GPU. With time-slicing configured as `replicas: 4`, the device plugin advertises `nvidia.com/gpu: 4`. Four pods can allocate the resource, sharing GPU access temporally.

**Configuration via Device Plugin ConfigMap:**

[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: device-plugin-config
  namespace: nvidia-gpu-operator
data:
  any: |-  # <1>
    version: v1
    sharing:
      timeSlicing:
        replicas: 4  # <2>
        renameByDefault: false  # <3>
----
<1> `any` applies to all GPU types; use specific device IDs for heterogeneous configurations
<2> Advertise each physical GPU as 4 allocatable resources (4x oversubscription)
<3> Keep resource name as `nvidia.com/gpu` (vs. `nvidia.com/gpu-shared`)

**Applying Time-Slicing Configuration:**

[source,bash]
----
# Create the ConfigMap
$ oc apply -f device-plugin-config.yaml

# Update ClusterPolicy to reference the ConfigMap
$ oc patch clusterpolicy gpu-cluster-policy --type='json' \
  -p='[{"op": "replace", "path": "/spec/devicePlugin/config/name", \
       "value": "device-plugin-config"}]'

# Device plugin DaemonSet restarts (2-3 minutes)
# GPUs are now oversubscribed by 4x
----

**Production Time-Slicing Example:**

A platform team has 3x NVIDIA T4 GPUs (16GB VRAM each) that do not support MIG. They run small BERT-based inference models averaging 2GB VRAM and <10% GPU compute utilization during request processing.

**Without time-slicing:**

* 3 GPUs available
* 3 inference pods (1 per GPU)
* Average utilization: 20%
* Wasted capacity: 80%

**With time-slicing (`replicas: 8`):**

* 3 physical GPUs → 24 advertised GPU resources
* 24 inference pods can run concurrently
* Average utilization: 75%
* Wasted capacity: 25%
* **Result:** 8x more concurrent workloads, 3.75x better utilization

**Time-Slicing Replica Count Decision Matrix:**

[cols="2,1,3",options="header"]
|===
|Workload Pattern |Replica Count |Expected Utilization Outcome

|Continuous high-compute (>80% GPU usage)
|1-2x
|Avoid time-slicing—use full GPU allocation or MIG

|Moderate compute, bursty (30-50% duty cycle)
|4-8x
|60-80% average utilization, acceptable latency variance

|Low compute, infrequent (<10% duty cycle)
|10-16x
|85%+ average utilization, high latency variance
|===

[WARNING]
====
Time-slicing provides **NO performance isolation**. A runaway process on one container consuming 100% GPU can starve all other containers sharing the GPU. Use time-slicing only for:

* **Trusted workloads** where you control all code running on shared GPUs
* **Dev/test environments** where performance unpredictability is acceptable
* **Non-production inference** where SLAs are not enforced

**Never use time-slicing for:**

* SLA-backed production services
* Multi-tenant platforms with untrusted workloads
* Latency-sensitive applications requiring predictable performance
====

[TIP]
====
Start with conservative time-slicing ratios (`replicas: 2-4`) in production. Monitor GPU utilization with DCGM metrics for 2 weeks. If average utilization remains below 50%, incrementally increase replicas. If workloads experience latency variability or timeouts, reduce replicas or migrate to MIG.
====

=== Preparing for MIG (Hardware Partitioning)

Multi-Instance GPU (MIG) technology partitions a single physical GPU into multiple hardware-isolated GPU instances, each with dedicated compute resources (Streaming Multiprocessors) and memory (framebuffer). MIG is covered comprehensively in Chapter 2—this section introduces concepts and prepares your ClusterPolicy configuration.

**MIG Advantages Over Time-Slicing:**

* **Guaranteed isolation:** Each MIG instance has dedicated SMs and VRAM—no resource contention
* **Predictable performance:** Workloads on MIG instances never experience performance variance from co-located workloads
* **QoS guarantees:** Enables SLA-backed inference services on multi-tenant platforms
* **Fault isolation:** Errors in one MIG instance do not affect other instances
* **Higher utilization:** 85-95% average utilization vs. 60-80% with time-slicing

**MIG Disadvantages:**

* **Hardware requirement:** Requires Ampere or Hopper architecture (A30, A100, H100)—V100, T4, A10 do not support MIG
* **Reconfiguration downtime:** Changing MIG profiles requires node drain and driver restart (10-15 minutes per node)
* **Fixed partitions:** MIG profiles are predefined (e.g., 1g.10gb, 2g.20gb, 3g.40gb)—cannot create arbitrary partition sizes

**MIG Partitioning Visualization (A100-40GB):**

----
Physical NVIDIA A100-40GB
├── 40GB VRAM (framebuffer)
└── 108 Streaming Multiprocessors (SMs)

MIG Profile: all-1g.5gb (homogeneous)
┌──────────────────────────────────────────────┐
│ [1g.5gb] [1g.5gb] [1g.5gb] [1g.5gb] [1g.5gb] │
│ [1g.5gb] [1g.5gb]                            │
│                                              │
│ 7 instances, each with:                     │
│   - 5GB VRAM                                 │
│   - 14 SMs (~13% compute)                    │
│   - Advertised as: nvidia.com/mig-1g.5gb     │
└──────────────────────────────────────────────┘

MIG Profile: mixed (heterogeneous)
┌──────────────────────────────────────────────┐
│ [1g.5gb] [1g.5gb] [3g.20gb──────────────]    │
│                                              │
│ 3 instances:                                 │
│   - 2x 1g.5gb (small models, 5GB each)       │
│   - 1x 3g.20gb (large model, 20GB VRAM)      │
│   - Flexible allocation for diverse workloads│
└──────────────────────────────────────────────┘
----

**Production MIG Example:**

A FinTech MaaS platform serves 15 models:

* 10x small models (DistilBERT variants, ~3GB VRAM requirement)
* 5x large models (Llama-2-7B, ~15GB VRAM requirement)

**MIG Configuration:**

Configure 5x NVIDIA A100-80GB GPUs with `mixed` MIG profile:

* Each GPU: 3x `1g.10gb` instances + 1x `3g.40gb` instance
* Total capacity: 15x `1g.10gb` (small) + 5x `3g.40gb` (large)
* Perfect fit for 10 small + 5 large models
* Average utilization: 92%
* **Result:** $200K GPU investment serves 15 concurrent models instead of 5

**ClusterPolicy MIG Preparation:**

[source,yaml]
----
spec:
  migManager:
    enabled: true  # <1>
  gfd:
    enabled: true  # <2>
----
<1> Enable MIG Manager for profile automation (Chapter 2)
<2> GPU Feature Discovery labels MIG instances with profiles

MIG configuration via node labels, custom mig-parted configurations, and verification are covered in Chapter 2.

[IMPORTANT]
====
MIG reconfiguration requires draining GPU workloads and restarting driver pods. Plan MIG profile changes during maintenance windows. Typical downtime: 10-15 minutes per node (drain + driver restart + MIG apply + pod reschedule).
====

== Integrating with OpenShift Monitoring

GPU telemetry must integrate with OpenShift's monitoring stack for unified observability. The DCGM Exporter exposes GPU metrics in Prometheus format, but these metrics are only collected if the GPU Operator namespace is configured for monitoring.

=== DCGM Exporter and Prometheus Integration

OpenShift's monitoring stack is based on Prometheus for metric collection and Thanos for long-term storage and federated queries. The DCGM Exporter integrates with this stack via ServiceMonitor resources.

**Integration Architecture:**

----
DCGM Daemon (per GPU node)
  ↓ collects GPU metrics every 30s
DCGM Exporter DaemonSet (port 9400/metrics)
  ↓ exposes Prometheus-format metrics
ServiceMonitor (nvidia-dcgm-exporter)
  ↓ discovered by Prometheus Operator
Prometheus User Workload Monitoring
  ↓ scrapes endpoints, stores time-series
Thanos Querier
  ↓ federated queries, long-term storage
Grafana (Chapter 3)
  ↓ visualization and dashboards
----

The GPU Operator automatically creates a ServiceMonitor resource when `dcgmExporter.enabled: true`. However, Prometheus only scrapes ServiceMonitors in namespaces labeled for monitoring.

=== Enabling Metric Collection

OpenShift's user workload monitoring must be configured to discover ServiceMonitors in the `nvidia-gpu-operator` namespace.

**Apply Monitoring Label to Namespace:**

[source,bash]
----
$ oc label namespace nvidia-gpu-operator \
  openshift.io/cluster-monitoring=true
----

**Why This Label Matters:**

* Without the label: Prometheus ignores ServiceMonitors in the namespace
* With the label: Prometheus discovers the `nvidia-dcgm-exporter` ServiceMonitor and begins scraping GPU metrics
* The label enables user-workload Prometheus to monitor custom applications and operators

**Verify ServiceMonitor Discovery:**

[source,bash]

== Production Operational Considerations

After deployment, understanding Day 2 operations is critical for maintaining a production MaaS platform. GPU platform changes have different impacts on workload availability, and choosing the right maintenance windows prevents service disruptions.

=== Day 2 Operations: Reconfiguration Impact

Different ClusterPolicy changes trigger different levels of disruption. Plan changes according to platform SLAs and traffic patterns.

[cols="2,1,2,2",options="header"]
|===
|Configuration Change |Pod Restarts Required |Workload Downtime |Recommended Change Window

|Add new GPU node to cluster
|No
|None—new capacity is added online
|Anytime

|Driver version upgrade (`driver.version`)
|Yes—driver DaemonSet pods restart
|5-8 minutes per node (rolling restart)
|Maintenance window

|Enable time-slicing (add `devicePlugin.config.name`)
|Yes—device plugin DaemonSet restarts
|2-3 minutes (device plugin restart)
|Low-traffic period

|Change MIG profile (node label)
|Yes—full node drain required, driver restart
|10-15 minutes per node (drain + reconfigure)
|Maintenance window

|Update DCGM configuration
|Yes—DCGM pods restart
|1-2 minutes (monitoring gap only)
|Anytime

|Add/remove GPU node label
|No—scheduler updates routing
|None—gradual workload migration
|Anytime
|===

**Example: Driver Version Upgrade Workflow**

. Update ClusterPolicy with new driver version:
+
[source,bash]
----
$ oc patch clusterpolicy gpu-cluster-policy --type='json' \
  -p='[{"op": "replace", "path": "/spec/driver/version", \
       "value": "535.161.07"}]'
----

. Driver DaemonSet performs rolling restart (one node at a time)
. Each node experiences 5-8 minutes of GPU unavailability during driver reload
. Total platform downtime: 0 (rolling restart) if you have >1 GPU node
. Total time for 10-node cluster: ~60 minutes (rolling)

[IMPORTANT]
====
Always test driver upgrades in a dev environment first. While NVIDIA maintains backward compatibility, specific CUDA workloads may have version sensitivities. Validate inference services, training jobs, and custom CUDA code against the new driver before upgrading production.
====

=== Workload Placement Best Practices

Use GPU Feature Discovery labels to ensure workloads land on appropriate GPU hardware.

**Example 1: Target specific GPU models for large LLMs**

[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: llama-70b-inference
spec:
  containers:
  - name: vllm-server
    image: vllm/vllm-openai:latest
    resources:
      limits:
        nvidia.com/gpu: 1
  nodeSelector:
    nvidia.com/gpu.product: "A100-PCIE-40GB"  # <1>
    nvidia.com/gpu.memory: "40960"  # <2>
----
<1> Only schedule on A100 GPUs (not T4 or V100)
<2> Require 40GB VRAM minimum

**Example 2: Ensure MIG capability for future flexibility**

[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: multi-tenant-inference
spec:
  containers:
  - name: inference-service
    image: inference:latest
    resources:
      limits:
        nvidia.com/gpu: 1
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: nvidia.com/mig.capable
            operator: In
            values: ["true"]  # <1>
----
<1> Only schedule on MIG-capable GPUs even if MIG is not currently enabled—allows future migration to MIG without workload changes

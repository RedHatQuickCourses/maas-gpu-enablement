= Hands-On Lab: Deploy and Configure Models-as-a-Service

_Estimated completion time: *{time_estimate} minutes*._

Objective::

In this hands-on lab, you will deploy and configure the complete Models-as-a-Service (MaaS) platform for ChocoTech Industries. You will enable the MaaS backend component, configure TLS for secure authentication, create organizational groups, publish models to the MaaS gateway, establish subscription-based governance with token limits, configure authorization policies, and generate API keys for secure model access. By the end of this lab, you will have a production-ready MaaS platform that enforces the Gold, Silver, and Bronze subscription plans required by ChocoTech's CIO.

== Lab Scenario

Recall ChocoTech's business requirements from the CIO:

[cols="1a,1a,2a",options="header",frame="all",grid="rows"]
|===
| Business Unit | Subscription Plan | Infrastructure Requirement

| **Quality Control**
| Gold Plan
| Production-grade access with guaranteed availability and highest token limits for real-time defect detection.

| **R&D**
| Silver Plan
| Development access with moderate token limits for recipe optimization testing.

| **Marketing**
| Bronze Plan
| Exploratory access with strict token limits for sentiment analysis to prevent budget overruns.
|===

Your mission: Build the governance infrastructure to deliver these three distinct service levels using MaaS subscriptions, authorization policies, and API key authentication.

=== Step 4: Verify Llama Stack Operator

The Llama Stack Operator is required for MaaS dashboard features including Gen AI Studio, API Keys page, and Playground.

**Check operator enablement in DataScienceCluster:**

[source,bash]
----
oc get datasciencecluster default -n redhat-ods-applications -o jsonpath='{.spec.components.llamastackoperator.managementState}'
----

Expected output: `Managed`

**Verify Llama Stack Operator pods:**

[source,bash]
----
oc get pods -n redhat-ods-applications -l app.kubernetes.io/name=llama-stack-operator
----

Expected output:
[source,text]
----
NAME                                      READY   STATUS    AGE
llama-stack-operator-5d8c9f6b5c-x4j2m     1/1     Running   2d
----

[IMPORTANT]
====
**If Llama Stack Operator is not enabled**, you will not be able to access:

* **Gen AI Studio** menu in the dashboard
* **API Keys** management page
* **AI Asset Endpoints** page
* **Playground** for model testing

Enable the operator:

[source,bash]
----
oc patch datasciencecluster default -n redhat-ods-applications \
  --type merge \
  -p '{"spec":{"components":{"llamastackoperator":{"managementState":"Managed"}}}}'
----

Wait ~60 seconds for the operator pod to start.
====

=== Step 5: Verify User Workload Monitoring

User Workload Monitoring is **critical** for MaaS. Without it, the Tenant resource will show `Degraded` status and MaaS metrics will not be collected.

**Check if User Workload Monitoring is enabled:**

[source,bash]
----
oc get configmap cluster-monitoring-config -n openshift-monitoring -o jsonpath='{.data.config\.yaml}' | grep enableUserWorkload
----

Expected output:
[source,yaml]
----
enableUserWorkload: true
----

**Verify User Workload Monitoring pods are running:**

[source,bash]
----
oc get pods -n openshift-user-workload-monitoring
----

Expected output (pods Running):
[source,text]
----
NAME                                   READY   STATUS    AGE
prometheus-operator-5f8d9c7b6d-q5k8m   2/2     Running   5d
prometheus-user-workload-0             6/6     Running   5d
prometheus-user-workload-1             6/6     Running   5d
thanos-ruler-user-workload-0           4/4     Running   5d
thanos-ruler-user-workload-1           4/4     Running   5d
----

[WARNING]
====
**If User Workload Monitoring is not enabled**, the MaaS Tenant will show `Degraded` status with the error:

```
User Workload Monitoring is not enabled
```

**Enable User Workload Monitoring:**

[source,bash]
----
cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF
----

Wait 2-3 minutes for the `openshift-user-workload-monitoring` namespace to be created and pods to start.
====

=== Step 6: Verify PostgreSQL Database Secret

MaaS requires a PostgreSQL database for API key storage and subscription metadata.

**Check for database secret:**

[source,bash]
----
oc get secret maas-db-config -n redhat-ods-applications
----

Expected output:
[source,text]
----
NAME             TYPE     DATA   AGE
maas-db-config   Opaque   1      5d
----

**Verify secret contains database connection string:**

[source,bash]
----
oc get secret maas-db-config -n redhat-ods-applications -o jsonpath='{.data.connection-string}' | base64 -d | cut -c1-20
----

Expected output (first 20 characters):
[source,text]
----
postgresql://maas_us
----

[NOTE]
====
The database secret should contain a connection string in the format:

[source,text]
----
postgresql://<username>:<password>@<host>:<port>/<database>?sslmode=require
----

Your cluster administrator has provisioned this PostgreSQL database during the initial OpenShift AI setup. The database stores:

* API keys and their metadata (owner, subscription, expiration)
* Subscription configurations
* Token consumption metrics (for observability)
====

**Troubleshooting:**

If the secret is missing, contact your cluster administrator. The database and secret are prerequisites that must be configured before enabling MaaS.

=== Step 7: Verify Gateway API Resources

MaaS uses OpenShift's Gateway API for external model access routing.

**Check GatewayClass:**

[source,bash]
----
oc get gatewayclass openshift-gateway-api
----

Expected output:
[source,text]
----
NAME                     CONTROLLER                        ACCEPTED   AGE
openshift-gateway-api    openshift.io/gateway-controller   True       5d
----

**Check MaaS Gateway:**

[source,bash]
----
oc get gateway maas-default-gateway -n openshift-ingress
----

Expected output:
[source,text]
----
NAME                   CLASS                    ADDRESS   PROGRAMMED   AGE
maas-default-gateway   openshift-gateway-api              True         5d
----

Verify Gateway is programmed:

[source,bash]
----
oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.conditions[?(@.type=="Programmed")].status}'
----

Expected output: `True`

[IMPORTANT]
====
**If the Gateway does not exist**, your cluster administrator must create it. The Gateway defines the external API endpoint for MaaS model access.

Example Gateway configuration:

[source,yaml]
----
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: maas-default-gateway
  namespace: openshift-ingress
spec:
  gatewayClassName: openshift-gateway-api
  listeners:
  - name: https
    port: 443
    protocol: HTTPS
    hostname: "maas.apps.<cluster-domain>"
    tls:
      mode: Terminate
      certificateRefs:
      - kind: Secret
        name: maas-tls-cert
----

Consult the OpenShift AI documentation for complete Gateway setup instructions.
====

=== Step 8: (Optional) Verify Cluster Observability Operator

The Cluster Observability Operator enables the MaaS observability dashboard (Technology Preview) for token consumption tracking and usage analytics.

[NOTE]
====
This operator is **optional**. MaaS core functionality works without it. The observability dashboard is a Tech Preview feature for usage monitoring and showback reporting.
====

**Check if Cluster Observability Operator is installed:**

[source,bash]
----
oc get csv -n openshift-operators | grep cluster-observability-operator
----

Expected output (if installed):
[source,text]
----
cluster-observability-operator.v0.3.0   Cluster Observability Operator   0.3.0   Succeeded
----

**If not installed and you want observability features:**

1. Open OpenShift Console → **Operators → OperatorHub**
2. Search for **"Cluster Observability Operator"**
3. Click **Install**
4. Select **stable** channel
5. Install to **All namespaces**
6. Click **Install**

After installation, you'll need to enable observability in:

* Kuadrant: `spec.observability.enable: true`
* Tenant: `spec.telemetry.enabled: true`

=== Step 9: Prerequisites Verification Summary

Run this comprehensive verification script to confirm all components:

[source,bash]
----
cat > verify-maas-prereqs.sh <<'EOF'
#!/bin/bash

echo "=== MaaS Prerequisites Verification ==="
echo ""

# 1. OpenShift Version
echo "1. OpenShift Platform Version:"
OCP_VERSION=$(oc version -o json 2>/dev/null | jq -r '.openshiftVersion' 2>/dev/null || echo "ERROR")
echo "   Version: ${OCP_VERSION}"
[[ "${OCP_VERSION}" == "ERROR" ]] && echo "   ❌ FAILED" || echo "   ✅ PASSED"
echo ""

# 2. OpenShift AI Operator
echo "2. Red Hat OpenShift AI Operator:"
RHOAI_CSV=$(oc get csv -n redhat-ods-applications 2>/dev/null | grep rhods-operator | awk '{print $1}')
echo "   CSV: ${RHOAI_CSV:-NOT FOUND}"
[[ -z "${RHOAI_CSV}" ]] && echo "   ❌ FAILED" || echo "   ✅ PASSED"
echo ""

# 3. Red Hat Connectivity Link
echo "3. Red Hat Connectivity Link Operator:"
RCL_CSV=$(oc get csv -n openshift-operators 2>/dev/null | grep connectivity-link | awk '{print $1}')
echo "   CSV: ${RCL_CSV:-NOT FOUND}"
KUADRANT_STATUS=$(oc get kuadrant kuadrant -n kuadrant-system -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}' 2>/dev/null || echo "ERROR")
echo "   Kuadrant Ready: ${KUADRANT_STATUS}"
[[ "${KUADRANT_STATUS}" == "True" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 4. Llama Stack Operator
echo "4. Llama Stack Operator:"
LLAMA_STATE=$(oc get datasciencecluster default -n redhat-ods-applications -o jsonpath='{.spec.components.llamastackoperator.managementState}' 2>/dev/null || echo "ERROR")
echo "   Management State: ${LLAMA_STATE}"
[[ "${LLAMA_STATE}" == "Managed" ]] && echo "   ✅ PASSED" || echo "   ⚠️  WARNING: Required for dashboard features"
echo ""

# 5. User Workload Monitoring
echo "5. User Workload Monitoring:"
UWM_ENABLED=$(oc get configmap cluster-monitoring-config -n openshift-monitoring -o jsonpath='{.data.config\.yaml}' 2>/dev/null | grep -o "enableUserWorkload: true" || echo "NOT ENABLED")
echo "   Status: ${UWM_ENABLED}"
[[ "${UWM_ENABLED}" == "enableUserWorkload: true" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 6. PostgreSQL Database Secret
echo "6. PostgreSQL Database Secret:"
DB_SECRET=$(oc get secret maas-db-config -n redhat-ods-applications -o jsonpath='{.metadata.name}' 2>/dev/null || echo "NOT FOUND")
echo "   Secret: ${DB_SECRET}"
[[ "${DB_SECRET}" == "maas-db-config" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 7. Gateway API
echo "7. Gateway API Resources:"
GW_CLASS=$(oc get gatewayclass openshift-gateway-api -o jsonpath='{.metadata.name}' 2>/dev/null || echo "NOT FOUND")
echo "   GatewayClass: ${GW_CLASS}"
GW_STATUS=$(oc get gateway maas-default-gateway -n openshift-ingress -o jsonpath='{.status.conditions[?(@.type=="Programmed")].status}' 2>/dev/null || echo "NOT FOUND")
echo "   Gateway Programmed: ${GW_STATUS}"
[[ "${GW_STATUS}" == "True" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

# 8. KServe Component
echo "8. KServe Component (Required for MaaS):"
KSERVE_STATE=$(oc get datasciencecluster default -n redhat-ods-applications -o jsonpath='{.spec.components.kserve.managementState}' 2>/dev/null || echo "ERROR")
echo "   Management State: ${KSERVE_STATE}"
[[ "${KSERVE_STATE}" == "Managed" ]] && echo "   ✅ PASSED" || echo "   ❌ FAILED"
echo ""

echo "=== Summary ==="
echo "Review any ❌ FAILED items above before proceeding with MaaS deployment."
echo ""
echo "Optional Components:"
echo "  • Cluster Observability Operator: For MaaS usage dashboard (Tech Preview)"
echo "  • ExternalModel CRD: For routing to external LLM providers (Tech Preview)"
EOF

chmod +x verify-maas-prereqs.sh
./verify-maas-prereqs.sh
----

Expected output shows ✅ PASSED for all required components.

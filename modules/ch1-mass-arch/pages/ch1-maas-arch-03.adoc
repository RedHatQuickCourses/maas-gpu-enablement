= 1.2: The MaaS architecture stack

Now that we understand the business problem—ChocoTech’s fragmented hardware and lack of centralized governance—we must look at *how* OpenShift AI solves it. 

Every request for model inference passes through three highly specialized layers: **The Gateway API**, **Kuadrant**, and **KServe**. 

---

== The Three-Layer Stack: Separation of Concerns

By decoupling the network routing, the policy governance, and the execution engine, MaaS provides extreme flexibility. You can adjust user rate limits without redeploying massive models, and you can swap serving runtimes without touching the security perimeter.

[cols="1a,2a,2a",options="header",frame="all",grid="rows"]
|===
| Layer | Core Component & Role | Enterprise Value ("The Why")

| **Layer 1: The Front Door**
| **Gateway API (Traffic Routing)**

* Exposes a single, unified endpoint (`maas-default-gateway`) for all models across the cluster.
* Handles TLS termination and path-based routing (e.g., `/v1/chat/completions`).
| **Simplified Connectivity:** Developers do not need to manage complex, per-model DNS routing or insecure internal IPs. It provides a standard, OpenAI-compatible entry point.

| **Layer 2: The Bouncer**
| **Kuadrant (Policy & Governance)**

* **Authorino:** Validates tokens and enforces zero-trust authentication. 
* **Limitador:** Tracks token consumption and enforces tier-based rate limits.
| **Security & Chargebacks:** This layer mathematically guarantees that a team cannot exhaust GPU resources. It provides the exact token-usage telemetry required for Finance to bill departments accurately.

| **Layer 3: The Engine Room**

| **KServe & vLLM/llm-d (Model Execution)**

* Orchestrates the actual model serving infrastructure.
* Uses distributed inference frameworks (like llm-d) to spread large models across multiple GPUs.
| **Maximum ROI:** By decoupling the request handling from low-level GPU tensor operations, KServe maximizes throughput and minimizes idle hardware time. 
|===

== The Life of an Inference Request

To understand how these components work together, let's trace a real request from ChocoTech's Marketing team as they try to use the shared `llama-3-8b` model for a sentiment analysis application.

1. **The Request Arrives (Gateway API):** The Marketing application sends a POST request to `/v1/chat/completions` containing their Bearer token and the prompt data. The Gateway API catches this at the network edge.
2. **Identity Verification (Kuadrant - Authorino):** The Gateway pauses the request and asks Authorino, "Is this token valid, and what subscription level does this user have?" Authorino confirms the user is in the "Bronze" subscription group. Next it checks the Authorization policy which (TobeAdded)
(If the token was invalid, it would immediately return a `401 Unauthorized` without touching the GPU).
3. **Policy Enforcement (Kuadrant - Limitador):** Policies (MaaSAuthPolicy) and subscriptions (MaaSSubscription) work together. Both use RBAC references (subjects or owners). A user must have both a matching policy (access) and a matching subscription (quota) to use a model. Limitador checks the Bronze subscription quota. "Has the Marketing team exceeded their 10,000 tokens-per-hour limit?" If they are within limits, the request is permitted. (If exceeded, it returns a `429 Too Many Requests`).
4. **Execution (KServe):** The validated, governed request is handed off to KServe. The vLLM runtime processes the prompt against the `llama-3-8b` weights and generates the response.
5. **The Audit Log:** The response flows back to the user, and Kuadrant meticulously logs the exact number of `prompt_tokens` and `completion_tokens` consumed by the Marketing application for the chargeback report.

This layered approach ensures that every request is authenticated, authorized, rate-limited, audited, and routed to the correct model serving infrastructure.

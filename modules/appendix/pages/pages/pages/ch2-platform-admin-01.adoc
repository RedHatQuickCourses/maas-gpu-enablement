The CIO at ChocoTech Industries just called an emergency meeting. 

"The infrastructure is ready," she says, "but before we deploy any models, I need to know how we are going to control access and manage costs across business units." 

She hands you the following requirements document:

[cols="1a,1a,2a",options="header",frame="all",grid="rows"]
|===
| Business Unit | Subscription Plan | Infrastructure Requirement

| **Quality Control**
| Gold Plan
| Needs production-grade access with guaranteed availability and the highest token limits for real-time defect detection. 

| **R&D**
| Silver Plan
| Needs development access with moderate token limits for recipe optimization testing.

| **Marketing**
| Bronze Plan
| Needs exploratory access with strict token limits for sentiment analysis to prevent budget overruns.
|===

Furthermore, the CIO mandates that all access must be authenticated to create definitive audit trails, and strict token quotas must prevent any single team from monopolizing the GPU resources.

== Learning objectives

By the end of this module, you will be able to:

* Explain how subscription-based governance prevents resource exhaustion and enables accurate chargebacks.
* Create OpenShift groups and map them to logical business units.
* Deploy `MaaSSubscription` custom resources to define token limits and group access rules.
* Deploy `MaaSAuthPolicy` custom resources to authorize Gateway traffic.
* Generate and manage permanent or temporary API Keys for secure model consumption.

---

== The deployment strategy

To achieve this, you cannot simply deploy a model and hope it works. You must build the governance foundation from the bottom up. In this module, you will execute the following strategy:

1. **Organizational Structure:** First, you will create the native OpenShift groups for ChocoTech's business units (Quality Control, R&D, Marketing) and prepare a dedicated namespace. 
2. **Subscription Governance:** Next, you will deploy `MaaSSubscription` resources to create the Gold, Silver, and Bronze plans, mapping them directly to your OpenShift groups.
3. **Gateway Authorization:** You will apply the `MaaSAuthPolicy` to physically open the network Gateway for your subscribed groups.
4. **API Key Generation:** Finally, you will generate the secure API keys that your developers and applications will use to interact with the models.

By the end of this module, ChocoTech's MaaS platform will be fully secured, authenticated, and ready to host high-performance models without the risk of resource exhaustion.

= 1.3: Enabling MaaS on your OpenShift AI platform

You've explored the business problem and understand the three-layer architecture. Now it's time to activate MaaS on ChocoTech's OpenShift cluster.

ChocoTech's GPU infrastructure is already operational from prior AI/ML implementations. Quality Control's A100 GPUs, R&D's L40S accelerators, Marketing's L4 GPUs, and Supply Chain's hardware are all running and handling existing workloads. Rather than starting from scratch, you'll transform this fragmented infrastructure into a centralized MaaS platform by enabling the components that provide governance, routing, and multi-tenant model serving.

This module guides you through the activation sequence: verifying required operators, creating infrastructure components, and enabling MaaS features in the OpenShift AI platform. By the end, ChocoTech's platform will be ready for the detailed configuration work in Module 2.

== Step 1: Configuration Check & Operator Verification

Before applying any configuration, ensure your cluster has the necessary operators installed from Ecosystem (previously OperatorHub). If any of these are missing, the deployment will silently fail—the commands will appear to succeed, but the expected pods and resources will never materialize.

*_ChocoTech's platform team learned this lesson the hard way when they attempted to enable MaaS without the Connectivity Link operator installed. The DataScienceCluster patch appeared to succeed, but the maas-api pods never appeared—no error messages, just silent failure. Verifying operators upfront prevents these frustrating troubleshooting sessions._*


== **Required operators**:

Red Hat OpenShift Service Mesh 3::
Provides the underlying Istio networking that powers the Gateway API implementation.

Red Hat Connectivity Link (v1.2+)::
Delivers the Kuadrant governance framework. The Connectivity Link operator manages the Kuadrant controller, which in turn deploys Authorino (authentication/authorization) and Limitador (rate limiting).

Red Hat OpenShift AI (v3.3+)::
The core platform operator that manages the DataScienceCluster and deploys MaaS components when enabled.

Leader Worker Set Operator::
Mandatory for llm-d orchestration. Distributed pod management for ultra-large models that exceed single-node VRAM capacity.

== Verify operator installation

Check that all required operators are installed and healthy:

[source,role="execute"]
----
oc get csv -n openshift-operators | grep -E "(servicemesh|rhcl|rhods|limitador|authorino)"
----

Expected output (all operators should show Succeeded):

----
authorino-operator.v1.3.0                 Authorino Operator                 1.3.0                 Succeeded
limitador-operator.v1.3.0                 Limitador Operator                 1.3.0                 Succeeded
rhcl-operator.v1.3.3                      Red Hat Connectivity Link          1.3.3                 Succeeded
rhods-operator.3.3.2                      Red Hat OpenShift AI               3.3.2                 Succeeded
servicemeshoperator3.v3.3.2               Red Hat OpenShift Service Mesh 3   3.3.2                 Succeeded
----

If any operators are missing, install them from OperatorHub before proceeding:

1. Navigate to **Ecosystem → Software Catalog** in the OpenShift console
2. Search for the missing operator by name
3. Click **Install** and follow the prompts
4. Wait for the operator to reach **Succeeded** status before continuing

== Step 2: Infrastructure Foundation

The OpenShift AI operator is a strict gatekeeper. It will not provision the MaaS API unless the foundational network routing and security engines are actively running on the cluster. This phase creates those prerequisites.

=== Initialize the Governance Engine (Kuadrant)

The Connectivity Link operator manages Kuadrant, but Kuadrant itself must be explicitly initialized. Creating a Kuadrant custom resource tells the operator to spin up Authorino (authentication/authorization) and Limitador (rate limiting and quota enforcement).

*_ChocoTech needs Kuadrant to enforce the tier-based quotas that prevent Marketing's bronze-tier applications from monopolizing GPU resources during campaign launches. Without Kuadrant's Limitador component, there would be no mechanism to track and limit token consumption across teams._*

---

==== Create the Kuadrant instance:

[source,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata:
  name: kuadrant
  namespace: openshift-operators
EOF
----

Expected output:

----
kuadrant.kuadrant.io/kuadrant created
----

Verify Kuadrant components are running:

[source,role="execute"]
----
oc get pods -n openshift-operators | grep -E "(kuadrant|limitador|authorino)"
----

Expected output (pods should reach Running status within 1-2 minutes):

----
NAME                                                    READY    STATUS    RESTARTS   AGE
authorino-xxxxxxxxx-xxxxx                                1/1     Running       0      2m41s
authorino-operator-xxxxxxxxx-xxxxx                       1/1     Running       0      44m
kuadrant-console-plugin-xxxxxxxxx-xxxxx                  1/1     Running       0      12m
kuadrant-operator-controller-manager-xxxxxxxxx-xxxxx     1/1     Running       0      12m
limitador-limitador-xxxxxxxxx-xxxxx                      1/1     Running       0      2m42s
limitador-operator-controller-manager-xxxxxxxxx-xxxxx    1/1     Running       0      12m
----

**Why this matters**: Authorino validates Bearer tokens and enforces model access policies. Limitador tracks requests per minute and token consumption quotas. Both are mandatory for the MaaS API to function.

---

=== Create the MaaS Gateway

KServe requires a specific Gateway resource to route external traffic to your models. This Gateway acts as the single API endpoint for all model inference traffic across ChocoTech's business units.

*The maas-default-gateway becomes the single API endpoint for all of ChocoTech's model inference traffic. Instead of Quality Control, R&D, Marketing, and Supply Chain each managing separate endpoints, they all send requests to one gateway that handles routing, authentication, and policy enforcement.*

Create the Gateway resource:

[source,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: maas-default-gateway
  namespace: openshift-ingress
spec:
  gatewayClassName: data-science-gateway-class 
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    allowedRoutes:
      namespaces:
        from: All
EOF
----

Expected output:

----
gateway.gateway.networking.k8s.io/maas-default-gateway created
----

Verify the Gateway is programmed correctly:

[source,role="execute"]
----
oc get gateway maas-default-gateway -n openshift-ingress
----

Expected output (PROGRAMMED should show True):

----
NAME                   CLASS                          ADDRESS   PROGRAMMED   AGE
maas-default-gateway   openshift-gateway-controller             True         7s
----

== Phase 3: Platform Activation

With the infrastructure in place—operators verified, Kuadrant running, Gateway created—you can now instruct the OpenShift AI operator to enable MaaS features.

=== Enable the MaaS Backend

Patching the DataScienceCluster sets the `modelsAsService` management state to `Managed`. This prompts the operator to build the maas-api pods and default RBAC roles that provide the MaaS API layer.

*This patch tells the OpenShift AI operator to provision the MaaS API layer. Once enabled, Platform Engineers can deploy models to a central catalog that all four business units can access via token-based authentication, eliminating the redundant Llama-3 deployments that were wasting 128GB of storage.*

Apply the DataScienceCluster patch:

[source,role="execute"]
----
oc patch datasciencecluster default-dsc --type=merge -p '{
  "spec": {
    "components": {
      "kserve": {
        "managementState": "Managed",
        "modelsAsService": {
          "managementState": "Managed"
        }
      }
    }
  }
}'
----

Expected output:

----
datasciencecluster.datasciencecluster.opendatahub.io/default-dsc patched
----

Verify the MaaS API pods are created:

[source,role="execute"]
----
oc get pods -n redhat-ods-applications | grep maas
----

Expected output (pods should appear within 1-2 minutes):

----
maas-api-xxxxxxxxx-xxxxx   1/1     Running   0          45s
----


=== Enable the Dashboard (Gen AI Studio)

Even though you're deploying models via GitOps in later modules, Platform Administrators still need the dashboard UI to manage user access tokens and monitor model usage. Enable the Gen AI Studio interface to provide self-service token generation.

*The Gen AI Studio interface allows ChocoTech's Platform Administrators to manage user access tokens and monitor model usage without writing custom scripts. Marketing's team lead can now generate API tokens for their developers directly through the dashboard, rather than filing IT tickets.*

Apply the dashboard configuration patch:

[source,role="execute"]
----
oc patch odhdashboardconfig odh-dashboard-config -n redhat-ods-applications --type=merge -p '{
  "spec": {
    "dashboardConfig": {
      "modelAsService": true,
      "genAiStudio": true
    }
  }
}'
----

Expected output:

----
odhdashboardconfig.opendatahub.io/odh-dashboard-config patched (no-change)
----

Verify the dashboard configuration:

[source,role="execute"]
----
oc get odhdashboardconfig odh-dashboard-config -n redhat-ods-applications -o jsonpath='{.spec.dashboardConfig}'
----

Expected output (should include both flags as true):

----
{"genAiStudio":true,"modelAsService":true}
----

The Gen AI Studio menu option will appear in the OpenShift AI dashboard after refreshing your browser. If it doesn't appear immediately, clear your browser cache and reload the page.

== Verification

Confirm all MaaS components are activated and healthy:

* ✓ Required operators (Service Mesh, Connectivity Link, OpenShift AI, LeaderWorkerSet) are installed with Succeeded status
* ✓ Kuadrant pods (authorino, limitador, kuadrant-controller) are Running in kuadrant-system namespace
* ✓ Gateway resource (maas-default-gateway) exists in openshift-ingress namespace with PROGRAMMED=True
* ✓ MaaS API pods are Running in redhat-ods-applications namespace
* ✓ DataScienceCluster shows modelsAsService.managementState=Managed
* ✓ Dashboard config has modelAsService and genAiStudio enabled

= Multi-GPU Strategies: Tensor Parallelism vs. Pipeline Parallelism
:author: Platform Engineering Guide
:toc: left
:icons: font
:stem: latex

**Goal:** Gain the technical expertise to distribute massive models across multiple GPUs. As models grow beyond the VRAM of a single H100/B200 (80GB–192GB), platform engineers must decide *how* to chop up the model without destroying inference performance.

---

== 1. Tensor Parallelism (TP) - "Splitting the Layer"

**Tensor Parallelism** splits individual weight matrices (tensors) across multiple GPUs. Instead of one GPU doing the whole math for a layer, four GPUs each do 25% of the math for that same layer simultaneously.

* **How it works:** Each GPU holds a "shard" of the layer. During the forward pass, GPUs must synchronize their partial results to get the final output of that layer.
* **Best For:** Intra-node scaling (GPUs within the same physical server).
* **Benefit:** Massive reduction in latency. Since math is done in parallel, the "Time per Token" drops significantly.



== 2. Pipeline Parallelism (PP) - "Splitting the Stack"

**Pipeline Parallelism** divides the model by its layers. For a 100-layer model, GPU 0 handles layers 1–50, and GPU 1 handles layers 51–100.

* **How it works:** GPU 0 finishes its layers and passes the "activations" (intermediate data) to GPU 1.
* **The "Bubble" Problem:** While GPU 1 is working, GPU 0 is idle. To fix this, we use **micro-batching**, where we feed small chunks of data through the pipeline so all GPUs stay busy most of the time.
* **Best For:** Inter-node scaling (linking multiple servers together) and models that are too large to fit in a single node.

---

== 3. Networking Requirements: The Performance Killer

The choice between TP and PP is almost entirely driven by your **interconnect speed**.

[cols="1,2,2", options="header"]
|===
| Feature | Tensor Parallelism (TP) | Pipeline Parallelism (PP)
| **Communication Frequency** | **Extremely High** (Multiple times per layer) | **Low** (Once per group of layers)
| **Data Transferred** | Large (Weights/Gradients) | Small (Activations only)
| **Required Latency** | < 1–2 microseconds | Milliseconds (more tolerant)
| **Ideal Hardware** | **NVLink / NVSwitch** | **InfiniBand / RoCE / 400G Eth**
| **Scope** | Intra-Node (Inside 1 box) | Inter-Node (Across boxes)
|===



[IMPORTANT]
====
**Platform Engineering Rule of Thumb:** Never run Tensor Parallelism over standard Ethernet or even InfiniBand. The "all-reduce" synchronization happens so often that the GPUs will spend 90% of their time waiting for the network. **TP stays on the NVLink bus.**
====

---

== 4. Decision Matrix: Scaling your Cluster

[cols="1,3", options="header"]
|===
| Scenario | Strategy
| **Model fits in 1 Node** | Use **Tensor Parallelism (TP)** across all 8 GPUs in the box for max speed.
| **Model fits in 2+ Nodes** | Use **TP** within each node (8 GPUs) and **Pipeline Parallelism (PP)** between the nodes.
| **Low Latency Req** | Maximize **TP**. It is faster than PP because it avoids the "pipeline bubble."
| **High Throughput (Batch)** | Use **Data Parallelism (DP)** + **PP**.
|===

== 5. Multi-GPU Knowledge Quizzes

****
**Quiz 3: The Latency Bottleneck**

**Scenario:** You have two servers. Each server has 8x H100 GPUs connected by NVLink. The two servers are connected to each other via standard 10Gbps Ethernet. 

A developer asks you to run a massive model using **Tensor Parallelism (TP) across all 16 GPUs** (8 in box A, 8 in box B). 

* **Question:** Why is this a bad architectural decision?
* **A.** The model will be too smart for the network.
* **B.** Tensor Parallelism requires ultra-low latency synchronization that 10Gbps Ethernet cannot provide, leading to a massive performance collapse.
* **C.** You cannot use TP on more than 8 GPUs.

.Reveal Solution
[%collapsible]
=====
* **Answer:** **B.** TP requires the high-speed, low-latency fabric of NVLink/NVSwitch. Crossing a standard Ethernet cable for "all-reduce" operations at every layer would make the model run significantly slower than if you just used one box.
=====
****

****
**Quiz 4: Identifying the Parallelism**

**Scenario:** You observe your GPU monitoring tool (e.g., `nvidia-smi`). You notice that **GPU 0** is at 100% utilization for 2 seconds, then it drops to 0% while **GPU 1** spikes to 100% for 2 seconds. This pattern repeats.

* **Question:** What type of parallelism is likely being used?
* **A.** Tensor Parallelism
* **B.** Pipeline Parallelism
* **C.** Data Parallelism

.Reveal Solution
[%collapsible]
=====
* **Answer:** **B. Pipeline Parallelism.** This "staggered" utilization is the classic sign of a pipeline. GPU 0 is processing the first half of the model, then waiting for GPU 1 to finish the second half before it can start the next batch. (Note: This suggests the "micro-batching" isn't tuned perfectly!)
=====

= How MIG Partitioning Works in Kubernetes

_Estimated reading time: *{time_estimate} minutes*._

MIG partitioning in OpenShift involves coordination between the GPU Operator components you deployed in Chapter 1. Understanding this workflow helps you troubleshoot configuration issues and plan maintenance windows effectively.

=== Component Roles in MIG Lifecycle

From Chapter 1, recall these GPU Operator components and their specific roles in MIG enablement:

* **MIG Manager**: Watches node labels, validates MIG profiles, applies MIG configuration via `nvidia-smi mig` commands
* **GPU Feature Discovery (GFD)**: Scans GPUs using NVML (NVIDIA Management Library), detects MIG instances, applies detailed node labels
* **Device Plugin**: Reads GFD-generated labels, advertises MIG instances as allocatable Kubernetes resources
* **NVIDIA Driver**: Provides kernel-level MIG mode enablement, instance creation, and hardware partitioning

These components coordinate through the operator reconciliation pattern you learned in Chapter 1, Section 2.

=== MIG Enablement Workflow

When you label a node with `nvidia.com/mig.config=all-2g.10gb`, this workflow executes:

----
1. MIG Manager DaemonSet (controller) detects label change
         ↓
2. MIG Manager validates profile against GPU capability
   - Checks GPU model supports requested profile
   - Verifies profile name exists in ConfigMap
         ↓
3. MIG Manager enables MIG mode on GPU (if not already enabled)
   - Executes: nvidia-smi -i 0 -mig 1
   - Requires GPU reset (brief workload interruption)
   - Takes ~10 seconds per GPU
         ↓
4. MIG Manager creates MIG instances per profile
   - Partitions SMs and memory per profile spec
   - Creates device handles for each instance
   - Takes ~30 seconds for profile application
         ↓
5. GPU Feature Discovery rescans GPU capabilities
   - Queries MIG instances via nvidia-smi
   - Detects MIG geometry and resource allocations
   - Applies labels: nvidia.com/mig-2g.10gb.count=3
         ↓
6. Device Plugin rediscovers allocatable resources
   - Reads GFD-generated MIG labels
   - Advertises: nvidia.com/mig-2g.10gb: 3
   - Updates Node.status.allocatable
         ↓
7. Kubernetes Scheduler sees new resources available
   - Pods requesting nvidia.com/mig-2g.10gb can schedule
   - Workloads begin utilizing MIG instances
----

**Total Time**: 10-20 minutes from label application to resource availability (OpenShift documentation guidance)

[WARNING]
====
**MIG Reconfiguration Requires Workload Drain**

Changing MIG profiles on a node with running GPU workloads requires these steps:

1. **Cordon node**: `oc adm cordon worker-gpu-0` (prevent new pod scheduling)
2. **Drain node**: `oc adm drain worker-gpu-0 --ignore-daemonsets --delete-emptydir-data` (evict running pods, 5-10 min for workload migration)
3. **Apply new label**: `oc label node worker-gpu-0 nvidia.com/mig.config=all-3g.20gb --overwrite`
4. **Wait for reconfiguration**: Monitor MIG Manager logs (10-20 min)
5. **Uncordon node**: `oc adm uncordon worker-gpu-0` (allow new workloads)

**Plan MIG configuration changes during maintenance windows.** Unlike time-slicing (which only restarts device plugin pods), MIG mode changes reset the GPU hardware, terminating all running workloads on that GPU.
====

=== Integration with ClusterPolicy (Chapter 1)

The ClusterPolicy Custom Resource you configured in Chapter 1 enables MIG Manager capability:

[source,yaml]
----
apiVersion: nvidia.com/v1
kind: ClusterPolicy
metadata:
  name: gpu-cluster-policy
spec:
  migManager:
    enabled: true  # <1>
  mig:
    strategy: mixed  # <2>
----
<1> Deploys MIG Manager DaemonSet—actual profile configuration happens via node labels
<2> Sets advertisement strategy (single or mixed) for cluster-wide MIG resource naming

This **decouples MIG capability from MIG configuration**:

* **Capability** (ClusterPolicy): Enables MIG Manager cluster-wide, sets strategy
* **Configuration** (node labels): Applies specific profiles per node

This design allows you to:

* Enable MIG capability cluster-wide once via ClusterPolicy
* Apply MIG profiles per-node dynamically via labels
* Change profiles without modifying ClusterPolicy or restarting operators

[TIP]
====
**Enable MIG Manager Even If Not Immediately Using MIG**

Set `migManager.enabled: true` in ClusterPolicy during initial GPU Operator deployment, even if you don't plan to use MIG immediately. This deploys the MIG Manager DaemonSet in standby mode, allowing you to enable MIG on individual nodes later via labels without cluster-wide operator reconfiguration.

This follows the production best practice from Chapter 1, Section 3: prepare infrastructure capabilities early, activate them as needed.
====

=== MIG Configuration Auto-Generation

Starting with GPU Operator v26.3.0, MIG Manager **automatically generates** MIG configurations for your hardware:

* Queries GPU capabilities via NVML (NVIDIA Management Library)
* Discovers supported MIG profiles for each GPU model
* Creates a ConfigMap named `<node-name>-mig-config` for each MIG-capable node
* Includes built-in profiles: `all-disabled`, `all-balanced`, and per-profile configs (e.g., `all-1g.10gb`, `all-2g.10gb`)

**Legacy behavior (pre-v26.3.0)**: Used static ConfigMap `default-mig-parted-config` with predefined profiles.

This auto-generation means you typically don't need custom ConfigMaps unless you require specialized heterogeneous combinations not covered by `all-balanced`.

== Enabling MIG via Node Labels

MIG configuration uses a label-driven workflow. The MIG Manager watches for specific labels on GPU nodes and applies the corresponding profile from its configuration.

=== Label-Driven Configuration Pattern

The GPU Operator uses this label schema:

[source,bash]
----
nvidia.com/mig.config=<profile-name>
----

**Built-in Profile Names** (no additional ConfigMap configuration needed):

* `all-disabled` - MIG mode off, GPU operates as single full device (default)
* `all-1g.5gb` - Partition into 7x `1g.5gb` instances (A100-40GB)
* `all-1g.10gb` - Partition into 7x `1g.10gb` instances (A100-80GB)
* `all-2g.10gb` - Partition into 3x `2g.10gb` instances (A100-40GB)
* `all-3g.20gb` - Partition into 2x `3g.20gb` instances (A100-40GB)
* `all-balanced` - Balanced heterogeneous mix (2x `1g.5gb` + 1x `2g.10gb` + 1x `3g.20gb` on A100-40GB)

**Example: Applying MIG Profile to Nodes**

[source,bash]
----
# Set MIG strategy in ClusterPolicy (one-time cluster-wide configuration)
$ oc patch clusterpolicy/gpu-cluster-policy --type='json' \
  -p='[{"op": "replace", "path": "/spec/mig/strategy", "value": "mixed"}]'

# Label GPU nodes for MIG configuration
$ oc label node worker-gpu-0.example.com nvidia.com/mig.config=all-2g.10gb --overwrite
node/worker-gpu-0.example.com labeled

$ oc label node worker-gpu-1.example.com nvidia.com/mig.config=all-2g.10gb --overwrite
node/worker-gpu-1.example.com labeled

# Monitor MIG Manager applying configuration
$ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager -f
time="2024-04-10T14:23:15Z" level=info msg="Applying MIG config" node=worker-gpu-0 profile=all-2g.10gb
time="2024-04-10T14:23:45Z" level=info msg="MIG configuration complete" instances=3

# Verify MIG instances are advertised (wait 10-20 minutes)
$ oc describe node worker-gpu-0.example.com | grep -A 10 "Allocatable:"
Allocatable:
  cpu:                          31500m
  memory:                       252455456Ki
  nvidia.com/mig-2g.10gb:       3  # <1>
  pods:                         250
----
<1> Three MIG instances now available for workload scheduling

=== Custom MIG Profiles via ConfigMap

For mixed profiles or custom combinations beyond `all-balanced`, create a `mig-parted` configuration:

[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: custom-mig-parted-config
  namespace: nvidia-gpu-operator
data:
  config.yaml: |
    version: v1
    mig-configs:
      mixed-inference:  # <1>
        - devices: all
          mig-enabled: true
          mig-devices:
            "3g.20gb": 1  # <2>
            "2g.10gb": 2  # <3>
----
<1> Custom profile name for node labeling
<2> One large instance for LLaMA-7B inference
<3> Two medium instances for BERT-class models

**Apply Custom Profile to ClusterPolicy:**

[source,bash]
----
$ oc patch clusterpolicy gpu-cluster-policy --type=merge -p '
  {
    "spec": {
      "migManager": {
        "config": {
          "name": "custom-mig-parted-config"  # <1>
        }
      }
    }
  }'
----
<1> Reference your custom ConfigMap

**Use Custom Profile on Nodes:**

[source,bash]
----
$ oc label node worker-gpu-2.example.com \
  nvidia.com/mig.config=mixed-inference  # <1>
----
<1> Must match profile name from ConfigMap `mig-configs` section

│  │

----

[IMPORTANT]
====
**Multi-Strategy Deployments in Large Platforms**

Production platforms often use **multiple strategies simultaneously** based on workload characteristics:

* **Training node pool**: Full GPU allocation
  - Workloads need maximum GPU memory and compute
  - Checkpointing requires exclusive access
  - Label: `node-role.kubernetes.io/gpu-training=true`

* **Inference node pool (A100-equipped)**: MIG partitioning
  - Multi-tenant model serving with SLA requirements
  - Hardware isolation for predictable latency
  - Label: `node-role.kubernetes.io/gpu-inference-mig=true`

* **Inference node pool (T4/V100-equipped)**: Time-slicing
  - No MIG support on these older architectures
  - Acceptable for non-SLA workloads
  - Label: `node-role.kubernetes.io/gpu-inference-timeslice=true`

Use node selectors and taints/tolerations to route workloads to appropriate node pools:

```yaml
nodeSelector:
  node-role.kubernetes.io/gpu-inference-mig: "true"
```

This architecture provides workload placement flexibility while optimizing each pool for its use case.

= MIG Advertisement Strategies

_Estimated reading time: *{time_estimate} minutes*._

The NVIDIA GPU Operator supports two strategies for exposing MIG instances to the Kubernetes scheduler. This strategy choice fundamentally affects how workloads request resources and how you manage node configurations.

=== Single Strategy (Homogeneous)

All MIG instances on a node must use the same profile. Instances are advertised as `nvidia.com/gpu` resources, providing **backward compatibility** with existing workloads.

**Characteristics**:

* **Resource advertisement**: `nvidia.com/gpu: 7` (for 7x `1g.5gb` instances)
* **Node configuration**: All GPUs on node must use identical profile
* **Workload requests**: Standard `nvidia.com/gpu: 1` (no profile-specific request)
* **Scheduling**: Kubernetes scheduler treats MIG instances identically to full GPUs

**Example**: A100-40GB cluster with `all-3g.20gb` profile:

* Each physical GPU creates 2 MIG instances
* Node with 4 A100 GPUs advertises `nvidia.com/gpu: 8`
* Existing inference services request `nvidia.com/gpu: 1` without modification

**Advantages**:

* Easy to understand and manage
* Backward compatible with existing pod specifications
* Predictable resource allocation (all instances identical)
* Simple capacity planning (uniform node pools)
* Best for uniform workload patterns

=== Mixed Strategy (Heterogeneous)

Different MIG profiles can coexist on the same physical GPU. Instances are advertised with **profile-specific resource names**.

**Characteristics**:

* **Resource advertisement**: `nvidia.com/mig-1g.5gb: 2`, `nvidia.com/mig-2g.10gb: 1`, `nvidia.com/mig-3g.20gb: 1`
* **Node configuration**: Each GPU can have different profile mix
* **Workload requests**: Must specify profile: `nvidia.com/mig-3g.20gb: 1`
* **Scheduling**: Kubernetes scheduler matches request to available profile

**Example**: A100-40GB node with `all-balanced` profile:

* Creates 2x `1g.5gb`, 1x `2g.10gb`, 1x `3g.20gb` per GPU
* Small models request `nvidia.com/mig-1g.5gb: 1`
* Large models request `nvidia.com/mig-3g.20gb: 1`
* Scheduler places workloads on nodes with matching profiles

**Advantages**:

* Accommodates diverse workload requirements on single node
* Optimizes resource utilization for heterogeneous platforms
* Maximum flexibility for workload placement
* Supports true multi-tenant deployments with varied resource needs

**Challenges**:

* More complex to manage and schedule
* Workload specifications must be profile-aware
* Per-GPU configuration tracking required
* Risk of scheduling failures if pod requests unavailable profile

=== Production Strategy Selection Matrix

[cols="2,2,2,2"]
|===
|Criterion |Single Strategy |Mixed Strategy |Production Guidance

|**Operational Complexity**
|✅ Low - uniform config
|❌ High - per-GPU tracking
|Single for teams <10 engineers

|**Resource Efficiency**
|⚠️ May waste resources if workloads vary
|✅ Optimal packing for diverse workloads
|Mixed for heterogeneous platforms

|**Scheduling Simplicity**
|✅ Simple - all instances identical
|⚠️ Complex - must match profiles
|Single for <50 total workloads

|**Backward Compatibility**
|✅ Existing pods work unchanged
|❌ Pods must specify MIG profiles
|Single for migrating existing apps

|**Reconfiguration Impact**
|✅ Change all nodes uniformly
|❌ Per-node config drift risk
|Single for stability-focused platforms

|**Multi-Tenancy Support**
|⚠️ Limited - all tenants same size
|✅ Excellent - right-size per tenant
|Mixed for true SaaS platforms

|**Cluster Size Sweet Spot**
|✅ Large clusters (node pools)
|✅ Small clusters (diverse nodes)
|Depends on node count and diversity
|===

[WARNING]
====
**Mixed Strategy Requires Profile-Aware Scheduling**

With mixed strategy, pods **must** request specific MIG profiles in resource limits:

```yaml
resources:
  limits:
    nvidia.com/mig-3g.20gb: 1  # Correct for mixed
```

Requesting generic `nvidia.com/gpu: 1` will **fail to schedule** on MIG-enabled nodes using mixed strategy. You must either:

* Update all pod specifications to request specific profiles
* Implement admission webhooks to inject correct resource requests
* Use Kueue ResourceFlavors (covered in later chapters) for automated profile selection
====

[TIP]
====
**Recommended Adoption Path: Start Single, Migrate to Mixed**

For teams new to MIG:

1. **Months 1-3**: Deploy Single strategy with one profile (e.g., `all-2g.10gb`)
   - Learn MIG operational patterns
   - Measure actual workload resource usage
   - Build confidence with uniform configuration

2. **Months 4-6**: Evaluate resource waste and workload diversity
   - Analyze DCGM metrics for over/under-provisioning
   - Identify distinct workload tiers (small, medium, large)
   - Calculate potential efficiency gains from mixed profiles

3. **Months 7+**: Migrate to Mixed strategy if justified
   - Migrate when >20% resource waste from uniform sizing
   - Requires dedicated platform team for GPU operations
   - Implement profile-aware admission controls first

This phased approach balances operational learning with efficiency optimization.
====

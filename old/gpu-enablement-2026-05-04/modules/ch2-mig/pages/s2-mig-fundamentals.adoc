== What is MIG and Why Use It

High-performance GPUs like the NVIDIA A100 cost $10,000-$30,000 per unit, but many AI inference workloads use only 20-40% of available GPU capacity. Multi-Instance GPU (MIG) partitions a single physical GPU into multiple isolated instances, allowing you to run 3-7 workloads concurrently with hardware-guaranteed performance isolation.

=== The GPU Utilization Problem

Consider a platform serving three teams with a single NVIDIA A100-40GB GPU ($15,000 investment):

* **Team A (ML Platform)**: LLaMA-7B inference requiring 14GB VRAM and 50% compute
* **Team B (Data Science)**: BERT embedding requiring 4GB VRAM and 10% compute
* **Team C (Computer Vision)**: ResNet-50 classification requiring 3GB VRAM and 15% compute

**Total requirements**: 21GB VRAM (53% of available 40GB) and 75% compute capacity

**Without MIG**: Only one team uses the GPU at a time, forcing others to wait in queue. Average utilization: 33%

**With MIG**: All three teams run concurrently on isolated instances. Average utilization: 78%

=== How MIG Works

MIG partitions the physical GPU into isolated instances at the hardware level:

* **Dedicated GPU compute resources** (streaming multiprocessors)
* **Dedicated memory** (VRAM partition)
* **Dedicated memory bandwidth** (isolated memory controllers)
* **Complete fault isolation** (errors in one instance don't affect others)

From a containerized workload's perspective, each MIG instance appears as a separate GPU device.

**Example: 3g.20gb Profile on A100-40GB**

* Allocates 3/7ths of GPU compute (approximately 46 streaming multiprocessors)
* Allocates 20GB dedicated VRAM partition
* Creates hardware barriers preventing cross-instance interference

[NOTE]
====
**MIG vs Time-Slicing:**

* **Time-slicing**: Workloads share the same physical resources temporally (software multiplexing). P99 latency variance: 15-40%
* **MIG**: Physical partitioning with dedicated resources (hardware isolation). P99 latency variance: <1%

This architectural difference is why MIG enables SLA-backed inference while time-slicing cannot.
====

== MIG Profiles and Planning

A physical GPU can be partitioned into different MIG instance sizes called "profiles". Understanding profile characteristics is essential for matching resources to workload requirements.

=== Profile Naming Convention

The profile format `Xg.YYgb` encodes resource allocations:

* **X**: Compute slice size (in 1/7ths of total GPU compute)
* **YY**: Memory allocation in gigabytes

Example: `3g.20gb` means 3/7ths of compute resources and 20GB VRAM

[IMPORTANT]
====
When MIG mode is enabled, NVIDIA reserves 1/8th of the GPU for MIG management. The remaining 7/8ths are available for partitioning. This is why maximum instances sum to 7g, not 8g.
====

=== Profile Selection Decision Framework

[TIP]
====
**Profile Selection Process:**

1. **Measure actual VRAM usage** during test runs (`nvidia-smi` inside pods)
2. **Add 20-25% overhead buffer** for framework allocators (PyTorch, TensorFlow)
3. **Round UP to nearest profile** (Example: 12GB requirement → choose `3g.20gb`, not `2g.10gb`)
4. **Validate profile combination** fits within GPU capacity (7 compute slices total)
====

== MIG Advertisement Strategies

The NVIDIA GPU Operator supports two strategies for exposing MIG instances to Kubernetes. This choice affects how workloads request resources and how you manage node configurations.

=== Single Strategy (Homogeneous)

All MIG instances on a node use the same profile. Instances are advertised as `nvidia.com/gpu` resources.

**Characteristics:**

* **Resource advertisement**: `nvidia.com/gpu: 7` (for 7x `1g.5gb` instances)
* **Node configuration**: All GPUs on node must use identical profile
* **Workload requests**: Standard `nvidia.com/gpu: 1`
* **Backward compatible**: Existing pod specs work unchanged

**Advantages:**

* Easy to understand and manage
* Predictable resource allocation (all instances identical)
* Simple capacity planning
* Best for uniform workload patterns

=== Mixed Strategy (Heterogeneous)

Different MIG profiles coexist on the same physical GPU. Instances are advertised with profile-specific resource names.

**Characteristics:**

* **Resource advertisement**: `nvidia.com/mig-1g.5gb: 2`, `nvidia.com/mig-2g.10gb: 1`, `nvidia.com/mig-3g.20gb: 1`
* **Node configuration**: Each GPU can have different profile mix
* **Workload requests**: Must specify profile: `nvidia.com/mig-3g.20gb: 1`
* **Scheduling**: Kubernetes matches request to available profile

**Advantages:**

* Accommodates diverse workload requirements on single node
* Optimizes resource utilization for heterogeneous platforms
* True multi-tenant deployments with varied resource needs

**Challenges:**

* More complex to manage
* Workload specifications must be profile-aware
* Risk of scheduling failures if pod requests unavailable profile

=== Strategy Selection Decision Matrix

[cols="2,2,2",options="header"]
|===
|Criterion |Single Strategy |Mixed Strategy

|**Operational Complexity**
|✅ Low - uniform config
|❌ High - per-GPU tracking

|**Resource Efficiency**
|⚠️ May waste resources
|✅ Optimal packing

|**Backward Compatibility**
|✅ Existing pods work
|❌ Pods must specify profiles

|**Multi-Tenancy Support**
|⚠️ Limited - same size
|✅ Right-size per tenant

|**Best For**
|Clusters <50 workloads
|Heterogeneous platforms
|===

[WARNING]
====
**Mixed Strategy Requires Profile-Aware Scheduling**

With mixed strategy, pods **must** request specific MIG profiles:

```yaml
resources:
  limits:
    nvidia.com/mig-3g.20gb: 1  # Correct for mixed
```

Requesting generic `nvidia.com/gpu: 1` will **fail to schedule** on MIG-enabled nodes using mixed strategy.
====

== Enabling MIG in Kubernetes

MIG configuration in OpenShift uses a label-driven workflow. The MIG Manager watches for specific labels on GPU nodes and applies the corresponding profile.

=== MIG Enablement Workflow

When you label a node with `nvidia.com/mig.config=all-2g.10gb`, this workflow executes:

```
1. MIG Manager detects label change
         ↓
2. MIG Manager validates profile against GPU capability
         ↓
3. MIG Manager enables MIG mode on GPU (~10 seconds)
   Executes: nvidia-smi -i 0 -mig 1
         ↓
4. MIG Manager creates MIG instances (~30 seconds)
   Partitions SMs and memory per profile spec
         ↓
5. GPU Feature Discovery rescans GPU capabilities
   Applies labels: nvidia.com/mig-2g.10gb.count=3
         ↓
6. Device Plugin rediscovers allocatable resources
   Advertises: nvidia.com/mig-2g.10gb: 3
         ↓
7. Kubernetes Scheduler sees new resources available
   Pods can now schedule
```

**Total Time**: 10-20 minutes from label application to resource availability

[WARNING]
====
**MIG Reconfiguration Requires Workload Drain**

Changing MIG profiles on a node with running GPU workloads requires:

1. **Cordon node**: Prevent new scheduling
2. **Drain node**: Evict running pods (5-10 min)
3. **Apply new label**: MIG Manager reconfigures GPU
4. **Wait**: 10-20 min for reconfiguration
5. **Uncordon node**: Allow new workloads

**Plan MIG configuration changes during maintenance windows.** MIG mode changes reset the GPU hardware, terminating all running GPU workloads.
====

=== Label-Driven Configuration

Apply MIG profiles using node labels:

[source,bash]
----
# Set MIG strategy in ClusterPolicy (one-time cluster-wide)
$ oc patch clusterpolicy/gpu-cluster-policy --type='json' \
  -p='[{"op": "replace", "path": "/spec/mig/strategy", "value": "mixed"}]'

# Label GPU node for MIG configuration
$ oc label node worker-gpu-0.example.com \
  nvidia.com/mig.config=all-2g.10gb --overwrite

# Monitor MIG Manager applying configuration
$ oc logs -n nvidia-gpu-operator -l app=nvidia-mig-manager -f

# Verify MIG instances are advertised (wait 10-20 minutes)
$ oc describe node worker-gpu-0.example.com | grep -A 10 "Allocatable:"
Allocatable:
  nvidia.com/mig-2g.10gb:  3
----

**Built-in Profile Names** (no additional ConfigMap needed):

* `all-disabled` - MIG mode off
* `all-1g.5gb` - 7x `1g.5gb` instances (A100-40GB)
* `all-1g.10gb` - 7x `1g.10gb` instances (A100-80GB)
* `all-2g.10gb` - 3x `2g.10gb` instances (A100-40GB)
* `all-3g.20gb` - 2x `3g.20gb` instances (A100-40GB)
* `all-balanced` - Mixed instances (2x `1g.5gb` + 1x `2g.10gb` + 1x `3g.20gb`)

=== Custom MIG Profiles via ConfigMap

For mixed profiles beyond `all-balanced`, create a custom `mig-parted` configuration:

[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: custom-mig-parted-config
  namespace: nvidia-gpu-operator
data:
  config.yaml: |
    version: v1
    mig-configs:
      mixed-inference:  # Custom profile name
        - devices: all
          mig-enabled: true
          mig-devices:
            "3g.20gb": 1  # One large instance
            "2g.10gb": 2  # Two medium instances
----

Apply to ClusterPolicy:

[source,bash]
----
$ oc patch clusterpolicy gpu-cluster-policy --type=merge -p '
  {
    "spec": {
      "migManager": {
        "config": {
          "name": "custom-mig-parted-config"
        }
      }
    }
  }'

# Use custom profile on nodes
$ oc label node worker-gpu-2.example.com \
  nvidia.com/mig.config=mixed-inference --overwrite
----

[TIP]
====
**Enable MIG Manager Early**

Set `migManager.enabled: true` in ClusterPolicy during initial GPU Operator deployment, even if not using MIG immediately. This deploys the MIG Manager DaemonSet in standby mode, allowing you to enable MIG on individual nodes later via labels without cluster-wide operator reconfiguration.
====

== When to Use MIG: Decision Framework

Use this decision flow to determine if MIG is right for your platform:

```
Do you have SLA-backed latency requirements?
│
├─ YES → Do you have A30/A100/H100 GPUs?
│  │
│  ├─ YES → Use MIG
│  │        ✅ Hardware isolation + high density
│  │        ✅ Predictable performance + cost efficiency
│  │
│  └─ NO → Use Full GPU
│           ⚠️ Lower density but guaranteed performance
│
└─ NO → Do you have MIG-capable GPUs?
   │
   ├─ YES → Use MIG
   │        ✅ Better utilization than time-slicing
   │        ✅ Fault isolation protects workloads
   │
   └─ NO → Use Time-Slicing
            ✅ Dev/test environments
            ✅ Cost optimization priority
```

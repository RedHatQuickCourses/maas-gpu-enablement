=== Scenario 3: Performance Troubleshooting

**Alert:** Production model deployment showing degraded performance despite "successful" deployment.

**Deployed Configuration:**

* Model: Llama-405B-INT8 (~405 GB)
* Hardware: 2 nodes, each with 8× H100-80GB
* Configuration: `--tensor-parallel-size=16` (TP across all 16 GPUs)
* Network: Nodes connected via 200 Gb InfiniBand

**Observed Symptoms:**

* Expected P99 latency: 65ms (based on benchmark for 8× H100 single-node)
* **Actual P99 latency:** 950ms (14.6× worse than expected)
* DCGM metrics: `DCGM_FI_DEV_GPU_UTIL` = 18% (GPUs mostly idle)
* DCGM metrics: `DCGM_FI_DEV_POWER_USAGE` = 310W per GPU (high power despite low utilization)

**Your Diagnostic Process:**

**[ ] Step 1: Verify network topology**

* Run: `oc debug node/<gpu-node> -- chroot /host nvidia-smi topo -m`
* **Expected for TP=16:** All 16 GPUs should show `NV18` (NVLink) between each other
* **Actual observation:** GPUs within each node show `NV18`, but across nodes show `SYS` (system connection via InfiniBand)

**Root Cause Identified:** TP is spanning nodes connected by InfiniBand (not NVLink).

**[ ] Step 2: Calculate synchronization overhead**

* Tensor Parallelism performs all-reduce after **every transformer layer** (Llama-405B: ~80 layers)
* InfiniBand latency: ~2-5 microseconds per sync operation
* Total sync overhead: 80 layers × 3 μs (average) = _____ μs per forward pass
* Actual compute time: ~10ms per forward pass
* **Percentage of time waiting for network:** _____ %

**[ ] Step 3: Explain low GPU utilization**

* GPUs idle 82% of time (100% - 18% utilization)
* **Reason:** Tensor Parallelism requires synchronization before proceeding to next layer
** All 16 GPUs must wait for slowest sync operation to complete
** Network latency dominates compute time
** GPUs sit idle waiting for cross-node synchronization

**[ ] Step 4: Explain high power consumption despite low utilization**

* Power usage: 310W per GPU despite 18% compute utilization
* **Reason:** GPUs consume power even when idle waiting for network
** CUDA kernels remain loaded in memory
** GPU cores ready to compute (consuming power) but blocked on network I/O
** Power consumption closer to idle power (~100W) + waiting overhead (~200W)

**[ ] Step 5: Recommend corrected configuration**

**Corrected Deployment Configuration:**

```yaml
# INCORRECT (current):
vllm_args:
  --tensor-parallel-size=16  # TP across InfiniBand - WRONG

# CORRECT (hybrid TP+PP):
vllm_args:
  --tensor-parallel-size=8      # TP within each node (NVLink)
  --pipeline-parallel-size=2    # PP across nodes (InfiniBand)
  --model=/models/llama-405b-int8
```

**Expected Performance After Fix:**

* **Latency:** ~95ms P99 (vs 950ms current, vs 65ms single-node baseline)
** Pipeline Parallelism adds ~30ms overhead ("bubble" between stages)
** Still 10× better than misconfigured TP=16
* **Throughput:** ~7,000 tokens/sec (vs ~400 tokens/sec current)
** 16 GPUs process more concurrent requests via pipeline
** Network latency no longer dominates (1× PP communication per pipeline stage, not 80× TP per forward pass)
* **GPU Utilization:** 80-85% (vs 18% current)
** GPUs spend <10% time on network synchronization (PP less frequent than TP)

**[ ] Step 6: Validation post-fix**

After deploying corrected configuration, confirm metrics improve:

[source,promql]
----

DCGM_FI_DEV_GPU_UTIL{deployment="llama-405b-corrected"}
# Target: >80% (vs 18% with misconfigured TP)

# Verify latency improvement
histogram_quantile(0.99,
  rate(vllm_time_to_first_token_seconds_bucket{deployment="llama-405b-corrected"}[5m])
)
# Target: <100ms (vs 950ms with misconfigured TP)

# Verify throughput increase
rate(vllm_request_success_total{deployment="llama-405b-corrected"}[5m])
# Target: >6,000 tokens/sec (vs ~400 tokens/sec with misconfigured TP)
----

== Integration with Course Concepts

Model selection decisions integrate knowledge across all prior chapters:

=== Chapter 1: GPU Operator and Infrastructure

**Driver Version Requirements**

* **FP8 quantization** requires CUDA 12.0+ support (driver ≥525.60.13)
* Check current driver: `nvidia-smi --query-gpu=driver_version`
* Update via ClusterPolicy if needed: `spec.driver.version: "535.129.03"`
* **Impact:** Hopper GPU with old driver cannot leverage FP8 Tensor Cores (loses 2× throughput advantage)

**ClusterPolicy Configuration**

* Model selection influences ClusterPolicy settings:
** MIG mode enabled/disabled based on model size (large models require `enabled: false` for full GPU access)
** Driver version selection based on quantization requirements
** DCGM enablement mandatory for model selection validation (metrics-driven decisions)

**Reference:** xref:../ch1-gpu-operator/s2-operators-overview.adoc[Chapter 1: GPU Operator and ClusterPolicy Configuration]

=== Chapter 2: MIG and Multi-Tenant Resource Allocation

**Large Model Constraints**

* Models >70B parameters **cannot use MIG instances** for multi-GPU parallelism:
** Largest MIG profile: `3g.40gb` = 40 GB VRAM per instance
** Llama-405B requires 405 GB (would need 11 MIG instances minimum)
** **Technical barrier:** Tensor Parallelism requires NVLink between GPUs; MIG instances have separate CUDA contexts (no inter-instance NVLink)

**When MIG Enables Model Serving**

* **Small models (<13B parameters)** benefit from MIG:
** Deploy 7× Llama-7B instances in separate `1g.10gb` profiles on single H100
** Multi-tenant isolation (different teams/customers per MIG instance)
** Cost optimization (maximize GPU utilization via multiple concurrent models)

**MIG Profile Selection Decision Matrix**

[cols="2,2,2", options="header"]
|===
| Model Size | MIG Viable? | Recommended Approach

| 7B-13B parameters
| ✅ Yes
| MIG profiles: `1g.10gb` or `2g.20gb` for multi-tenant serving

| 70B parameters
| ⚠️ Marginal
| Full GPU or MIG `3g.40gb` (borderline fit with INT8)

| 135B+ parameters
| ❌ No
| Full GPU allocation with TP across multiple GPUs
|===

**Reference:** xref:../ch2-mig/s1-mig-overview.adoc#selecting-mig-profiles[Chapter 2: MIG Profile Selection and Constraints]

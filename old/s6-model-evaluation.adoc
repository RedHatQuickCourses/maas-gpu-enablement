== Architecture Selection: Dense vs Mixture-of-Experts

=== **Dense Models**

In a dense model, every parameter activates for every token generated. A 70B parameter model performs 70 billion calculations per token, using all weights simultaneously.

**Characteristics:**

* **Predictable performance:** Compute and memory requirements scale linearly with parameter count
* **Robust sequential reasoning:** All parameters available for every decision, enabling deep multi-step synthesis
* **Easier optimization:** Fits standard inference engines (vLLM, TGI) with minimal tuning
* **Lower total VRAM footprint:** Only store parameters actually used

**Trade-offs:**

* **Higher compute cost per token:** Processing 70B parameters for simple queries wastes resources
* **Lower throughput:** Memory bandwidth becomes bottleneck before compute saturation
* **Scaling requires larger models:** Improving capability requires training bigger models (linear cost increase)

=== **Sparse (Mixture-of-Experts) Models**

An MoE model contains multiple "expert" sub-networks. For each token, a router selects which experts to activate (typically 2-8 experts from a pool of 8-64 total experts).

**Example:** Mixtral 8×7B has 8 expert networks of 7B parameters each (56B total), plus shared layers (≈14B), totaling 70B parameters. Per token, the router activates 2 experts (14B active parameters) plus shared layers.

**Characteristics:**

* **Massive total parameter count, small active count:** DeepSeek-V3: 671B total, 37B active per token
* **Specialization via routing:** Coding queries route to "coding expert," creative tasks to "language expert"
* **High throughput per GPU:** Less compute per token enables faster generation
* **Lower compute cost per token:** Only 5-20% of parameters activate per inference

**Trade-offs:**

* **Large VRAM requirement:** Must load ALL parameters into memory for router accessibility
* **Routing dependency:** Incorrect expert selection degrades reasoning (router becomes single point of failure)
* **Complex memory management:** Requires tensor parallelism across multiple GPUs for large models
* **Infrastructure complexity:** Multi-node clusters and specialized partitioning often mandatory

== Model Sizing: Calculating GPU Memory Requirements


=== VRAM Components and Calculation Formula

Total VRAM requirement consists of four components:

*VRAM_total = weights + KV_cache + overhead + margin*


**Component 1: Model Weights** (VRAM~weights~)

Calculate based on parameter count and precision:

[cols="2,1,2", options="header"]
|===
| Precision | Bytes per Parameter | Example: 70B Model

| **FP16 / BF16**
| 2 bytes
| 70B × 2 = **140 GB**

| **INT8 / FP8**
| 1 byte
| 70B × 1 = **70 GB**

| **INT4**
| 0.5 bytes
| 70B × 0.5 = **35 GB**
|===

**Component 2: KV Cache** (VRAM~KV_cache~)

The "memory" of conversation context. Stores attention keys and values for each token processed.

**Estimation rule for 70B-class models:**

* **1 GB per 8,000 tokens** of context window at FP16 precision
* **0.5 GB per 8,000 tokens** at INT8/FP8 precision

**Example:** Llama-70B with 128K context window at FP16:

KV_cache = (128,000 \div 8,000) * 1GB = 16GB


**Scaling for different model sizes:**

KV cache scales with number of layers and hidden dimensions, not just parameters. For rough estimates:

* **7B-13B models:** KV cache ≈ 30% of 70B model KV cache
* **135B-175B models:** KV cache ≈ 2× to 2.5× of 70B model KV cache
* **400B+ models:** KV cache ≈ 5× to 6× of 70B model KV cache

**Component 3: Framework Overhead** (VRAM~overhead~)

Inference frameworks (vLLM, TGI, TensorRT-LLM) require additional memory:

* **vLLM typical overhead:** 10-15% of (weights + KV cache)
* **TGI typical overhead:** 8-12% of (weights + KV cache)
* **Buffer allocations, CUDA kernels, scheduler state:** Varies by batch size and concurrency

**Component 4: Safety Margin** (VRAM~margin~)

Production deployments should never run at 100% VRAM utilization.

**Recommended margin:** 10-15% of total VRAM capacity

**Reason:** Prevents CUDA OOM errors during traffic spikes, allows for KV cache growth with longer-than-expected queries, provides buffer for framework version updates that may increase overhead.

=== HOW: Step-by-Step VRAM Sizing Procedure

Follow this procedure for every model deployment decision:

**Step 1: Calculate base weight requirement**

Identify model parameter count and target quantization precision:

[source,text]

[source,text]
----
Example: 128K context window required for RAG application

----

**Important:** Use production context window length, not development/testing length. Testing with 2K context then deploying for 128K context is a common failure mode.

**Step 3: Add framework overhead**

Apply percentage to base + KV cache:

[source,text]
----
Example: vLLM deployment (assume 12% overhead)
Base + KV = 135 GB + 32 GB = 167 GB
VRAM_overhead = 167 GB × 0.12 = 20 GB
----

**Step 4: Add safety margin**

Reserve capacity for unexpected scenarios:

[source,text]
----
Example: 10% safety margin
Subtotal = 135 + 32 + 20 = 187 GB
VRAM_margin = 187 GB × 0.10 = 19 GB

Total VRAM requirement = 187 + 19 = 206 GB
----

[WARNING]
====
**Common Sizing Mistake: Ignoring KV Cache for Long-Context Models**

For context windows beyond 32K tokens, KV cache can **exceed model weight size**.

**Example: Llama-70B-INT8 with 256K context**

* **Weights:** 70 GB
* **KV cache:** (256K ÷ 8K) × 1 GB = 32 GB
* **Total:** 102 GB (KV cache is 46% of total VRAM)

**For 512K context:** KV cache grows to 64 GB, nearly matching weight size.

**Mitigation strategies:**

1. **Quantize KV cache:** vLLM supports `--kv-cache-dtype fp8` reducing KV cache by 50%
2. **Page attention:** vLLM's PagedAttention manages KV cache more efficiently than naive implementations
3. **Shorter context windows:** Question whether 512K context is truly required, or if 128K suffices

Always size for **production context window length**, not development testing length (2-4K tokens).

== Quantization Strategies: Matching Precision to Hardware

=== WHY: Quantization as Infrastructure Cost Optimization

Quantization reduces the numerical precision of model weights and activations, enabling powerful models to fit on cost-effective hardware. This is not optional optimization—it's mandatory for production-scale deployments.

**Business case:**

Llama-405B model in FP16 precision requires 810 GB VRAM (405B × 2 bytes). Deployment options:

* **11× H100-80GB** at $32K each = **$352K hardware cost**
* **OR: Quantize to INT8** (405B × 1 byte = 405 GB) → **5× H100-80GB = $160K hardware cost**

Quantization saves $192K in initial hardware procurement while typically introducing <1% accuracy degradation for well-optimized models.

=== WHAT: Quantization Precision Formats and Hardware Alignment

Understanding quantization naming conventions from Red Hat AI repository:

**Naming Convention Examples:**

* **`.w4a16`**: Weights quantized to 4-bit, activations at 16-bit (FP16/BF16) - "weight-only quantization"
* **`.w8a8-INT8`**: Both weights and activations at 8-bit integer precision
* **`.w8a8-FP8`**: Weights and activations at 8-bit floating-point format

**Precision Comparison with Hardware Alignment:**

[cols="1,2,2,2", options="header"]
|===
| Precision | Best Use Case | Hardware Acceleration | Accuracy Risk

| **FP16 / BF16**
| Baseline accuracy, maximum precision
| All modern GPUs (universal support)
| **None** (baseline reference)

| **FP8**
| Optimal balance of speed and accuracy for Hopper/Blackwell GPUs
| Natively accelerated on **H100 (Hopper), L4/L40S (Ada), B100 (Blackwell)** - 2× throughput vs FP16
| **Low** (~0.1-0.3% perplexity degradation)

| **INT8**
| Good speed and memory savings on Ampere architecture
| **A100 (Ampere)** has INT8 Tensor Cores; widely supported across GPU generations
| **Medium** (~0.3-0.8% perplexity degradation with proper calibration)

| **INT4**
| Maximum memory reduction for fitting large models on limited hardware
| Supported via software (vLLM, TGI); no dedicated hardware acceleration
| **Highest** (~1-3% perplexity degradation; requires advanced quantization methods like GPTQ/AWQ)
|===

**Critical insight:** Quantization effectiveness depends on **alignment between precision format and GPU hardware capabilities**.

**Example:**

* **H100 deployment with INT8 quantization:** Misses native FP8 Tensor Core acceleration (2× speed advantage unused)
* **A100 deployment with FP8 quantization:** No native hardware support; runs at FP16 speed despite memory savings

== Multi-GPU Strategies: Tensor and Pipeline Parallelism

=== WHY: When Single-GPU Capacity Is Exhausted

// SUGGESTION: Added business context for multi-GPU deployment necessity
**Scenario: The 405B model scaling challenge**

Customer requires Llama-405B deployment (405 GB in INT8) for enterprise knowledge base serving 200 concurrent users. Single H100-80GB provides only 80 GB VRAM.

**Naive approach:** "We'll add more GPUs using any parallelism strategy."

**Cost of network misconfiguration:**

Platform team configures Tensor Parallelism across 8 GPUs in two servers connected by 10 Gbps Ethernet (not NVLink). Deployment starts but performance collapses:

* **Expected throughput:** 4,000 tokens/sec (based on single-GPU benchmarks × 8)
* **Actual throughput:** 400 tokens/sec (10% of expected)
* **GPU utilization:** 15% (DCGM shows GPUs idle 85% of time)
* **P99 latency:** 950 ms (vs 65 ms target)

**Root cause:** Tensor Parallelism performs all-reduce synchronization after every transformer layer (80+ times per forward pass). Over 10 Gbps Ethernet:

* Synchronization latency: ~100 microseconds per layer
* Total sync overhead: 80 layers × 100 μs = 8,000 μs (8 ms)
* Actual compute time: ~1 ms per layer
* **GPUs spend 88% of time waiting for network synchronization**

**Business impact:** $240K GPU hardware investment (8× H100) delivering 10% of expected capacity. Customer demo fails SLA requirements. Project credibility destroyed.

**Correct approach (from this section):** Verify network topology before parallelism configuration. Use Tensor Parallelism only within NVLink-connected GPUs (single node), Pipeline Parallelism across nodes with slower interconnects.
// END SUGGESTION

=== WHAT: Understanding Tensor and Pipeline Parallelism

**Tensor Parallelism (TP): "Splitting the Layer"**

Tensor Parallelism distributes individual weight matrices across multiple GPUs. Each GPU computes a portion of each layer's operations in parallel.

**How it works:**

* Each GPU holds a "shard" of layer weights (e.g., 4 GPUs each hold 25% of weights)
* During forward pass, GPUs perform partial matrix multiplications simultaneously
* Results are synchronized via all-reduce operations before proceeding to next layer
* Synchronization occurs **after every layer** (80-100 times per model forward pass)

**Best for:**

* **Intra-node scaling:** GPUs within same physical server connected by NVLink (900 GB/s bidirectional bandwidth)
* **Latency-critical workloads:** Parallel computation reduces time-per-token significantly
* **Models too large for single GPU:** Splits model across available VRAM without layer-wise dependencies

**Pipeline Parallelism (PP): "Splitting the Stack"**

Pipeline Parallelism divides model by layers. Different GPUs process different layer ranges sequentially.

**How it works:**

* Model's 80 layers split across GPUs (e.g., GPU 0: layers 1-40, GPU 1: layers 41-80)
* GPU 0 processes input through its layers, passes activations to GPU 1
* GPU 1 processes activations through its layers, produces final output
* Uses micro-batching to keep all GPUs busy (process multiple requests in pipeline stages)

**The "bubble" problem:**

Without micro-batching, GPU 1 sits idle while GPU 0 processes first half of model. Micro-batching splits each request into smaller chunks, pipelining them so GPU 1 starts processing chunk 1 while GPU 0 processes chunk 2.

**Best for:**

* **Inter-node scaling:** Multiple servers connected by InfiniBand or high-speed Ethernet (200-400 Gbps)
* **Throughput-critical workloads:** Process many requests in parallel through pipeline stages
* **Models requiring multi-node deployment:** When model exceeds capacity of largest single node

**Networking Requirements: Critical Performance Factor**

[cols="2,2,2", options="header"]
|===
| Characteristic | Tensor Parallelism (TP) | Pipeline Parallelism (PP)

| **Communication Frequency**
| **Extremely High** (every layer: 80-100× per forward pass)
| **Low** (once per pipeline stage)

| **Data Volume Transferred**
| Large (activation tensors, gradients)
| Small (activations only)

| **Required Latency**
| <1-2 microseconds
| Milliseconds (more tolerant)

| **Ideal Network Hardware**
| **NVLink / NVSwitch** (900 GB/s, <1 μs latency)
| **InfiniBand / RoCE / 400G Ethernet** (200-400 Gbps, 1-5 μs latency)

| **Deployment Scope**
| **Intra-node only** (inside single server)
| **Inter-node** (across servers)
|===

=== HOW: Configuring Tensor Parallelism for Production

**Prerequisites:**

* Model VRAM requirement calculated from "Model Sizing" section above
* NVLink topology verified (see Step 1 below)

**Step 1: Verify NVLink Topology**

Before configuring Tensor Parallelism, confirm GPUs are NVLink-connected (not PCIe only):

[source,bash]
----
oc debug node/<gpu-worker-node> -- chroot /host nvidia-smi topo -m
----

**Expected output for NVLink-connected GPUs:**

```
        GPU0    GPU1    GPU2    GPU3    GPU4    GPU5    GPU6    GPU7
GPU0     X      NV18    NV18    NV18    NV18    NV18    NV18    NV18
GPU1    NV18     X      NV18    NV18    NV18    NV18    NV18    NV18
GPU2    NV18    NV18     X      NV18    NV18    NV18    NV18    NV18
...
```

**Key indicator:**

* ✅ **`NV18`** = NVLink connection (18 lanes, 900 GB/s bidirectional bandwidth) - **TP viable**
* ❌ **`PHB`** = PCIe Host Bridge only (no NVLink) - **TP NOT viable, use Data Parallelism instead**

**Step 2: Calculate TP Degree**

TP degree = number of GPUs sharing tensor shards (must all be on same node).

**Formula:**

[stem]
++++
TP_{degree} = \lceil VRAM_{model} \div VRAM_{per\_GPU} \rceil
++++

**Example:** Llama-405B-INT8 on H100-80GB cluster

```
Model VRAM: 405 GB (from Step 5 of sizing procedure)
Available VRAM per GPU: 80 GB
Minimum TP degree: 405 ÷ 80 = 5.06 → round up to 6 GPUs
```

**Validate headroom for KV cache:**

```
VRAM per GPU with TP=6: 405 GB ÷ 6 = 67.5 GB
Remaining for KV cache: 80 GB - 67.5 GB = 12.5 GB per GPU
Total KV cache capacity: 12.5 GB × 6 = 75 GB
```

**For 128K context:** Requires ~80 GB KV cache total (from sizing formula). **TP=6 is marginal.**

**Revised decision:** Use TP=8 for comfortable headroom:

```
VRAM per GPU with TP=8: 405 GB ÷ 8 = 50.6 GB
Remaining for KV cache: 80 GB - 50.6 GB = 29.4 GB per GPU
Total KV cache capacity: 29.4 GB × 8 = 235 GB (sufficient for 128K context)
```

[WARNING]
====
**Critical: Never Use Tensor Parallelism Across Slow Networks**

**Never configure TP across:**

* Standard Ethernet (1 GbE, 10 GbE, even 100 GbE)
* PCIe connections between GPUs (without NVLink)
* Inter-node connections (even with InfiniBand)

**Why this configuration fails catastrophically:**

Tensor Parallelism performs all-reduce synchronization **after every transformer layer**. For typical 80-layer model:

* **NVLink latency:** <1 microsecond per sync → 80 μs total overhead per forward pass
* **10 GbE latency:** ~100 microseconds per sync → 8,000 μs (8 ms) total overhead
* **Actual compute time per forward pass:** ~10 ms

**Performance calculation:**

* **With NVLink:** 80 μs sync + 10 ms compute = ~10 ms total (sync is 0.8% overhead)
* **With 10 GbE:** 8 ms sync + 10 ms compute = ~18 ms total (sync is 44% overhead, but actually worse due to blocking)

**Observed failure mode in production:**

* GPU utilization drops to <20% (GPUs idle waiting for network 80% of time)
* Throughput collapses to <10% of single-GPU performance
* Token generation appears "stuttering" or "frozen" to end users
* DCGM shows low compute utilization despite high power draw (GPUs waiting consumes power)

**Correct approach:**

* **TP only within single node** connected by NVLink (verify with `nvidia-smi topo -m`)
* **Use Pipeline Parallelism** for inter-node scaling (tolerates millisecond-scale latency)
* **Hybrid TP+PP:** TP=8 within each node (NVLink), PP=2 across nodes (InfiniBand)

**Multi-Node Scaling: Hybrid TP+PP Strategy**

For models requiring multiple nodes (>8 GPUs with NVLink per node):

**Example:** Llama-405B deployment on 2 nodes, each with 8× H100-80GB NVLink-connected

**Configuration:**

[source,yaml]
----
vllm_args:
  --tensor-parallel-size=8      # TP within each node (NVLink)
  --pipeline-parallel-size=2    # PP across 2 nodes (InfiniBand)
  # Total GPUs: 8 × 2 = 16 GPUs
----

**How layers are distributed:**

* **Node 1:** Layers 1-63 (first half), each layer sharded across 8 GPUs via TP
* **Node 2:** Layers 64-126 (second half), each layer sharded across 8 GPUs via TP

**Performance characteristics:**

* **Latency:** Higher than single-node TP=8 (pipeline introduces "bubble" overhead ~30 ms)
* **Throughput:** Much higher (16 GPUs process more concurrent requests)
* **Best for:** Batch workloads prioritizing throughput over latency

[NOTE]
====
**MIG Limitations for Multi-GPU Deployments**

Reference Chapter 2 MIG constraints: Why can't Tensor Parallelism span MIG instances?

**Technical reason:**

* Each MIG instance has separate CUDA context (isolated GPU partition)
* NVLink operates at physical GPU level, not between MIG instances
* TP all-reduce requires direct GPU-to-GPU communication via NVLink
* MIG instances on same physical GPU cannot communicate via NVLink (architectural isolation)

**Implication:** Models requiring multi-GPU TP (>70B parameters typically) must use full GPU allocation, not MIG instances.

**When to use MIG for model serving:**

* **Multiple small models:** Deploy 7× Llama-7B instances in separate MIG profiles (7× `1g.10gb` instances)
* **Multi-tenant isolation:** Different teams/customers get dedicated MIG instances with guaranteed resources
* **Development/testing:** Partition GPU for parallel experimentation (different quantizations, hyperparameters)

**Design rule:** Large models (>70B) requiring multi-GPU → use full GPUs with TP/PP. Small models (<13B) in multi-tenant environments → consider MIG for cost efficiency and isolation.

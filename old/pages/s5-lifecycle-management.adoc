= Operator Lifecycle Management

_Estimated reading time: *{time_estimate} minutes*._

Objective::

Understand how to manage operator lifecycle including configuration changes, upgrades, rollbacks, and monitoring operator health for day 2 operations on production AI platforms.

== Introduction

Deploying an operator is just the beginning. Day 2 operations - ongoing management, configuration changes, upgrades, and monitoring - are where operators deliver long-term value for enterprise AI platforms.

This section covers:

* Modifying operator configuration by editing Custom Resources
* Managing operator upgrades through OLM
* Monitoring operator and component health
* Understanding rollback limitations and strategies

== Modifying Operator Configuration

Operators are designed for declarative reconfiguration. To change configuration, edit the Custom Resource spec, and the operator automatically reconciles to the new desired state.

=== Example: Changing GPU Driver Version

A common day 2 operation is upgrading GPU drivers for bug fixes, performance improvements, or new feature support.

**Current state:**

[source,bash]
----
oc get clusterpolicy gpu-cluster-policy -o yaml | grep -A 3 driver:
----

[source,yaml]
----
spec:
  driver:
    enabled: true
    version: "535.129.03"
----

**Goal:** Upgrade to driver version `550.54.15`

**Step 1: Edit the ClusterPolicy**

[source,bash]
----
oc edit clusterpolicy gpu-cluster-policy
----

**Step 2: Change driver version in the spec**

[source,yaml]
----
spec:
  driver:
    enabled: true
    version: "550.54.15"  # Updated from 535.129.03
----

Save and close the editor.

**Step 3: Operator reconciles automatically**

The GPU Operator controller immediately detects the ClusterPolicy change:

1. **Observes:** ClusterPolicy spec changed
2. **Compares:** Desired driver version (550.54.15) vs. current DaemonSet image (535.129.03)
3. **Acts:** Updates `nvidia-driver-daemonset` image tag
4. **Kubernetes performs rolling update:** One node at a time
5. **Updates status:** ClusterPolicy status reflects new driver version

**Step 4: Monitor the rolling update**

[source,bash]
----
# Watch DaemonSet update progress
oc get daemonset nvidia-driver-daemonset -n nvidia-gpu-operator -w

# Check rollout status
oc rollout status daemonset/nvidia-driver-daemonset -n nvidia-gpu-operator

# Verify new driver version on nodes
oc get nodes -o json | jq '.items[].metadata.labels |
  select(."nvidia.com/cuda.driver.major") |
  {node: ."kubernetes.io/hostname", driver_major: ."nvidia.com/cuda.driver.major", driver_minor: ."nvidia.com/cuda.driver.minor"}'
----

=== Rolling Update Behavior

Most GPU Operator components use DaemonSets with `RollingUpdate` strategy:

[source,yaml]
----
updateStrategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 1  # Update one node at a time
----

**Impact during driver update:**

* Nodes update sequentially (one at a time by default)
* During update, node's GPUs are temporarily unavailable
* Running GPU workloads on updating node are evicted
* Update proceeds to next node only after previous node is ready

**Production Best Practice: Controlled Updates**

For production GPU clusters, coordinate driver updates with workload teams:

[source,bash]
----
# 1. Cordon GPU node (prevent new workloads)
oc adm cordon <gpu-node-name>

# 2. Drain GPU workloads gracefully
oc adm drain <gpu-node-name> \
  --ignore-daemonsets \
  --delete-emptydir-data \
  --grace-period=300

# 3. Update proceeds on drained node
# (GPU Operator detects spec change and updates driver)

# 4. Verify node is healthy after update
oc get node <gpu-node-name>
oc describe node <gpu-node-name> | grep nvidia.com/gpu

# 5. Uncordon node when ready
oc adm uncordon <gpu-node-name>

# 6. Repeat for remaining nodes
----

This controlled approach ensures:

* Predictable update timing
* Graceful workload shutdown
* Verification before proceeding to next node

=== Example: Enabling Additional Components

You can enable or disable GPU Operator components by editing the ClusterPolicy.

**Scenario:** Enable Node Status Exporter for additional GPU telemetry

**Current state:**

[source,yaml]
----
spec:
  nodeStatusExporter:
    enabled: false
----

**Edit ClusterPolicy:**

[source,bash]
----
oc edit clusterpolicy gpu-cluster-policy
----

**Change:**

[source,yaml]
----
spec:
  nodeStatusExporter:
    enabled: true  # Changed from false
----

**Operator reconciliation:**

* Detects `nodeStatusExporter.enabled: true`
* Creates `nvidia-node-status-exporter` DaemonSet
* Deploys exporter pods on all GPU nodes
* Begins publishing node status metrics to Prometheus

**Verification:**

[source,bash]
----
# Check new DaemonSet was created
oc get daemonset nvidia-node-status-exporter -n nvidia-gpu-operator

# Verify pods are running
oc get pods -n nvidia-gpu-operator -l app=nvidia-node-status-exporter

# Check metrics endpoint
oc exec -n nvidia-gpu-operator <exporter-pod> -- curl localhost:9400/metrics
----

== Operator Upgrades

OLM manages operator upgrades based on the Subscription's channel and approval strategy.

=== How OLM Detects New Versions

1. **Catalog polling:** OLM periodically checks CatalogSources for new operator versions
2. **Channel tracking:** Based on Subscription's `channel`, OLM identifies latest available version
3. **Version comparison:** If newer version exists, OLM creates InstallPlan
4. **Approval workflow:** Automatic or manual based on `installPlanApproval` setting

=== Viewing Current Operator Version

[source,bash]
----
# Check installed GPU Operator version
oc get csv -n nvidia-gpu-operator

NAME                               DISPLAY              VERSION   PHASE
gpu-operator-certified.v23.9.0     NVIDIA GPU Operator  23.9.0    Succeeded
----

=== Checking for Available Updates

[source,bash]
----
# View all available versions in GPU Operator's channels
oc get packagemanifest gpu-operator-certified -o json | \
  jq '.status.channels[] | {name:.name, version:.currentCSVDesc.version}'
----

**Example output:**

[source,json]
----
{
  "name": "v23.9",
  "version": "23.9.2"
}
{
  "name": "v24.3",
  "version": "24.3.0"
}
----

This shows:

* `v23.9` channel: Latest version is 23.9.2 (minor update available)
* `v24.3` channel: New major version available

=== Automatic Upgrade Workflow

**Subscription configured with automatic approval:**

[source,yaml]
----
spec:
  channel: "v23.9"
  installPlanApproval: Automatic
----

**When v23.9.2 is released:**

1. OLM detects new version in `v23.9` channel
2. OLM creates InstallPlan for upgrade to v23.9.2
3. **InstallPlan is auto-approved** (no admin action required)
4. OLM upgrades CSV from v23.9.0 to v23.9.2
5. GPU Operator pod restarts with new version
6. Operator reconciles any CRD or CR changes

**Monitoring automatic upgrades:**

[source,bash]
----
# Watch for CSV changes
oc get csv -n nvidia-gpu-operator -w

# Check InstallPlan history
oc get installplan -n nvidia-gpu-operator --sort-by=.metadata.creationTimestamp
----

[WARNING]
====
Automatic upgrades can occur anytime during business hours. For production AI platforms serving revenue-generating workloads, use **Manual approval** instead.
====

=== Manual Upgrade Workflow

**Subscription configured with manual approval:**

[source,yaml]
----
spec:
  channel: "v23.9"
  installPlanApproval: Manual
----

**When v23.9.2 is released:**

1. OLM detects new version in `v23.9` channel
2. OLM creates InstallPlan for upgrade
3. **InstallPlan waits for manual approval**
4. Admin reviews, tests in dev, then approves
5. OLM proceeds with upgrade

**Manual approval workflow:**

**Step 1: Check for pending upgrades**

[source,bash]
----
oc get installplan -n nvidia-gpu-operator

NAME            CSV                                  APPROVAL   APPROVED
install-xxxxx   gpu-operator-certified.v23.9.0      Automatic  true      # Current
install-yyyyy   gpu-operator-certified.v23.9.2      Manual     false     # Pending
----

**Step 2: Review InstallPlan details**

[source,bash]
----
oc describe installplan install-yyyyy -n nvidia-gpu-operator
----

Shows:

* CSV version to install (v23.9.2)
* CRDs to create/update
* RBAC resources changes
* Required Kubernetes version

**Step 3: Review release notes (external)**

Check NVIDIA GPU Operator release notes:

* https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/release-notes.html
* Look for breaking changes, new features, bug fixes
* Check for required ClusterPolicy configuration changes

**Step 4: Test upgrade in development environment**

Before approving production upgrade:

1. Deploy same OpenShift version in dev cluster
2. Install same GPU Operator version as production (v23.9.0)
3. Approve upgrade to v23.9.2 in dev
4. Test representative GPU workloads
5. Monitor for issues

**Step 5: Approve production upgrade**

[source,bash]
----
# Approve InstallPlan
oc patch installplan install-yyyyy -n nvidia-gpu-operator \
  --type merge --patch '{"spec":{"approved":true}}'

installplan.operators.coreos.com/install-yyyyy patched
----

**Step 6: Monitor upgrade progress**

[source,bash]
----
# Watch CSV status
oc get csv -n nvidia-gpu-operator -w

NAME                               DISPLAY              VERSION   PHASE
gpu-operator-certified.v23.9.0     NVIDIA GPU Operator  23.9.0    Replacing
gpu-operator-certified.v23.9.2     NVIDIA GPU Operator  23.9.2    Installing
gpu-operator-certified.v23.9.2     NVIDIA GPU Operator  23.9.2    Succeeded

# Check operator pod restarted with new version
oc get pods -n nvidia-gpu-operator -l app=gpu-operator

# Verify ClusterPolicy still reconciling
oc get clusterpolicy gpu-cluster-policy -o yaml | grep -A 5 status:
----

=== Changing Channels (Major Version Upgrades)

Upgrading to a new major version (e.g., v23.9 → v24.3) requires changing the Subscription channel.

**Current Subscription:**

[source,yaml]
----
spec:
  channel: "v23.9"
----

**Edit Subscription to change channel:**

[source,bash]
----
oc edit subscription gpu-operator-certified -n nvidia-gpu-operator
----

**Change:**

[source,yaml]
----
spec:
  channel: "v24.3"  # Changed from v23.9
----

**OLM behavior:**

* Detects channel change
* Resolves latest version in `v24.3` channel (e.g., v24.3.0)
* Creates InstallPlan for v24.3.0
* If `installPlanApproval: Manual`, waits for approval
* If `installPlanApproval: Automatic`, proceeds immediately

[WARNING]
====
**Major version upgrades may include breaking changes:**

* CRD schema changes (ClusterPolicy fields added/removed/renamed)
* Deprecated features removed
* New configuration requirements
* Different driver version defaults

**Always test major upgrades in development environment first.**

Review the GPU Operator upgrade documentation for version-specific migration steps.
====

== Monitoring Operator Health

Effective operator management requires monitoring operator health and component status.

=== ClusterServiceVersion (CSV) Status

The CSV tracks operator deployment and health:

[source,bash]
----
# Check CSV status
oc get csv gpu-operator-certified.v23.9.0 -n nvidia-gpu-operator -o yaml | \
  grep -A 20 status:
----

**Key status fields:**

[source,yaml]
----
status:
  phase: Succeeded         # Pending, Installing, Succeeded, Failed, Replacing
  reason: InstallSucceeded
  message: install strategy completed with no errors
  conditions:
  - type: Ready
    status: "True"
    reason: AllRequirementsMet
    lastTransitionTime: "2024-04-10T15:30:45Z"
----

**CSV Phases:**

* **Succeeded:** Operator installed and running normally
* **Installing:** Operator deployment in progress
* **Replacing:** Being replaced by newer version during upgrade
* **Failed:** Installation or operation failed (check events and logs)
* **Pending:** Waiting for dependencies or requirements

=== Operator Pod Logs

View operator controller logs for reconciliation activity and errors:

[source,bash]
----
# Find operator pod
oc get pods -n nvidia-gpu-operator -l app=gpu-operator

# Stream logs
oc logs -f -n nvidia-gpu-operator deployment/gpu-operator

# View recent logs
oc logs -n nvidia-gpu-operator deployment/gpu-operator --tail=100
----

**Common log patterns:**

* **Reconciliation cycles:** `Reconciling ClusterPolicy gpu-cluster-policy`
* **Errors:** `ERROR: Failed to create DaemonSet: ...`
* **Warnings:** `WARN: Node missing label: nvidia.com/gpu.present`
* **Status updates:** `ClusterPolicy status updated: state=ready`

=== Custom Resource Status

The ClusterPolicy status subresource shows component health:

[source,bash]
----
# Check ClusterPolicy status
oc get clusterpolicy gpu-cluster-policy -o yaml | grep -A 30 status:
----

**Example status:**

[source,yaml]
----
status:
  state: ready              # notReady, ignored, ready
  conditions:
  - type: Ready
    status: "True"
    lastUpdateTime: "2024-04-10T15:30:45Z"
    reason: AllComponentsReady
  - type: DriverReady
    status: "True"
  - type: ToolkitReady
    status: "True"
  - type: DevicePluginReady
    status: "True"
  - type: DCGMReady
    status: "True"
  - type: GFDReady
    status: "True"
----

**Interpreting status:**

* `state: ready` → All components deployed and running successfully
* `state: notReady` → Some components failed or pending (check conditions)
* Individual component conditions show granular health

**If ClusterPolicy shows `state: notReady`:**

[source,bash]
----
# Check which component is failing
oc get clusterpolicy gpu-cluster-policy -o yaml | \
  grep -A 3 'status: "False"'

# Check DaemonSet status for that component
oc get daemonsets -n nvidia-gpu-operator

# Check pod events
oc get events -n nvidia-gpu-operator --sort-by='.lastTimestamp' | tail -20
----

=== Prometheus Metrics (Advanced)

The GPU Operator exposes operator metrics for monitoring:

[source,bash]
----
# Check if ServiceMonitor exists
oc get servicemonitor -n nvidia-gpu-operator
----

**Operator metrics available:**

* `gpu_operator_reconcile_count` - Number of reconciliation loops
* `gpu_operator_reconcile_errors_total` - Reconciliation failures
* `gpu_operator_reconcile_duration_seconds` - Reconciliation latency

These metrics can be queried in Prometheus or viewed in Grafana dashboards.

== Rollback Considerations

OLM does **not** support automatic operator rollback. Rollbacks require manual intervention and carry risks.

=== Why Rollbacks are Risky

**CRD schema evolution example:**

* **v23.9:** ClusterPolicy has field `driver.version`
* **v24.3:** Field renamed to `driver.imageTag` (breaking schema change)

**Upgrade scenario:**

1. Start with v23.9, ClusterPolicy uses `driver.version: "535.129.03"`
2. Upgrade operator to v24.3
3. v24.3 CRD expects `driver.imageTag`, not `driver.version`
4. Existing ClusterPolicy may fail validation

**Rollback scenario:**

1. Delete v24.3 CSV, install v23.9 CSV
2. v23.9 CRD expects `driver.version`
3. If ClusterPolicy was updated to use `driver.imageTag`, it's now invalid
4. Result: Broken configuration

=== Manual Rollback Process

[WARNING]
====
Only attempt rollback if absolutely necessary. Test rollback procedure in development environment first.
====

**Step 1: Delete current CSV**

[source,bash]
----
# This stops the operator but doesn't delete CRDs or CRs
oc delete csv gpu-operator-certified.v24.3.0 -n nvidia-gpu-operator
----

**Step 2: Edit Subscription to previous channel**

[source,bash]
----
oc edit subscription gpu-operator-certified -n nvidia-gpu-operator

# Change: channel: "v23.9"  (from v24.3)
----

**Step 3: OLM reinstalls previous version**

OLM resolves v23.9 channel, creates InstallPlan, deploys v23.9 CSV.

**Step 4: Verify CRDs are compatible**

[source,bash]
----
# Check if ClusterPolicy is valid against v23.9 CRD
oc get clusterpolicy gpu-cluster-policy -o yaml
----

If validation errors occur, you may need to edit the ClusterPolicy to match v23.9 schema.

**Step 5: Verify components are running**

[source,bash]
----
oc get pods -n nvidia-gpu-operator
oc get clusterpolicy gpu-cluster-policy
----

=== Avoiding Rollbacks: Best Practices

1. **Test upgrades in dev environment first:** Catch issues before production
2. **Use manual approval for production:** Control when upgrades occur
3. **Backup Custom Resources before upgrade:**

   [source,bash]
   ----
   oc get clusterpolicy gpu-cluster-policy -o yaml > clusterpolicy-backup-$(date +%Y%m%d).yaml
   ----

4. **Review release notes:** Understand breaking changes before upgrading
5. **Pin to version-specific channels:** Avoid unexpected major version jumps

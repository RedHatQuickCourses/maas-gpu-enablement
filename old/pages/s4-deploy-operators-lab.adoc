= Lab: Deploying the Base Stack

_Estimated reading time: *{time_estimate} minutes*._

Objective::

Install the Node Feature Discovery Operator and NVIDIA GPU Operator, configure the ClusterPolicy to deploy all GPU stack components, and verify successful installation by inspecting the daemonset pods.

== Before you begin

This lab requires:

* Access to an OpenShift cluster (version 4.20 or higher)
* Cluster administrator privileges
* At least one worker node with NVIDIA GPU hardware
* The `oc` CLI tool installed and configured

== Instructions

=== 1. Install the Node Feature Discovery Operator

Node Feature Discovery must be installed first as it provides the hardware detection capabilities required by the GPU Operator.

. Log in to your OpenShift cluster as a cluster administrator
+
[source,console,subs="verbatim,quotes"]
----
$ *oc login -u admin https://api.cluster.example.com:6443*
Login successful.
----

. Create the Node Feature Discovery Operator subscription
+
[source,console,subs="verbatim,quotes"]
----
$ *cat <<EOF | oc apply -f -*
*apiVersion: v1*
*kind: Namespace*
*metadata:*
*  name: openshift-nfd*
*---*
*apiVersion: operators.coreos.com/v1*
*kind: OperatorGroup*
*metadata:*
*  name: nfd-operator-group*
*  namespace: openshift-nfd*
*spec:*
*  targetNamespaces:*
*  - openshift-nfd*
*---*
*apiVersion: operators.coreos.com/v1alpha1*
*kind: Subscription*
*metadata:*
*  name: nfd*
*  namespace: openshift-nfd*
*spec:*
*  channel: "stable"*
*  name: nfd*
*  source: redhat-operators*
*  sourceNamespace: openshift-marketplace*
*EOF*

----

====
.look for the following output indicating successful creation of the subscription and operator group:


 * namespace/openshift-nfd created

 * operatorgroup.operators.coreos.com/nfd-operator-group created

 * subscription.operators.coreos.com/nfd created
====
+
[NOTE]
====
**What just happened? Understanding the OLM Workflow**

You created three resources:

1. **Namespace** (`openshift-nfd`): Isolates NFD operator resources from other operators
2. **OperatorGroup**: Tells OLM this operator manages resources in the `openshift-nfd` namespace
3. **Subscription**: Declares you want the `nfd` operator from the `redhat-operators` catalog on the `stable` channel

**Behind the scenes, OLM is now working:**

* OLM's Catalog Operator detected your Subscription
* It resolved the latest `stable` version of NFD from the `redhat-operators` catalog
* It created an InstallPlan with the execution steps
* It's deploying the NFD operator's ClusterServiceVersion (CSV)
* The NFD operator pod is starting

Check the InstallPlan OLM created:

[source,bash]
----
oc get installplan -n openshift-nfd
----

You'll see OLM's execution plan for installing NFD.
====

. Wait for the NFD Operator to be ready
+
[source,console,subs="verbatim,quotes"]
----
$ *oc get csv -n openshift-nfd*
NAME                  DISPLAY                      VERSION   REPLACES   PHASE
nfd.v4.14.0-202401   Node Feature Discovery       4.14.0               *Succeeded*
----

. Create the NodeFeatureDiscovery instance
+
[source,console,subs="verbatim,quotes"]
----
$ *cat <<EOF | oc apply -f -*
*apiVersion: nfd.openshift.io/v1*
*kind: NodeFeatureDiscovery*
*metadata:*
*  name: nfd-instance*
*  namespace: openshift-nfd*
*spec:*
*  operand:*
*    image: registry.redhat.io/openshift4/ose-node-feature-discovery:v4.14*
*  workerConfig:*
*    configData: |*
*      sources:*
*        pci:*
*          deviceClassWhitelist:*
*            - "0300"*
*            - "0302"*
*          deviceLabelFields:*
*            - "vendor"*
*EOF*
nodefeaturediscovery.nfd.openshift.io/nfd-instance created
----
+
[NOTE]
====
**Custom Resource in Action**

You just created a `NodeFeatureDiscovery` Custom Resource (CR). This is an instance of the `NodeFeatureDiscovery` CRD that the NFD operator installed.

**What's happening now:**

* **Desired state (your CR spec):** NFD should run on all nodes, scan for PCI devices in classes `0300` and `0302` (GPU display controllers)
* **NFD operator's controller:** Watching for `NodeFeatureDiscovery` CRs
* **Reconciliation:** Operator creates NFD worker DaemonSet to match your desired state

**The operator pattern's self-healing:**

Try this experiment after NFD is running:

[source,bash]
----
# Delete the NFD worker DaemonSet
oc delete daemonset nfd-worker -n openshift-nfd

# Watch it get recreated automatically by the operator
oc get daemonset nfd-worker -n openshift-nfd -w
----

The operator detects the missing DaemonSet and recreates it because your CR says NFD should be running. This is the reconciliation loop in action - the operator continuously ensures actual state matches desired state.
====

. Verify NFD is labeling GPU nodes
+
[source,console,subs="verbatim,quotes"]
----
$ *oc get nodes -l feature.node.kubernetes.io/pci-10de.present=true*
NAME                         STATUS   ROLES    AGE   VERSION
worker-gpu-0.example.com     Ready    worker   5d    v1.27.6+b49f9d1
worker-gpu-1.example.com     Ready    worker   5d    v1.27.6+b49f9d1
----
+
[NOTE]
====
The label `feature.node.kubernetes.io/pci-10de.present=true` indicates NVIDIA hardware (vendor ID 10de) was detected on these nodes.
====

=== 2. Install the NVIDIA GPU Operator

Now that NFD is running and detecting GPU hardware, install the NVIDIA GPU Operator to deploy the complete software stack.

. Create the namespace and operator subscription
+
[source,console,subs="verbatim,quotes"]
----
$ *cat <<EOF | oc apply -f -*
*apiVersion: v1*
*kind: Namespace*
*metadata:*
*  name: nvidia-gpu-operator*
*---*
*apiVersion: operators.coreos.com/v1*
*kind: OperatorGroup*
*metadata:*
*  name: nvidia-gpu-operator-group*
*  namespace: nvidia-gpu-operator*
*spec:*
*  targetNamespaces:*
*  - nvidia-gpu-operator*
*---*
*apiVersion: operators.coreos.com/v1alpha1*
*kind: Subscription*
*metadata:*
*  name: gpu-operator-certified*
*  namespace: nvidia-gpu-operator*
*spec:*
*  channel: "v23.9"*
*  name: gpu-operator-certified*
*  source: certified-operators*
*  sourceNamespace: openshift-marketplace*
*EOF*
namespace/nvidia-gpu-operator created
operatorgroup.operators.coreos.com/nvidia-gpu-operator-group created
subscription.operators.coreos.com/gpu-operator-certified created
----
+
[IMPORTANT]
====
**Production Decision Point: Update Approval Strategy**

Notice the Subscription spec doesn't include `installPlanApproval`. This defaults to `Automatic`, meaning OLM will automatically upgrade the GPU Operator when new versions are released in the `v23.9` channel.

**For production AI platforms, consider using Manual approval:**

[source,yaml]
----
spec:
  channel: "v23.9"
  name: gpu-operator-certified
  source: certified-operators
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual  # Add this line for production
----

**Trade-offs:**

* **Automatic approval:**
  * Pros: Security patches applied immediately, always up-to-date
  * Cons: Risk of breaking changes, upgrades during business hours
  * Use case: Development environments

* **Manual approval:**
  * Pros: Control upgrade timing, test in dev first, review release notes
  * Cons: Manual overhead, delayed security patches
  * Use case: Production environments

**If using Manual approval, approve upgrades with:**

[source,bash]
----
# View pending upgrades
oc get installplan -n nvidia-gpu-operator

# Approve specific InstallPlan
oc patch installplan <plan-name> -n nvidia-gpu-operator \
  --type merge --patch '{"spec":{"approved":true}}'
----

For this lab, we're using Automatic for simplicity, but production deployments should use Manual.
====

+
[source,console,subs="verbatim,quotes"]
----

gpu-operator-certified.v23.9.0   NVIDIA GPU Operator   23.9.0               *Succeeded*

=== 3. Configure the ClusterPolicy

The ClusterPolicy custom resource configures how the GPU Operator deploys all stack components.

. Create a customized ClusterPolicy
+
[source,console,subs="verbatim,quotes"]
----
$ *cat <<EOF | oc apply -f -*
*apiVersion: nvidia.com/v1*
*kind: ClusterPolicy*
*metadata:*
*  name: gpu-cluster-policy*
*spec:*
*  operator:*
*    defaultRuntime: crio*
*  driver:*
*    enabled: true*
*    version: "535.129.03"*
*  toolkit:*
*    enabled: true*
*  devicePlugin:*
*    enabled: true*
*  dcgm:*
*    enabled: true*
*  dcgmExporter:*
*    enabled: true*
*  gfd:*
*    enabled: true*
*  migManager:*
*    enabled: true*
*  nodeStatusExporter:*
*    enabled: true*
*EOF*
clusterpolicy.nvidia.com/gpu-cluster-policy created
----
+
[NOTE]
====
**ClusterPolicy: The GPU Stack's Desired State**

The `ClusterPolicy` Custom Resource you just created is the central configuration for the entire GPU software stack. It tells the GPU Operator your desired state:

* Deploy driver version `535.129.03`
* Enable DCGM monitoring
* Enable DCGM Exporter for Prometheus metrics
* Enable GPU Feature Discovery for detailed node labels
* Enable MIG Manager for Multi-Instance GPU support
* Enable Node Status Exporter for additional telemetry

**The GPU Operator controller reads this CR and reconciles cluster state:**

1. Creates DaemonSets for each component (driver, toolkit, device-plugin, DCGM, etc.)
2. Creates ConfigMaps with component configuration
3. Creates Services for DCGM metrics endpoints
4. Monitors all resources and recreates if deleted
5. Updates ClusterPolicy status subresource with observed state

**This is declarative configuration in action.** To change the driver version later, simply edit the ClusterPolicy:

[source,bash]
----
oc edit clusterpolicy gpu-cluster-policy
# Change: spec.driver.version: "550.54.15"
----

The operator detects the change and performs a rolling update of the driver DaemonSet automatically.
====

=== 5. Verify OLM Resources

Confirm that OLM successfully installed both operators and created the necessary CRDs.

. Check installed operators (CSVs)
+
[source,console,subs="verbatim,quotes"]
----
$ *oc get csv -A | grep -E '(nfd|gpu-operator)'*
openshift-nfd              nfd.v4.14.0-202401                Node Feature Discovery   4.14.0    Succeeded
nvidia-gpu-operator        gpu-operator-certified.v23.9.0    NVIDIA GPU Operator      23.9.0    Succeeded
----
+
Both CSVs show `Succeeded` phase, confirming operators are installed and running.

. View Custom Resource Definitions created by the operators
+
[source,console,subs="verbatim,quotes"]
----
$ *oc get crd | grep -E '(nodefeaturediscoveries|clusterpolicies)'*
clusterpolicies.nvidia.com                           2024-04-10T14:23:45Z
nodefeaturediscoveries.nfd.openshift.io              2024-04-10T14:15:12Z
----
+
These CRDs extend the Kubernetes API with new resource types specific to NFD and GPU management.

. Check InstallPlans to see OLM's installation execution history
+
[source,console,subs="verbatim,quotes"]
----
$ *oc get installplan -n nvidia-gpu-operator*
NAME            CSV                                  APPROVAL    APPROVED
install-xxxxx   gpu-operator-certified.v23.9.0      Automatic   true
----
+
The InstallPlan shows what OLM installed (CSV version, CRDs, RBAC resources).

. View the Subscriptions managing operator lifecycle
+
[source,console,subs="verbatim,quotes"]
----
$ *oc get subscription -A | grep -E '(nfd|gpu-operator)'*
openshift-nfd          nfd                      nfd                      redhat-operators        stable
nvidia-gpu-operator    gpu-operator-certified   gpu-operator-certified   certified-operators     v23.9
----
+
These Subscriptions tell OLM to keep the operators updated according to their respective channels.

[NOTE]
====
**OLM Workflow Review**

You just witnessed the complete OLM workflow:

1. **User action:** Created Subscription
2. **OLM resolution:** Resolved latest operator version from catalog
3. **InstallPlan:** OLM created execution plan
4. **CSV deployment:** OLM deployed operator metadata and pod
5. **CRD creation:** Operator's CRDs registered with Kubernetes API
6. **Operator running:** Operator pod started and ready to reconcile
7. **CR creation:** User created Custom Resource (NodeFeatureDiscovery, ClusterPolicy)
8. **Reconciliation:** Operator controllers made actual state match desired state

This declarative, GitOps-friendly approach is how modern Kubernetes platforms manage complex software lifecycles at scale.
====

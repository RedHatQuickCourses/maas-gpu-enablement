= Operator Lifecycle Manager and Custom Resources

_Estimated reading time: *{time_estimate} minutes*._

Objective::

Navigate OpenShift's Operator Lifecycle Manager architecture, understand Custom Resources and Controllers, and differentiate between cluster and add-on operators for production AI platform decisions.

== Introduction

You've learned why operators are valuable and what components the GPU Operator manages. Now you need to understand the mechanism that makes operators work in OpenShift: the Operator Lifecycle Manager (OLM).

OLM is OpenShift's built-in system for discovering, installing, and managing operators throughout their lifecycle. Think of it as a package manager (like `apt` or `yum`) specifically designed for Kubernetes operators.

This section covers:

* How OLM discovers and installs operators
* The relationship between Subscriptions, CSVs, and InstallPlans
* How Custom Resources and Controllers extend Kubernetes
* When to use Red Hat vs. Certified vs. Community operators

== Operator Lifecycle Manager: Package Management for Operators

=== The Challenge OLM Solves

Without OLM, installing an operator manually requires:

1. Creating namespaces and RBAC permissions
2. Deploying Custom Resource Definitions (CRDs)
3. Creating ServiceAccounts with appropriate permissions
4. Deploying the operator pod
5. Managing upgrades manually
6. Resolving dependency conflicts
7. Tracking installed versions

For a platform with 20+ operators, this becomes unmanageable.

=== OLM's Role

OpenShift's Operator Lifecycle Manager automates operator management:

* **Discovery:** Browse available operators from curated catalogs
* **Installation:** Automated deployment from catalog with dependency resolution
* **Upgrades:** Managed version updates with approval workflows
* **Dependencies:** Automatic resolution of operator requirements
* **Permissions:** RBAC policies defined in operator metadata
* **Lifecycle:** Health monitoring and status tracking

== OLM Architecture: Catalog Sources

OpenShift includes four pre-configured operator catalogs, each serving different use cases.

=== redhat-operators

**Purpose:** Red Hat supported operators for platform extensions

**Characteristics:**

* Full Red Hat support with SLA
* Tested against OpenShift versions
* Coordinated releases with OpenShift
* Production-ready

**Examples:**

* Node Feature Discovery (NFD)
* Red Hat OpenShift AI
* OpenShift Logging
* OpenShift Serverless

**When to use:** Production deployments of Red Hat platform components

=== certified-operators

**Purpose:** Vendor-certified third-party operators

**Characteristics:**

* Vendor support (NVIDIA, MongoDB, etc.)
* Red Hat certified (tested against OpenShift)
* Independent release cycles
* Commercial software

**Examples:**

* NVIDIA GPU Operator (NVIDIA support)
* MongoDB Enterprise Operator (MongoDB support)
* Crunchy PostgreSQL Operator (Crunchy Data support)

**When to use:** Production deployments of vendor software with vendor support contracts

=== community-operators

**Purpose:** Community-maintained operators

**Characteristics:**

* No official support or SLA
* Community contributions
* Varying quality and maintenance levels
* May lag behind upstream versions

**When to use:** Development, testing, POCs, non-production environments only

[WARNING]
====
**Never use community operators in production AI platforms.** They lack support SLAs and may have security vulnerabilities or stability issues.
====

=== redhat-marketplace

**Purpose:** Commercial operators from Red Hat Marketplace

**Characteristics:**

* Pay-per-use or subscription licensing
* Vendor support varies
* Usage metering and billing integration

**When to use:** Commercial ISV software with marketplace billing

== Core OLM Resources

OLM uses several Kubernetes custom resources to manage operator lifecycles. Understanding their relationships is critical for troubleshooting and day 2 operations.

=== The OLM Resource Workflow

----
User Creates          OLM Processes           Operator Starts
─────────────         ─────────────           ───────────────

Subscription    ──►   OLM resolves     ──►    InstallPlan
(desired               operator               (execution
 operator)             version                 plan)
                           │
                           ▼
                      Creates CSV      ──►    Operator Pod
                      (operator               starts
                       metadata)                  │
                           │                      ▼
                           ▼                  Creates CRDs
                      Grants RBAC             (extends API)
                      permissions                 │
                                                  ▼
                                             User creates
                                             Custom Resources
                                                  │
                                                  ▼
                                             Operator
                                             reconciles
----

=== 1. Subscription

A Subscription declares your intent to install or subscribe to an operator.

**Purpose:** Tells OLM which operator you want, from which catalog, on which channel.

**Example:**

[source,yaml]
----
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: gpu-operator-certified
  namespace: nvidia-gpu-operator
spec:
  channel: "v23.9"                           # <1>
  name: gpu-operator-certified               # <2>
  source: certified-operators                # <3>
  sourceNamespace: openshift-marketplace     # <4>
  installPlanApproval: Automatic             # <5>
----
<1> **Channel:** Version stream to subscribe to (stable, fast, v23.9, etc.)
<2> **Name:** Operator package name from catalog
<3> **Source:** Which catalog source (certified-operators, redhat-operators, etc.)
<4> **sourceNamespace:** Where catalog sources live (always openshift-marketplace)
<5> **installPlanApproval:** Automatic or Manual upgrade approval

**Key Configuration: installPlanApproval**

* **Automatic:** OLM auto-approves operator upgrades when new versions are published
  * _Pros:_ Always up-to-date, security patches applied immediately
  * _Cons:_ Risk of breaking changes, unexpected behavior changes
  * _Use case:_ Development environments, non-critical workloads

* **Manual:** Admin must manually approve each upgrade
  * _Pros:_ Control upgrade timing, test in dev first, review release notes
  * _Cons:_ Manual overhead, delayed security patches
  * _Use case:_ Production AI platforms, mission-critical workloads

**Recommendation for Production GPU Platforms:** Use `Manual` approval to control when driver and operator upgrades occur.

=== 2. ClusterServiceVersion (CSV)

The CSV is the operator's metadata and deployment specification.

**Purpose:** Defines what the operator is, what it manages, and how to deploy it.

**What a CSV Contains:**

* Operator name, version, description
* Custom Resource Definitions (CRDs) it owns
* RBAC permissions it requires
* Deployment spec (operator pod template)
* Update channels and versioning
* Dependencies on other operators
* Installation requirements

**Viewing CSVs:**

[source,bash]
----
# List all installed operators (CSVs)
oc get csv -A

# Example output:
NAMESPACE               NAME                               DISPLAY              VERSION   PHASE
nvidia-gpu-operator     gpu-operator-certified.v23.9.0     NVIDIA GPU Operator  23.9.0    Succeeded
openshift-nfd           nfd.v4.14.0-202401                 Node Feature Discovery 4.14.0  Succeeded
----

[source,bash]
----
# View detailed CSV information
oc describe csv gpu-operator-certified.v23.9.0 -n nvidia-gpu-operator
----

**CSV Status Phases:**

[cols="1,3",options="header"]
|===
|Phase
|Meaning

|**Pending**
|Waiting for dependencies or requirements

|**InstallReady**
|Ready to install, waiting for approval (if manual)

|**Installing**
|Operator deployment in progress

|**Succeeded**
|Operator installed and running successfully

|**Failed**
|Installation failed, check events and logs

|**Replacing**
|Being replaced by newer version during upgrade
|===

=== 3. OperatorGroup

An OperatorGroup defines which namespaces an operator can manage.

**Purpose:** Scopes where the operator can create and manage resources.

**Example:**

[source,yaml]
----
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: nvidia-gpu-operator-group
  namespace: nvidia-gpu-operator
spec:
  targetNamespaces:
  - nvidia-gpu-operator  # <1>
----
<1> Operator can only manage resources in the `nvidia-gpu-operator` namespace

**Scope Options:**

**Single Namespace Mode:**

[source,yaml]
----
spec:
  targetNamespaces:
  - my-namespace
----

Operator manages resources only in `my-namespace`.

**Multi-Namespace Mode:**

[source,yaml]
----
spec:
  targetNamespaces:
  - namespace-1
  - namespace-2
  - namespace-3
----

Operator manages resources in multiple specific namespaces.

**All Namespaces Mode (Cluster-wide):**

[source,yaml]
----
spec:
  targetNamespaces: []  # Empty list = all namespaces
----

Operator manages cluster-scoped resources and resources in any namespace.

**GPU Operator Example:**

The GPU Operator typically uses **all namespaces** mode because it manages cluster-wide resources (nodes, DaemonSets that run on all GPU nodes).

=== 4. InstallPlan

An InstallPlan is OLM's execution plan for installing or upgrading an operator.

**Purpose:** Details what OLM will install (CSVs, CRDs, RBAC resources).

**When InstallPlans are Created:**

* Initial operator installation (from Subscription)
* Operator upgrades (when new version available in channel)

**Viewing InstallPlans:**

[source,bash]
----
# List InstallPlans
oc get installplan -n nvidia-gpu-operator

# Example output:
NAME            CSV                                  APPROVAL   APPROVED
install-xxxxx   gpu-operator-certified.v23.9.0      Automatic  true
install-yyyyy   gpu-operator-certified.v23.9.2      Manual     false  # <1>
----
<1> Pending manual approval for upgrade to v23.9.2

**InstallPlan Details:**

[source,bash]
----
oc describe installplan install-yyyyy -n nvidia-gpu-operator
----

Shows:

* CSV version to install
* CRDs to create
* RBAC resources (ServiceAccounts, Roles, RoleBindings)
* Approval status

**Manual Approval Workflow:**

[source,bash]
----
# Approve pending InstallPlan
oc patch installplan install-yyyyy -n nvidia-gpu-operator \
  --type merge --patch '{"spec":{"approved":true}}'

# Watch upgrade proceed
oc get csv -n nvidia-gpu-operator -w
----

== Custom Resources and Controllers: Extending Kubernetes

Operators work by extending Kubernetes with new resource types specific to the application domain. This is accomplished through Custom Resource Definitions (CRDs) and Controllers.

=== The Problem CRDs Solve

Kubernetes has built-in resource types:

* **Pod:** Running container(s)
* **Service:** Network endpoint
* **Deployment:** Declarative pod updates
* **ConfigMap:** Configuration data

But these generic resources don't capture domain-specific concepts like:

* "GPU cluster configuration with driver version and monitoring settings"
* "Multi-Instance GPU partitioning profile"
* "Inference service with autoscaling and model registry integration"

CRDs let you create new resource types that represent these domain concepts.

=== Custom Resource Definition (CRD)

A CRD extends the Kubernetes API with a new resource type.

**Purpose:** Defines the schema, validation rules, and behavior of a new resource type.

**Example: ClusterPolicy CRD**

The NVIDIA GPU Operator creates a CRD called `ClusterPolicy`:

[source,bash]
----
oc get crd clusterpolicies.nvidia.com -o yaml
----

This CRD defines:

* **API group:** `nvidia.com`
* **API version:** `v1`
* **Resource name:** `clusterpolicies`
* **Schema:** What fields are valid (driver.version, dcgm.enabled, etc.)
* **Validation:** Rules like "driver.version must match semantic version pattern"
* **Subresources:** Status tracking for operator to update

**Effect:** After the CRD is created, you can use `oc get clusterpolicy` just like `oc get pod`.

=== Custom Resource (CR)

A Custom Resource is an instance of a CRD - your actual configuration.

**Purpose:** Represents the desired state of your application.

**Example: ClusterPolicy CR**

[source,yaml]
----
apiVersion: nvidia.com/v1       # <1>
kind: ClusterPolicy             # <2>
metadata:
  name: gpu-cluster-policy
spec:                           # <3>
  driver:
    enabled: true
    version: "535.129.03"
  toolkit:
    enabled: true
  devicePlugin:
    enabled: true
  dcgm:
    enabled: true
  dcgmExporter:
    enabled: true
  gfd:
    enabled: true
  migManager:
    enabled: true
----
<1> API version defined by the CRD
<2> Resource type defined by the CRD
<3> Desired state - what you want

**This CR declares:**

* "I want GPU driver version 535.129.03"
* "I want DCGM monitoring enabled"
* "I want MIG Manager enabled"
* "I want all GPU stack components deployed"

**Viewing Custom Resources:**

[source,bash]
----
# List ClusterPolicy resources
oc get clusterpolicy

# Get detailed YAML
oc get clusterpolicy gpu-cluster-policy -o yaml

# Check status
oc get clusterpolicy gpu-cluster-policy -o yaml | grep -A 10 status
----

=== Controller: The Reconciliation Engine

A Controller watches Custom Resources and reconciles actual cluster state to match the desired state.

**Purpose:** Contains the business logic to make reality match the CR spec.

**GPU Operator Controller Workflow:**

1. **Watch:** Controller watches for `ClusterPolicy` resource changes
2. **Read:** Controller reads current cluster state
   * What DaemonSets exist?
   * What driver version is deployed?
   * Are monitoring pods running?
3. **Compare:** Desired state (ClusterPolicy spec) vs. actual state
4. **Reconcile:** If mismatch, take action:
   * Create missing DaemonSets
   * Update DaemonSet image tags to match driver version
   * Create ConfigMaps with configuration
   * Create Services for metrics endpoints
5. **Update status:** Write observed state to ClusterPolicy status subresource
6. **Repeat:** Loop every ~5 seconds

**Example Reconciliation Scenario:**

**Initial State:** No GPU software deployed

**Admin Creates ClusterPolicy:**

[source,yaml]
----
spec:
  driver:
    enabled: true
    version: "535.129.03"
  dcgm:
    enabled: true
----

**Controller Reconciles:**

[source,bash]
----
# Reconciliation loop detects new ClusterPolicy
# Controller creates:
oc get daemonsets -n nvidia-gpu-operator

NAME                              DESIRED   CURRENT   READY
nvidia-driver-daemonset           3         3         3
nvidia-container-toolkit-ds       3         3         3
nvidia-device-plugin-daemonset    3         3         3
nvidia-dcgm                       3         3         3
nvidia-dcgm-exporter              3         3         3
gpu-feature-discovery             3         3         3
----

**Self-Healing Example:**

**Scenario:** Admin accidentally deletes DCGM DaemonSet

[source,bash]
----
# Admin deletes DaemonSet
oc delete daemonset nvidia-dcgm -n nvidia-gpu-operator

# DaemonSet is gone
oc get daemonset nvidia-dcgm -n nvidia-gpu-operator
Error from server (NotFound): daemonsets.apps "nvidia-dcgm" not found
----

**Controller detects and auto-recovers:**

[source,bash]
----
# Next reconciliation loop (5-10 seconds later)
# Controller compares:
#   Desired state (ClusterPolicy): dcgm.enabled: true
#   Actual state: No DCGM DaemonSet exists
#   Action: Create DCGM DaemonSet

# Watch it get recreated automatically
oc get daemonset nvidia-dcgm -n nvidia-gpu-operator -w

NAME           DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE
nvidia-dcgm    0         0         0       0            0
nvidia-dcgm    3         0         0       0            0
nvidia-dcgm    3         3         0       3            0
nvidia-dcgm    3         3         3       3            3
----

**Recovery time:** ~30 seconds, zero human intervention required.

This is the operator pattern's self-healing capability in action.

== Cluster Operators vs. Add-on Operators

OpenShift has two categories of operators with different lifecycles and support models.

=== Cluster Operators

**Purpose:** Manage core OpenShift platform components.

**Characteristics:**

* Shipped with OpenShift installation
* Installed automatically during cluster provisioning
* Managed by the Cluster Version Operator (CVO)
* Cannot be uninstalled (part of core platform)
* Lifecycle tied to OpenShift version upgrades
* Always supported by Red Hat

**Examples:**

[source,bash]
----
oc get clusteroperators

NAME                   VERSION   AVAILABLE   PROGRESSING
authentication         4.14.2    True        False
console                4.14.2    True        False
dns                    4.14.2    True        False
etcd                   4.14.2    True        False
image-registry         4.14.2    True        False
ingress                4.14.2    True        False
kube-apiserver         4.14.2    True        False
monitoring             4.14.2    True        False
network                4.14.2    True        False
storage                4.14.2    True        False
----

**Key Point:** You cannot uninstall cluster operators. They are essential platform infrastructure.

=== Add-on Operators

**Purpose:** Extend platform with additional capabilities.

**Characteristics:**

* Installed post-cluster-deployment via OLM
* User manages lifecycle (install, upgrade, uninstall)
* Independent versioning from OpenShift
* Optional (not required for cluster operation)

**Three Categories of Add-on Operators:**

==== 1. Red Hat Operators (redhat-operators)

**Support:** Full Red Hat support with SLA

**Examples:**

* Node Feature Discovery (NFD)
* Red Hat OpenShift AI
* Red Hat OpenShift Logging
* Red Hat OpenShift Serverless

**Characteristics:**

* Tested and certified by Red Hat
* Included in OpenShift subscription
* Coordinated release cycles with OpenShift
* Production-ready with enterprise support

**When to use:** Production deployments of Red Hat platform extensions

==== 2. Certified Operators (certified-operators)

**Support:** Vendor support (NVIDIA, MongoDB, etc.)

**Examples:**

* NVIDIA GPU Operator (NVIDIA support)
* MongoDB Enterprise Operator (MongoDB support)
* Crunchy PostgreSQL Operator (Crunchy Data support)
* Portworx Enterprise (Portworx support)

**Characteristics:**

* Vendor-developed and supported
* Red Hat certified (tested against OpenShift)
* Requires vendor support contract
* Independent vendor release cycles

**When to use:** Production deployments of vendor software where vendor support is available

==== 3. Community Operators (community-operators)

**Support:** Community only, no SLA

**Examples:**

* Prometheus Operator (upstream)
* Argo CD Operator
* Various open-source projects

**Characteristics:**

* Community contributions
* No official support
* Varying quality and maintenance
* May be outdated or unmaintained

**When to use:** Development, testing, POCs only - never production

== Update Strategies: Automatic vs. Manual Approval

OLM supports two update strategies via the `installPlanApproval` field in Subscriptions.

=== Automatic Approval

**Configuration:**

[source,yaml]
----
spec:
  installPlanApproval: Automatic
----

**Behavior:**

* OLM automatically approves InstallPlans for upgrades
* Operator upgrades as soon as new version published to channel
* Zero manual intervention required

**Pros:**

* Always up-to-date with latest security patches
* Reduced manual overhead
* Automatic bug fix delivery

**Cons:**

* Risk of breaking changes in upgrades
* No opportunity to review release notes first
* Potential for unexpected behavior changes
* Upgrades may occur during business hours

**Use Case:** Development and test environments

=== Manual Approval

**Configuration:**

[source,yaml]
----
spec:
  installPlanApproval: Manual
----

**Behavior:**

* OLM creates InstallPlan but waits for admin approval
* Admin reviews release notes and approves when ready
* Controlled upgrade timing

**Pros:**

* Control over when upgrades occur
* Opportunity to review release notes and test in dev first
* Schedule upgrades during maintenance windows
* Avoid unexpected breaking changes

**Cons:**

* Manual overhead for each upgrade
* Delayed security patches if not monitored
* Requires process to track available updates

**Use Case:** Production environments, mission-critical workloads

**Manual Approval Workflow:**

[source,bash]
----
# 1. OLM detects new operator version and creates InstallPlan
oc get installplan -n nvidia-gpu-operator

NAME            CSV                                  APPROVAL   APPROVED
install-yyyyy   gpu-operator-certified.v23.9.2      Manual     false

# 2. Admin reviews InstallPlan
oc describe installplan install-yyyyy -n nvidia-gpu-operator

# 3. Admin reviews NVIDIA GPU Operator v23.9.2 release notes
# (external: https://docs.nvidia.com/...)

# 4. Admin tests upgrade in dev environment first

# 5. Admin approves production upgrade
oc patch installplan install-yyyyy -n nvidia-gpu-operator \
  --type merge --patch '{"spec":{"approved":true}}'

# 6. OLM proceeds with upgrade
oc get csv -n nvidia-gpu-operator -w
NAME                               DISPLAY              VERSION   PHASE
gpu-operator-certified.v23.9.0     NVIDIA GPU Operator  23.9.0    Replacing
gpu-operator-certified.v23.9.2     NVIDIA GPU Operator  23.9.2    Installing
gpu-operator-certified.v23.9.2     NVIDIA GPU Operator  23.9.2    Succeeded
----

=== Channel Selection

Operators publish updates on multiple channels representing different stability levels.

**Common Channel Patterns:**

[cols="1,3,1",options="header"]
|===
|Channel
|Description
|Use Case

|**stable**
|Production-ready, thoroughly tested releases
|Production

|**fast**
|Newer features, less testing than stable
|Early access

|**candidate**
|Release candidates, pre-release testing
|Testing only

|**vX.Y**
|Version-pinned (e.g., v23.9, v24.3)
|Production (version control)
|===

**NVIDIA GPU Operator Channels:**

The GPU Operator uses version-specific channels:

* `v23.9` - All v23.9.x releases (23.9.0, 23.9.1, 23.9.2)
* `v24.3` - All v24.3.x releases
* `v24.6` - All v24.6.x releases

**Best Practice for Production:**

Pin to version-specific channel to control major version upgrades:

[source,yaml]
----
spec:
  channel: "v23.9"           # Only receive v23.9.x updates
  installPlanApproval: Manual  # Control when patches are applied
----

**Changing Channels (Major Version Upgrade):**

[source,bash]
----
# Edit Subscription to change channel
oc edit subscription gpu-operator-certified -n nvidia-gpu-operator

# Change: channel: "v24.3"
# OLM creates InstallPlan for v24.3.0 (if manual approval, requires approval)
----

[WARNING]
====
Major version upgrades may include breaking changes:

* CRD schema changes (ClusterPolicy fields may change)
* Deprecated features removed
* New configuration requirements

Always test major version upgrades in development environment first.
====
